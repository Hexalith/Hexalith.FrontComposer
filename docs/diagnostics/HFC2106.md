## Problem

The Shell could not apply a persisted Theme or Density preference during hydration. It continues with
feature defaults or defers preference hydration until browser storage becomes available.

## Common Causes

The preference key has no stored value, the stored value is unreadable or invalid, or browser storage
is temporarily unavailable during prerendering or circuit startup. Density emissions include a
`Reason` such as `Empty`, `Corrupt`, or `BrowserStorageUnavailable`.

## How To Fix

For an empty value, no action is normally required: the feature default is the expected first-visit
behavior. For `Corrupt`, clear or repair the scoped preference value. For
`BrowserStorageUnavailable`, verify that hydration retries after interactive browser storage becomes
available. Do not log preference contents, tenant identifiers, or user identifiers while diagnosing
the issue.

## Example

```text
HFC2106: Density hydration found no stored value — bootstrap defaults apply until the viewport watcher emits. Reason=Empty.
```

## Suppression Guidance

This is an informational runtime diagnostic. If the expected first-visit signal is too noisy, use a
narrow logging filter and document the rationale; do not disable higher-severity Shell diagnostics.

## Migration/Deprecation

Use `FcDiagnosticIds.HFC2106_PreferenceHydrationFallback` in source code. The former
`HFC2106_ThemeHydrationEmpty` public constant remains a non-error obsolete alias for source and binary
compatibility; the emitted identifier remains exactly `HFC2106`.
