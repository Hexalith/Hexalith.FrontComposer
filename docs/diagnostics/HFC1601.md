## Problem

The Shell runtime rejected a customization manifest before trusting it. The ID remains allocated in the SourceTools numeric range because the descriptor contract originated there, while the runtime emission is intentionally Shell-owned through the registry exception.

## Common Causes

Common causes are stale generated manifests, descriptor contract drift after a package update, or a hand-authored customization manifest that no longer matches the generated SourceTools contract.

## How To Fix

Regenerate the manifest from the current descriptor contract or correct the invalid entry. Keep any evidence bounded and sanitized, then rerun the diagnostic registry tests so the structured cross-package exception remains valid.

## Example

```text
HFC1601: What: Customization manifest is invalid. Expected: Shell runtime customization manifests match the SourceTools descriptor contract. Got: A manifest failed validation before it could be trusted. Fix: Regenerate or correct the manifest and keep evidence sanitized.
```

## Suppression Guidance

Suppression is `discouraged-error`. Prefer `.editorconfig` or a narrowly scoped pragma for analyzer diagnostics, and record migration or architecture rationale for Error-level suppressions.

## Migration/Deprecation

No migration action is required unless the registry later changes this diagnostic lifecycle.
