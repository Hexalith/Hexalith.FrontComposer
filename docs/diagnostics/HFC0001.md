## Problem

The v1.12 `QueryRequest` surface stores projection criteria alongside tenant, EventStore routing, validators, and cache metadata. New code should compose the request from a `ProjectionQuery`, which exclusively owns the projection type, paging, column and status filters, search query, and ordering.

The flattened constructor, flattened criteria properties, 19-value deconstruction, and `Filter` property remain available as compatibility shims throughout 2.x and are targeted for removal in v3.0.0. They stay synchronized with `QueryRequest.Criteria` and preserve the v1.12 flat JSON representation. `Filter` has no canonical replacement because it remains only for legacy EventStore payload forwarding.

HFC0001 originally covered only `Filter` beginning in v0.2.0. The composed-query migration broadens that existing diagnostic identity to the complete flattened criteria surface; it does not retroactively claim that every flattened member was deprecated in v0.2.0.

## Common Causes

HFC0001 appears when net10 code constructs `QueryRequest` with the 19-parameter constructor, reads or writes a flattened criteria property, uses the legacy deconstruction method, or accesses `Filter`. Canonical construction and access through `QueryRequest.Criteria` do not emit the diagnostic.

On `netstandard2.0`, the contract uses the platform-compatible `ObsoleteAttribute` shape and the compiler reports `CS0618` instead of the custom HFC0001 identifier. The API remains loadable and behaves identically on both target frameworks.

## How To Fix

Move projection criteria into `ProjectionQuery` and leave tenant, routing, validator, and cache metadata on `QueryRequest`.

```csharp no-compile reason="Illustrative adopter migration omits package references and using directives."
var request = QueryRequest.Create(
    new ProjectionQuery(
        "OrdersProjection",
        Skip: 0,
        Take: 25,
        ColumnFilters: new Dictionary<string, string> { ["Status"] = "Open" },
        SortColumn: "CreatedAt",
        SortDescending: true),
    TenantId: "tenant-a",
    Domain: "orders",
    AggregateId: "orders",
    QueryType: "ListOrders",
    CacheDiscriminator: "orders-grid");
```

Read canonical values through `request.Criteria`, for example `request.Criteria.Take` and `request.Criteria.SearchQuery`. Direct JSON serialization remains flat; do not add a nested `criteria` member to adopter serializers.

## Example

```text
HFC0001: What: Flattened QueryRequest projection criteria are deprecated. Expected: Compose QueryRequest.Criteria from ProjectionQuery. Got: A legacy flattened constructor, criteria property, or deconstruction entry point was used. Fix: Move projection type, paging, filtering, search, and ordering values into ProjectionQuery before v3.0.0.
```

## Suppression Guidance

Suppression is `allowed-with-rationale` for a bounded compatibility adapter that must read `Filter` or validate legacy source behavior. Prefer a narrowly scoped pragma and record the v3.0.0 removal dependency. Repository-owned application code should use the composed surface.

## Migration/Deprecation

Migration steps:

1. Create a `ProjectionQuery` from projection type, paging, column/status filters, search, and ordering values.
2. Pass that value to `QueryRequest.Create`.
3. Keep tenant, EventStore routing, ETags, and cache metadata on `QueryRequest`.
4. Replace flattened reads and `with` updates with `request.Criteria` reads and `request with { Criteria = request.Criteria with { ... } }`.
5. Rebuild net10 consumers until HFC0001 is absent; rebuild netstandard2.0 consumers until the corresponding `CS0618` fallback is absent. A bounded adapter that must still forward `Filter` cannot eliminate the warning and should use the narrow rationale-based suppression described above throughout 2.x.

The compatibility surface remains supported throughout 2.x and has an explicit v3.0.0 removal target. It was not removed in v0.4.0.
