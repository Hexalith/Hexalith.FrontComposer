# FrontComposer 3.1 to 4.0 Migration

Use this guide when upgrading a `3.1.x` application that referenced the public
`Hexalith.FrontComposer.Mcp.Skills.SkillBenchmark*` surface. FrontComposer 4.0 removes those
26 benchmark declarations from the shipped MCP assembly. There is no runtime NuGet replacement.

## Affected Versions

The break applies to `Hexalith.FrontComposer.Mcp` consumers moving from `3.1.x` to `4.0.0`.
Applications that only host MCP resources and tools are unaffected. Source or binaries that use
`SkillBenchmarkPrompt`, policies, gates, evidence writers, scores, or the offline scorer must
remove that integration and rebuild.

## Why This Changed

The benchmark harness is repository release evidence, not MCP runtime behavior. Keeping it public
would make the runtime package own provider budgets, benchmark prompts, baseline writes, and
evidence-file policy. FrontComposer 4.0 relocates that harness to the non-packable
`Hexalith.FrontComposer.Shell.Tests.Bench` executable and leaves the MCP package with its runtime
skill-corpus parser, loader, validators, resource provider, and generated-code validator.

The historical benchmark namespace remains inside the repository only to preserve implementation
bodies during relocation. Because the destination is not a NuGet package, it is not an adopter API.
No facade or type forwarder is shipped.

## Old Code

```csharp no-compile reason="The SkillBenchmark APIs were removed from the FrontComposer 4.0 MCP package."
using Hexalith.FrontComposer.Mcp.Skills;

SkillBenchmarkPromptSet prompts = SkillBenchmarkPromptSet.LoadEmbeddedV1();
SkillBenchmarkScore score = SkillBenchmarkOfflineScorer.Score(
    prompts.Prompts[0],
    Array.Empty<GeneratedCodeFile>());
```

## New Code

```csharp no-compile reason="FrontComposer 4.0 intentionally has no adopter-facing benchmark replacement."
namespace Acme.FrontComposer;

public static class McpRegistration
{
    public static void Configure()
    {
        // Remove application-owned SkillBenchmark* integration.
        // Keep only the application's MCP host and resource/tool registration.
    }
}
```

Do not copy the repository harness into application runtime code. If an application needs its own
provider evaluation, own that implementation and its data contract outside FrontComposer's runtime
packages.

## Analyzer And Code Fix

No HFC diagnostic or code fix is assigned to this package API removal. Existing binaries fail to
load the removed types, and source references fail compilation. Remove the references manually,
upgrade all FrontComposer packages together, rebuild, and run the application's MCP contract tests.

## Skill Corpus Evidence

The published FrontComposer skill corpus is unchanged. Its markdown resources, hashes, references,
generated-code validation, and MCP resource URIs remain MCP-owned. The benchmark prompt JSON is
repository evidence and no longer ships in the MCP package.

Repository maintainers validate the relocated harness with:

```bash
python3 eng/llm_benchmark.py validate-prompt-set --root . --output artifacts/benchmark/prompt-set.json
dotnet test tests/Hexalith.FrontComposer.Shell.Tests.Bench/Hexalith.FrontComposer.Shell.Tests.Bench.csproj --filter "Category=Performance&FullyQualifiedName~BenchmarkHarnessTests"
```

Those commands are for FrontComposer repository maintenance; they do not identify a replacement
runtime package for adopters.
