## Executive summary (Feynman-level, ~45 sec)

Today every `[Projection]`-annotated type emits the same generic DataGrid via `RazorEmitter` — the existing `ProjectionRoleAttribute` declares intent (ActionQueue / StatusOverview / DetailRecord / Timeline / Dashboard) but nothing in the generator or runtime consumes that intent. Story 4.1 **wires the role from attribute → IR → generated view body** so five rendering strategies land:

1. **Parse-stage extension.** `ProjectionRoleAttribute` gains an optional `WhenState` string (CSV of state-enum member names — the "explicit states" mandated by the epic AC block for ActionQueue). `AttributeParser` carries it + the existing role name through `DomainModel` into `RazorModel` (L04 full-name determinism preserved — no name transforms).
2. **Transform-stage branch.** `RazorModelTransform` selects a strategy per role (`Default` when attribute absent, `ActionQueue` / `StatusOverview` / `DetailRecord` / `Timeline` / `Dashboard` otherwise) and emits a discriminated `ProjectionRenderStrategy` onto the IR. Unsupported / unknown role values fall back to `Default` and raise a build warning.
3. **Emit-stage role dispatch.** `RazorEmitter` gains one `Emit{Strategy}Body` path per strategy. Each path still wraps the three-state Loading / Empty / Data shell (FluentSkeleton placeholders matching layout → FcEmptyState hook reserved for Story 4.6 → strategy-specific body). Contextual subtitle (UX-DR44) renders above each body and pulls its count from `IBadgeCountService` (3-5) rather than a per-type query.
4. **Runtime wiring.** Generated views consume cascading `RenderContext` (tenant / user / mode / density / read-only) already established by Story 2-2; badge-count subscription uses the `IState<FrontComposerCapabilityDiscoveryState>` mirror pattern proven in 3-5's `FrontComposerNavigation` consumer path (one subscription per view, not per row).
5. **Contract boundary.** 4-1 ships **the view shell + subtitle + per-role body skeleton**. DataGrid filters / sort (4.3), virtualization (4.4), expand-in-row (4.5), `FcEmptyState` content (4.6), and `FcFieldPlaceholder` expansion (4.6) all consume extension points this story freezes — no overlap, no re-litigation.

The design honors the "capability contract" principle: attribute shape and role enum match **PRD FR4 verbatim** (not the epic AC's `[ProjectionRoleHint]` / `Default` drift — see D1), keeps the source-generator deterministic and cache-stable (no Roslyn symbol references past Parse), lands decisions inside the L06 feature-story cap (≤ 25), and leaves every cross-story seam explicit per L01.

## Story

As a developer,
I want projections annotated with `[ProjectionRole(role)]` (optionally with an explicit-states filter) to render according to their declared role — action queue, status overview, detail record, chronological timeline, or default DataGrid — with a contextual subtitle that orients business users,
so that business users get purpose-fit views of my domain without me writing a single line of rendering code.

**Business value:** The first shipped surface where FrontComposer's role-hint promise becomes visible. Until 4-1 lands, every generated projection looks identical to every other projection — a generic DataGrid, useful but not intent-driven. After 4-1, annotating `[ProjectionRole(ProjectionRole.ActionQueue, WhenState="Pending,Submitted")]` on an `OrdersProjection` yields an inline-action DataGrid sorted by priority with the subtitle "3 orders awaiting your approval" — WITHOUT custom Razor, WITHOUT renderer registration, WITHOUT overriding the generator. This is FR4's entire premise; it is also the first story an adopter can paste into their press release as a full FrontComposer differentiator vs. hand-rolled Fluent UI apps.

### ADR-051 — PRD FR4 is authoritative over Epic 4 AC drift on attribute name + 5th role

**Status:** Accepted (Story 4-1 D1).

**Context:** Epic 4 Story 4.1 AC text reads `[ProjectionRoleHint(RoleHint.ActionQueue, WhenState = "Pending,Submitted")]` with roles `ActionQueue, StatusOverview, DetailRecord, Timeline, Default`. PRD FR4 reads `[ProjectionRole(role)]` with roles `ActionQueue, StatusOverview, DetailRecord, Timeline, Dashboard — capped at 5`. The existing codebase ships `ProjectionRoleAttribute` + `ProjectionRole { ActionQueue, StatusOverview, DetailRecord, Timeline, Dashboard }` (matches PRD). Story 3-5's `IActionQueueProjectionCatalog` already depends on both the attribute name and `ProjectionRole.ActionQueue`.

**Decision:** PRD FR4 wins. Keep `ProjectionRoleAttribute` + `ProjectionRole` enum unchanged. Treat Epic AC's `[ProjectionRoleHint]` + `Default` as drift between Epic authoring (narrative story breakdown) and PRD FR4 (versioned capability contract). "Default" becomes an internal `ProjectionRenderStrategy.Default` transform-stage sentinel — NEVER a public enum value.

**Rejected alternatives:**

- **Rename `ProjectionRoleAttribute` → `ProjectionRoleHintAttribute` AND add `ProjectionRole.Default` enum value.** Breaks the PRD FR4 capability contract + the 3-5 catalog's reflection scan (`GetCustomAttribute<ProjectionRoleAttribute>()?.Role == ProjectionRole.ActionQueue`). Requires migration for the adopter sample (Counter) + every test that references the types by their current names. Cosmetic rename with zero capability benefit.
- **Treat Epic as authoritative over PRD.** Inverts the spec hierarchy established by §implementation-readiness-report-2026-04-12.md, which ships the PRD as v1.0 contract and the Epic as its story-breakdown. Every other Epic 1-3 story has resolved PRD-vs-Epic drift by deferring to PRD. Changing the rule for Epic 4 creates a one-off precedent that future Epic authors can weaponize.
- **Introduce both names as aliases.** Both `[ProjectionRole]` and `[ProjectionRoleHint]` attributes registered as equivalent. The catalog checks both. Doubles the attribute surface + test matrix, violates PRD FR4's "capped at 5 — no more, no less" discipline.

**Consequences:**

- Epic 4 text remains unchanged (editorial drift acceptable; story spec is the canonical place to resolve it).
- 3-5 catalog works unmodified.
- Counter sample + all existing tests compile unchanged.
- A future adopter reading Epic 4 expecting `[ProjectionRoleHint]` gets a compile error on the annotation; Dev Notes §Known gaps records the drift with a pointer to PRD FR4 as the authoritative shape. Story 9-5 documentation synchronizes the Epic text with PRD at next docs pass.

**Verification:** `ProjectionRoleAttributeTests.AttributeNameMatchesPrdFR4`, `ProjectionRoleTests.EnumMembersMatchPrdFR4CapOf5`, `IActionQueueProjectionCatalogTests.StillScansProjectionRoleAttribute` (regression lock — Story 3-5 coverage).

### ADR-052 — Role-dispatched body emission via single-switch strategy, not polymorphism

**Status:** Accepted (Story 4-1 D4 + D5).

**Context:** Six render strategies (Default, ActionQueue, StatusOverview, DetailRecord, Timeline, Dashboard) share a common view shell (class header, lifecycle hooks, formatters, Loading skeleton, Empty placeholder, subtitle) but differ in the `state.Items.Count > 0` body. Two canonical structures: (a) `switch (strategy)` dispatch to strategy-specific static helpers; (b) polymorphism via `IProjectionBodyEmitter` implementations with `IEnumerable<IProjectionBodyEmitter>` injection.

**Decision:** Single-switch strategy in `RazorEmitter.Emit`, delegating to `ProjectionRoleBodyEmitter.Emit{Strategy}Body` static methods. **Switch is an EXPRESSION (not statement) with a throwing default arm** — per Winston review, this guarantees compile-time exhaustiveness. When a future story adds a strategy without wiring the case, the build fails with a clear error instead of shipping a blank-view regression. Shared scaffolding (loading shell, empty shell, formatters, subtitle host, class header) stays in `RazorEmitter` private helpers.

**Rejected alternatives:**

- **Polymorphism via `IProjectionBodyEmitter`.** Runtime DI is unavailable in netstandard2.0 source generators; source generators cannot use `Microsoft.Extensions.DependencyInjection`. A hand-rolled `IEnumerable<>` via reflection-scanned strategy classes adds complexity with no adapter-extensibility payoff (adopters extend rendering via the Story 6 customization gradient, not by implementing emitter strategies).
- **Six separate emitter classes, each owning the full class header + scaffolding.** Duplicates loading/empty/subtitle code six times; any future shared-scaffolding change (e.g., adding telemetry hooks in Story 9-4) requires six file edits instead of one.
- **String-templated body via Razor-like placeholder resolution.** Loses compile-time exhaustiveness check on strategies; a missing case produces silent empty emissions. The switch with SourceTools-internal enum surfaces exhaustiveness warnings at compile time when roles are added.

**Consequences:**

- Adding a new role (e.g., Story 6-3 `Dashboard` specialisation, or a future `SummaryKpi`) requires (a) adding an enum member, (b) adding a `case` in the switch, (c) adding one `EmitXBody` static — compile-time exhaustiveness catches missing switch arms.
- `RazorEmitter.cs` line count grows modestly (Default body stays; additions are the switch + calls); `ProjectionRoleBodyEmitter.cs` stays small (5-10 static methods, each ~30-60 lines).
- Approval-test harness catches strategy-specific output regressions via snapshot per role.

**Verification:** `RazorEmitterStrategyDispatchTests.EveryStrategyDispatchIsExhaustive`, `ProjectionRoleBodyEmitterTests.Emit{Strategy}Body_MatchesApprovedSnapshot` (six methods × 1 test each = 6), `RazorEmitterScopeGuardrailTests.ProjectionViewBodyStaysMinimal` (D22).

### ADR-053 — `FcProjectionEmptyPlaceholder` is a Story-4.6-replaceable Shell component, NOT an emitter-level empty block

**Status:** Accepted (Story 4-1 D7).

**Context:** Every generated projection view needs an empty-state body. Story 4.6 specifies the full FcEmptyState with CTA button, secondary text, role=status, aria-label — that work is not 4-1's scope. Two choices: (a) emit a minimal inline empty block in 4-1, replaced by a Story-4.6 regeneration pass; (b) emit a stable `<FcProjectionEmptyPlaceholder>` invocation in 4-1, where Story 4.6 swaps the Shell component body without touching the generator.

**Decision:** Emit `<FcProjectionEmptyPlaceholder ProjectionType="@typeof({TypeName})" />` in 4-1. Story 4.6 replaces the component body (CTA button, secondary text, etc.) WITHOUT changing the generator call-site.

**Rejected alternatives:**

- **Inline minimal empty block in 4-1; regenerate in 4.6.** Every adopter rebuilds on 4.6 upgrade. Generator diff under `\.g\.razor\.cs$` creates noise. Regression surface (a 4.6 generator bug re-emits all projection views).
- **Ship a full empty state in 4-1 (CTA, icon, secondary text).** Overlaps Story 4.6's AC scope. Couples 4-1 to command-per-bounded-context resolution (which command is "the" one to pre-fill for CTA?). Violates scope guardrail L07 (test count inflation).
- **Generate no empty block; adopter registers a global empty component.** Negates the "works without ceremony" promise (FR11). First-run Counter has a blank region on empty data — UX regression.

**Consequences:**

- 4-1 ships a functional empty placeholder today (muted icon + "No {entities} yet." text).
- Story 4.6 becomes a pure Shell-level change: replace the component body, update its tests, keep parameter list + type name stable.
- The emitter contract `<FcProjectionEmptyPlaceholder ProjectionType="@typeof(T)" Role="ProjectionRole.{role}" />` is frozen; **parameter list is EXPLICITLY APPEND-ONLY** (Graph-of-Thoughts finding at review round 2): Story 4.6 MAY add parameters (e.g., `SecondaryText`, `CtaCommand`) but MUST NOT remove, rename, or reorder existing parameters. This is the same append-only discipline applied to `ProjectionAttribute` + `ProjectionRoleAttribute` at Story 1-2 — consistent across the framework's public surfaces.
- **Role parameter emitted in 4-1 even though 4-1 doesn't consume it** (H4 per review round 2): the generator emits `Role="ProjectionRole.{role}"` on every `<FcProjectionEmptyPlaceholder>` invocation. 4-1's component ignores the value; 4.6 consumes it for role-differentiated CTA copy ("Create first order" on ActionQueue-empty vs. "View past activity" on Timeline-empty). Zero emitter diff when 4.6 lands — saves every adopter a rebuild.
- **This is the first instance of a broader pattern** — Shell-component-as-pluggable-slot. The generator emits a stable invocation to a Shell component whose body is replaced out-of-band by future stories. Story 6-3's `[SlotOverride]` mechanism consumes this pattern (not yet implemented; the pattern precedent matters). The mechanism: frozen parameter list + component name; body swap is a Shell-internal concern. Any future "Story X replaces component Y" spec should cite ADR-053 as the template.
- No generator regeneration on 4.6 upgrade.

**Verification:** `FcProjectionEmptyPlaceholderTests.RendersMinimalPlaceholder_WithEntityPluralAndIcon`, `FcProjectionEmptyPlaceholderTests.ParameterSurface_IsFrozenForStory4_6` (documents the frozen shape; the Role parameter is declared but not consumed in 4-1 — test asserts it is present + optional + typed `ProjectionRole`), `RazorEmitterScopeGuardrailTests.ProjectionViewBodyStaysMinimal` asserts the emitter never emits "SendYourFirstCommand" / CTA copy inline.
