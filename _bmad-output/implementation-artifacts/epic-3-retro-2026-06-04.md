### What Went Well

1. **Contract-first command work reduced ambiguity.** Story 3.3 created the FC-CMD identity and
   correlation contract; Story 3.5 created the EventStore status endpoint contract; Story 3.6 created
   the budget contract. Later stories stopped reopening earlier decisions.

2. **The GUID escape hatch was removed.** Story 3.1 moved generated form correlation allocation to
   `IUlidFactory.NewUlid()`, and Story 3.3 removed the `UlidFactory` GUID fallback. Both `MessageId`
   and `CorrelationId` are now validated as canonical 26-character Crockford ULIDs in pending state.

3. **Lifecycle UI became visible and typed.** Story 3.4 surfaced `Acknowledged`, preserved editable
   rejected forms, carried typed rejection metadata, and kept rejection details plain-text bounded.

4. **Status polling now follows backend truth.** Story 3.5 replaced the EventStore-enabled
   `NullPendingCommandStatusQuery` path with `EventStorePendingCommandStatusQuery`, while preserving
   null behavior for base shell hosts.

5. **Command polling was decoupled from projection fallback cadence.** Story 3.6 moved command status
   polling to a scoped `TimeProvider` driver and kept `PendingCommandPollingCoordinator` as the single
   mutation boundary.

### Challenges And Growth Areas

1. **Documentation and File List drift repeated.** Every Epic 3 review except Story 3.6 recorded at
   least one File List or completion-note correction. This is the same workflow weakness called out in
   Epic 2.

2. **Environment-blocked test commands still add noise.** Multiple stories recorded socket, named-pipe,
   or network-audit blockers before the exact solution-level test command could run. Focused fallback
   evidence was useful, but the record now has inconsistent "VSTest blocked" versus "VSTest ran here"
   language depending on the reviewer environment.

3. **Story numbering drift remains in comments and legacy notes.** Some code comments still carry
   older story labels. They are not behaviorally wrong, but they make future archaeological work harder.

4. **Row-level new-item producer remains deliberately out of scope.** Epic 3 confirmed again that the
   current EventStore status response and projection nudge seams do not carry precise row identity for
   `FcNewItemIndicator` producer wiring.

## 3. Key Learnings

- **Confirm-and-pin works when the brownfield implementation is mostly present.** Stories 3.1 and
  3.2 added high-value guardrails without production churn.
- **Identity contracts need failure behavior, not just happy-path shape.** Removing the GUID fallback
  mattered because a rare fallback path could silently violate every downstream ULID invariant.
- **Backend route names are not the same as frontend identity ownership.** EventStore names the route
  parameter `correlationId`, but FrontComposer correctly queries by pending `MessageId` because
  EventStore defaults command-status correlation to the submitted message ID.
- **A polling seam should schedule separately from the mutation boundary.** Story 3.6 kept driver
  timing separate from coordinator mutation, which made cadence, cap, expiry, and first-wins behavior
  testable.
- **Deferrals should name missing payload shape.** The new-item producer remains deferred because the
  current status/nudge payloads lack row identity, not because the team forgot the feature.

## 6. Significant Discovery Check

No discovery invalidates Epic 4, but one planning constraint is important: Epic 4's retry and
concurrency work must not reinterpret Epic 3 polling expiry as an automatic retry policy. Epic 3
sets automatic client retry to `0`; Epic 4 may add explicit retry behavior, but it must be designed as
a separate policy layered over the existing lifecycle and pending-command resolver.
