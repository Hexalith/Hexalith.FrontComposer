## Executive summary (Feynman-level, ~45 sec)

Today every generated projection view renders `state.Items.AsQueryable()` into a bare `FluentDataGrid` without virtualization — adopters whose projections hold 500+ rows see the browser stall because the render tree materialises every row up-front. Story 4-3 shipped filters/sort/search to narrow those sets, but a 10,000-row `OrdersProjection` with a "last 90 days" filter still returns 8,200 rows — below any sensible filter-threshold but still large enough to blow the render budget. Story 4-4 closes that gap: **every generated view gains `Virtualize="true"` + `DisplayMode="Table"` + an `ItemSize`-tuned row budget + an `ItemsProvider` server-side lane for 500+ items** while Story 2-2's reserved `ScrollTop` field becomes live (scroll-position captured → persisted via the 3-6 effect → restored within-session).

1. **Virtualize default ON.** `RazorEmitter.EmitStandardDataGrid` now emits `Virtualize="true"` + `DisplayMode="DataGridDisplayMode.Table"` + `ItemSize="{densityRowHeightPx}"` on every generated `FluentDataGrid` (D1 / D21 — Table mode is the Fluent-recommended partner for `Virtualize`; Grid mode is known to exhibit scroll glitches per Fluent docs). The density-to-row-height mapping reuses Story 3-3's `DensityLevel` enum: `Compact=32px`, `Comfortable=44px`, `Roomy=56px`. The value is resolved at render time from the cascading `RenderContext.Density`, NOT at emit time (density is user-scoped, emit-time is compilation-scoped).
2. **Client-side vs server-side lane.** For projections where `state.Items.Count < FcShellOptions.VirtualizationServerSideThreshold` (default `500`), the grid consumes `Items` directly — client-side virtualization only renders visible rows plus `OverscanCount="3"` (D2). For counts at or above the threshold, the generated view routes through a `GridItemsProvider<T>` callback that dispatches `LoadPageAction(viewKey, skip, take, filters, sort, search)` and awaits `IState<LoadedPageState>.Value.PagesByKey[{viewKey, skip}]` — the reducer accumulates fetched pages, and the callback resolves with the requested slice. ETag reuse is inherited from Story 1-5's `IQueryService.QueryAsync` via `QueryRequest.ETag` (no new cache layer).
3. **Server-side pagination mechanics.** `GridItemsProvider<T>` signature is `Func<GridItemsProviderRequest<T>, ValueTask<GridItemsProviderResult<T>>>`. The generated view emits a provider method that (a) reads the hydrated filter/sort/search state from the Fluxor snapshot, (b) dispatches `LoadPageAction(viewKey, request.StartIndex, request.Count, filters, sort, search)`, (c) `await`s a `TaskCompletionSource<GridItemsProviderResult<T>>` the reducer signals when the page lands, (d) cancels pending awaits via `request.CancellationToken` flowing into `LoadPageAction` (D3). No new Fluxor effect — `LoadPageAction` dispatches into the existing Story 1-5 `LoadAction`-adjacent service call with `QueryRequest.Skip` + `QueryRequest.Take` populated.
4. **Scroll-position capture + restore.** The generated grid container binds `@onscroll` which dispatches `ScrollCapturedAction` per-event. A PURE reducer updates `GridViewSnapshot.ScrollTop` (no chained dispatch — Fluxor purity per D4 revised). `ScrollPersistenceEffect` (new per Winston review) listens, debounces coalesced signals via 150 ms `TimeProvider` delay, then dispatches `CaptureGridStateAction` so 3-6's persistence effect takes over. On view mount, the existing Story 4-3 `RestoreGridStateAction` dispatch (in `OnInitialized`) pulls the snapshot — `ScrollTop` is applied via a JS interop `scrollToOffset(viewKey, top)` call AFTER the first render tick via the unified `OnAfterRenderAsync` ordered-hook list (D5 — within-session restore only; cross-session is reset to 0 because hours-later scroll context is an anti-pattern).
5. **FcColumnPrioritizer gate at 15 fields.** A new Shell component `FcColumnPrioritizer` wraps the `FluentDataGrid`'s column surface. When `model.Columns.Count > 15` (UX-DR63 / UX-DR7 trigger), the emitter wraps the grid's `ChildContent` in a priority-aware `@foreach` that renders ONLY the first 8-10 columns by priority (D6) and appends a `FluentButton Appearance="Outline"` labeled `"More columns ({N} hidden)"` at the end of the header row. Clicking the button opens a `FluentPopover` with a `FluentCheckbox` list of hidden columns; toggling a checkbox dispatches `ColumnVisibilityChangedAction(viewKey, columnKey, isVisible)`. Visibility writes back to `GridViewSnapshot.Filters["__hidden"]` as a CSV of hidden column names (D7 — another reserved key, mirroring 4-3 D3 `__status`/`__search` packing — no blob schema bump).
6. **Column priority attribute.** New `[ColumnPriority(int)]` property-level attribute in Contracts, parsed by `AttributeParser` into `PropertyModel.ColumnPriority`. Lower-numbered = higher priority (1 = most important); unannotated properties get `int.MaxValue` and slot into declaration order after explicitly-prioritised columns. The transform stage (Story 1-5) sorts columns stable-by-priority-then-declaration-order when the count exceeds 15. HFC1028 (Information) fires when two columns share an explicit priority — deterministic tiebreaker is declaration order.
7. **Safety rails — `MaxUnfilteredItems` + `SlowQueryThresholdMs`.** Two new `FcShellOptions` properties: `MaxUnfilteredItems` (int, default 10000) and `SlowQueryThresholdMs` (int, default 2000). When a projection's unfiltered page fetch returns exactly `MaxUnfilteredItems` items (server-capped), the generated view renders a `FluentMessageBar Intent="Info"` with the localised `MaxItemsCapNoticeTemplate` copy. When the first page fetch's elapsed time exceeds `SlowQueryThresholdMs`, the view renders a second `FluentMessageBar Intent="Info"` with the `SlowQueryNoticeMessage` copy — the banner is auto-dismissing after 5 s (matches Story 2-4 `ConfirmedToastDurationMs` precedent). Measurement is `Stopwatch`-based inside the `LoadPageAction`-adjacent service call path (D8).
8. **Row count header display.** When no filters are active, the view renders a simple `"{count} {entityPlural}"` (e.g. `"47 orders"`) above the grid. When filters ARE active, Story 4-3's `FcFilterSummary` already owns the `"{filteredCount} of {totalCount}"` display — 4-4 does NOT duplicate. Count comes from `state.TotalCount` server-supplied per Story 4-3 D13 (D9 — routing discipline: exactly one count surface visible at a time).

The design honors the "capability contract" principle: `GridViewSnapshot` schema UNCHANGED (column visibility packed in reserved key, scroll position uses the already-reserved `ScrollTop` field), decisions land inside the L06 feature-story cap (≤ 25 — 4-4 targets 22), every cross-story seam is explicit (L01), and fail-closed tenant/user scoping is INHERITED unchanged from Story 3-6 per L03 (no new scope-check code). Scope is deliberately narrow: **virtualization + column prioritization + safety rails — nothing else.** Expand-in-row (4.5), empty states (4.6), `FcNewItemIndicator` (Story 3-5 follow-up), and Epic 6 customization overrides are OUT of scope.

## Story

As a business user,
I want DataGrids to handle large datasets smoothly via virtual scrolling and auto-manage column visibility when projections have many fields,
so that performance stays fast on 500+-row projections and the view remains usable when a projection declares dozens of properties.

**Business value:** Until 4-4 lands, every `OrdersProjection` with 5,000 rows is a render-tree materialization bomb — opening the view stalls the browser tab for 3-4 seconds while Blazor builds the full DOM, and scrolling through is stuttery because every row is a real DOM node. After 4-4 ships, the same projection renders in < 200ms because only the ~15 visible rows are in the DOM at any time; scrolling triggers the Fluent `Virtualize` machinery which tears down off-screen rows and fetches server-side pages transparently. For a fulfillment operator scanning 5,000 daily orders or a finance analyst filtering a 10,000-row transaction ledger, this is the difference between "the tool loads" and "the tool is unusable." NFR4's "P95 render time < 300ms on 500 virtualized rows" is UNREACHABLE without 4-4. It is also the story that operationalises Story 2-2's reserved `ScrollTop` field (dormant since 2-2 landed) and unlocks the FR12 "smart browsing" promise for the common case of wide projections (>15 fields) where the current output is visually unwieldy.

## Dev Notes

### Relevant architecture patterns and constraints

- **Zero-override architecture** (`architecture.md:30`): `FluentDataGrid`, `FluentPopover`, `FluentCheckbox`, `FluentButton`, `FluentMessageBar` all consumed verbatim. No custom CSS, no token overrides. The `Virtualize` + `DisplayMode="Table"` pairing follows the Fluent team's own published guidance (`FluentDataGrid` MCP notes) — we inherit their tuning, we don't second-guess.
- **Convention-based rendering** (`architecture.md:493-497`): "IR metadata (field types, role hints, badge mappings)" — 4-4 operationalises the (previously empty) `ColumnModel.Priority` IR field. After 4-4, every generated view has a cap on its visible-column count that scales with projection complexity.
- **Fluxor single-source-of-truth discipline** (Story 1-3 ADR-008): every state change is an immutable `record` action; reducers are PURE `(state, action) => state`; effects are the only dispatch-boundary. 4-4's `LoadPageAction` dispatches through `LoadPageEffects` (sole IO gate); `ColumnVisibilityPersistenceEffect` + `ScrollPersistenceEffect` (new per Winston review 2026-04-23) listen for `ColumnVisibilityChangedAction` + `ResetColumnVisibilityAction` + `ScrollCapturedAction` and dispatch `CaptureGridStateAction` so persistence stays single-pipelined through Story 3-6's existing effect. **Reducers NEVER chain dispatches** — this is the correction to the pre-review design.
- **IR cache stability** (Story 1-4 incremental-build contract): 4-4 adds `PropertyModel.ColumnPriority` + `ColumnModel.Priority` as derived fields; `RazorModel` unchanged in shape but carries the sorted `Columns` list. IR byte-stability preserved — Story 1-4 cacheability tests stay green (the stable sort is deterministic over the same input).
- **Blazor Server + Blazor Auto compatibility** (UX-DR59): `IJSRuntime` for scroll interop works in both modes; `DataGridScrollInterop` is circuit-scoped in Server, app-scoped in WASM (standard Blazor scope rules). The JS module `fc-datagrid.js` ships via the Shell's static web assets (`_content/Hexalith.FrontComposer.Shell/js/`).
- **Persistence fail-closed** (L03 tenant isolation): relies on Story 3-6's existing fail-closed `TryResolveScope` in `DataGridNavigationEffects`. 4-4 does NOT add new scope-check code; it relies on the existing guard. `LoadPageEffects.HandleLoadPage` routes through `IQueryService.QueryAsync` which honours `QueryRequest.TenantId` (populated upstream). Any null/empty tenant short-circuits at the service boundary — the 4-4 effect does not need its own guard because it's not persisting anything that could leak across tenants.
- **NFR4 performance budget**: 500 virtualized rows P95 < 300ms (user-perceived browser paint). Fluent UI v5's `Virtualize` implementation renders only visible rows + `OverscanCount="3"` — ~15-18 rows in the render tree at any time. bUnit harness in T4.9 asserts **component-construction P95 < 50ms** as a regression rail (per AC3 revised); full browser-paint NFR4 verification is deferred to Story 10-2 Playwright CI (Murat review 2026-04-23 — bUnit cannot measure real layout or GPU paint).
- **Fluxor synchronous dispatch assumption (NEW 2026-04-23 late-evening per elicitation mentor-apprentice)**: the TCS double-registration idempotency guard in D3 (effect checks `ContainsKey`, dispatches `LoadPageCancelledAction`, then registers the new TCS) relies on Fluxor's DEFAULT synchronous dispatch semantics — `IDispatcher.Dispatch` returns only after ALL reducers have processed the action. Under stock `Store` configuration this holds. If an adopter adds middleware that delays dispatch (async middleware, batching middleware), the ordering assumption breaks and a rapid double-dispatch could race the cancel+register sequence. The `LoadPageTCSLifecycleTests` double-registration test uses stock configuration and is the gate. If future work adopts non-stock Fluxor configurations, re-audit D3 step (1a).
- **Author obligation — `ItemKey` non-null discipline (NEW 2026-04-23 late-evening per elicitation mentor-apprentice)**: `_itemKeyAccessor` (resolved per D13 revised) boxes whatever the first-matched accessor returns, including `null`. Authors MUST ensure the chosen property (`AggregateId` / `Id` / `Key`) is non-null for every row visible in a projection list view. First-match resolution does NOT fall through on a null runtime value — it fails open to Blazor's `@key` with null, which collapses all null-ID rows to one. Authors projecting optional / nullable identifiers for certain aggregate states (pre-commit, draft) MUST either: (a) non-nullify in the projection constructor, (b) exclude such rows from list views, or (c) define a deterministic non-null synthetic key. HFC1029 does NOT detect this — it's an adopter-side contract.
- **Backgrounded-tab timer behaviour (NEW 2026-04-23 late-evening per elicitation performance-profiler-panel)**: the JS `setTimeout`-based throttle in `captureScrollThrottled` is subject to browser background-tab throttling (Chrome limits to 1 Hz for hidden tabs; Safari suspends). A user scrolling, then backgrounding the tab mid-gesture, will have the pending trailing timer fire late (up to 1s+) on re-foreground — carrying the `pendingScrollTop` captured at schedule time. Since users cannot scroll a backgrounded tab, this is benign (reducer runs with a technically-stale-but-logically-correct position; the debounce effect then fires normally on next real scroll). No mitigation needed; noted for completeness.

### Adopter operational guidance (NEW 2026-04-23 late-evening per elicitation customer-support-theater + performance-profiler-panel)

**Tuning the three knobs — `MaxUnfilteredItems`, `VirtualizationServerSideThreshold`, `MaxCachedPages`:**

| Adopter scenario | `MaxUnfilteredItems` | `VirtualizationServerSideThreshold` | `MaxCachedPages` | Notes |
|---|---|---|---|---|
| Small records (<1 KB/row), ample memory (desktop) | 10 000 (default) | 500 (default) | 200 (default) | Out-of-the-box defaults fit. |
| Large records (>5 KB/row), ample memory | 5 000 | 200 | 300 | Lower cap to manage payload; lower server-side threshold so less materialises client-side; raise cache to smooth scroll-back. |
| Resource-constrained edge / low-end mobile | 2 000 | 100 | 50 | All three dialled down; expect frequent refetch on scroll-back. |
| High-concurrency call-center (50+ rows visible) | 20 000 | 1 000 | 500 | Raise cap for bulk scan; raise threshold so more stays client-side; raise cache for rapid scroll-back. |
| Very-wide projections (>30 columns) | 10 000 | 500 | 200 | Knobs unchanged — `FcColumnPrioritizer` + virtualization handle the width; tune column `[ColumnPriority]` annotations instead. |

**Database / backend prerequisite (NEW per elicitation performance-profiler-panel):** projection queries with high `Skip` values (routinely > 1 000) require WELL-INDEXED sort paths on the backend. Without an index on the `QueryRequest.SortColumn`, server-side pagination at `Skip=9 000` degenerates to O(N) full scan — 4-4's virtualization moves the perf problem from client render-tree to backend database. Adopters fielding projections with many thousands of rows MUST verify their `IQueryService` implementation uses an index-friendly pagination (keyset pagination, indexed covering queries). This is NOT FrontComposer's responsibility; it IS the adopter's responsibility to verify.

**Logging / observability guidance (NEW per elicitation performance-profiler-panel):** `LoadPageEffects` emits Information-level logs on eviction (per D10 re-revised — one log per eviction event). High-volume sessions (active users scrolling rapidly) can emit 200-500 eviction events/hour/user. Adopters with log-volume SLAs should FILTER the `Hexalith.FrontComposer.Shell.State.DataGridNavigation.LoadedPageReducers` category to Warning-or-above in their logging configuration (standard `Microsoft.Extensions.Logging` category filter) — the operator-visible signal is preserved under normal load; high-churn sessions are filtered without losing Fluxor-level error visibility (which uses other categories). A structured rate-limiting / batching diagnostic is Epic 5 "Reliable real-time experience" observability workstream territory.

- **Page-refetch UX during eviction-driven re-fetch (NEW 2026-04-23 late-evening per elicitation customer-support-theater).** When a previously-loaded page is evicted from `PagesByKey` and then re-requested due to scroll-back, Fluent's default skeleton rows render while `ItemsProvider` re-fetches — but FrontComposer provides NO contextual indicator ("loading previous page…", scroll-velocity-predictive cache retention, or pre-fetch of adjacent pages). Users scrolling rapidly perceive the brief blank-skeleton moment as "data disappeared and came back". `FcSlowQueryNotice` fires only for > `SlowQueryThresholdMs` queries, not for normal eviction re-fetches. Owning story: **Epic 5 "Reliable real-time experience"** (SignalR + read-ahead + scroll-velocity-based cache-retention heuristic). - **Defensive-finally dispatch visual noise during tab-close (NEW 2026-04-23 late-evening per elicitation customer-support-theater).** D3's defensive `finally` may dispatch `LoadPageFailedAction(viewKey, skip, "effect exited without terminal dispatch")` during a store-disposal race (user closes the tab mid-fetch). The orphan-prevention win is WORTH the occasional flash, but acknowledge the visual. Workaround options for adopters who find the flash distracting: (a) gate toast routing on `!action.ErrorMessage.StartsWith("effect exited")`; (b) gate on `!IsStoreDisposing` once Fluxor exposes such a flag (Fluxor v6+ candidate); (c) accept. Document as known minor visual in adopter release notes. NOT a blocker.
