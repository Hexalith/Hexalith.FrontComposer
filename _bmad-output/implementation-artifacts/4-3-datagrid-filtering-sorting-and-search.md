# Story 4.3: DataGrid Filtering, Sorting & Search

Status: done

> **Epic 4 § 100–142** · **FR12** · **UX-DR40** (DataGrid filtering surfaces) · **UX-DR41** (filter persistence + empty-filtered state) · **NFR87** (first-task completion < 30s with zero training) · applies lessons **L01, L02, L03, L04, L06, L07, L09, L10, L11**. Consumes the `GridViewSnapshot` / `CaptureGridStateAction` / `RestoreGridStateAction` reducer contract frozen in Story 2-2 D30 and the persistence effects landed in Story 3-6 ADR-050; consumes Story 1-5's `ColumnModel.TypeCategory` routing; consumes Story 4-1's role-dispatched `Emit{Strategy}Body` shell and Story 4-2's `FcStatusBadge` + `SlotAppearanceTable` for status-chip rendering. Provides the FIRST shipped PRODUCER of `CaptureGridStateAction` — turning the 2-2 / 3-6 "half-a-state-machine" into an end-to-end filter/sort/search loop across sessions.

## Executive summary (Feynman-level, ~45 sec)

Today every generated projection view renders `state.Items.AsQueryable()` into a bare `FluentDataGrid` with no filter UI, no sort UI, no search box, and no persistence write-path. Story 2-2 shipped a reducer that captures `GridViewSnapshot` (scroll + filters + sort + expanded row) but nothing dispatches `CaptureGridStateAction`. Story 3-6 shipped the persistence effects (`DataGridNavigationEffects.HandleCaptureGridState` + `HandleRestoreGridState`) but no renderer calls them. Story 4-3 is where the loop closes: **the generated view dispatches on filter/sort/search change, the reducer updates `ViewStates`, the effect debounces and persists, and the next visit restores.**

1. **Column filters — `FluentSearch` in every column header.** `RazorEmitter` emits a `FluentSearch` into the `HeaderCellItemTemplate` of every `PropertyColumn` whose `TypeCategory ∈ { Text, Numeric, Enum, DateTime }` (Boolean and Collection get no filter affordance — D14 + HFC1027 Information for the Collection case). Typing into the search box fires a 300 ms debounce (`TimeProvider`-anchored cancellation-token pattern mirroring Story 3-6 D16) that dispatches `ColumnFilterChangedAction(viewKey, columnKey, filterValue)`. The reducer updates `GridViewSnapshot.Filters`; the effect persists.
2. **Status filter chips.** When the projection has at least one enum column with `[ProjectionBadge]` mappings (reusing Story 1-4's `ColumnModel.BadgeMappings` IR field), the emitter renders a `<FcStatusFilterChips>` strip ABOVE the grid. The component loops over the DISTINCT `BadgeSlot` values present in `state.Items` and renders one `FluentBadge` toggle chip per slot, reusing Story 4-2's `SlotAppearanceTable` for color. Toggling a chip flips `Appearance.Tint` (inactive) ↔ `Appearance.Filled` (active) and dispatches `StatusFilterToggledAction(viewKey, slotName)`. Multiple slots can be active simultaneously (OR semantics).
3. **Global search — adopter hook, not framework feature.** The generated view resolves `IProjectionSearchProvider<T>` via nullable DI (`IServiceProvider.GetService<...>()`). When the service is registered, a `<FcProjectionGlobalSearch>` renders below the view title; when absent, the component is OMITTED from the render tree — no empty search box, no placeholder (per UX-DR40's "framework hook" design and Story 3-5 D15 nullable-DI precedent). Typing into the search box fires the same 300 ms debounce and dispatches `GlobalSearchChangedAction(viewKey, query)`. The adopter's provider implements `Task<IReadOnlyList<T>> SearchAsync(string query, CancellationToken)` — 4-3 wires the hook; it does NOT ship a reference full-text implementation.
4. **Server-side routing via `QueryRequest` extensions.** Column filters, status filters, search query, and sort order all travel through the existing `QueryRequest` record as additive nullable fields (`ColumnFilters`, `StatusFilters`, `SearchQuery`, `SortColumn`, `SortDescending`). The `Filter` legacy field (Story 1-3 stub) is `[Obsolete]` soft-deprecated per D18. ETag caching continues to work unchanged — query parameters are part of the cache key composition already per NFR19.
5. **Persistence via the 2-2 / 3-6 pipeline.** Every filter/sort/search change dispatches `CaptureGridStateAction(viewKey, snapshot)` where `viewKey` = `"{boundedContext}:{projectionTypeFqn}"` (Story 2-2 D30 format). Story 3-6 `DataGridNavigationEffects.HandleCaptureGridState` takes over with its 250 ms per-viewKey debounce; writes land in LocalStorage under `{tenantId}:{userId}:datagrid:{viewKey}`. On view mount, `{TypeName}View.OnInitialized` dispatches `RestoreGridStateAction(viewKey)`; 3-6's effect reads the blob and dispatches `GridViewHydratedAction` which the 2-2 reducer applies before the first data fetch.
6. **Filter visibility + reset + empty-filtered state.** `FcFilterSummary` renders a row below the grid header when any filter is active: `"Filtered: Status = Pending, Approved | Name = \"foo\" | 12 of 47 orders"`. A `FcFilterResetButton` (Outline appearance) dispatches `FiltersResetAction(viewKey)` which the 2-2 reducer translates to clearing the snapshot's Filters/SearchQuery/StatusFilters AND 3-6's `HandleClearGridState` effect removes the persisted blob. `FcFilterEmptyState` activates when filters are active AND `state.Items.Count == 0` AND `state.TotalCount > 0` — distinct from Story 4-1's `FcProjectionEmptyPlaceholder` which handles the zero-total-items case (Story 4-6 takes that over later).
7. **Keyboard affordance.** A new `/` shortcut registers via Story 3-4's `IShortcutService`; the handler is scope-gated (only fires when `document.activeElement` is inside a DataGrid — matches Story 3-4 D12 focus-scope pattern) and focuses the first column filter input in the active grid.
8. **Schema stability via reserved-key packing.** `GridViewPersistenceBlob` from Story 3-6 has a `Filters: Dictionary<string, string>` field — instead of extending the blob schema (which would trip `GridViewPersistenceBlobSchemaLockedTests.BlobSerializesToExpectedJsonShape`), 4-3 packs status filters and global search into the same dictionary under reserved keys (`__status` = CSV of active slot names, `__search` = search query). No schema bump, no blob migration; the 3-6 contract stays frozen per D3.

The design honors the "capability contract" principle: Fluxor state shape unchanged (reducer consumes additional actions, never new fields), persistence blob unchanged, diagnostic IDs reserved up-front (HFC1027), decisions live inside the L06 feature-story cap (≤ 25 — 4-3 targets 22), and every cross-story seam is explicit (L01). Scope is deliberately narrow: **filters/sort/search + persistence — nothing else.** Virtual scrolling (4.4), expand-in-row (4.5), empty states with CTAs (4.6), `FcColumnPrioritizer` (4.4), and `FcNewItemIndicator` (FR21 / 3-5 follow-up) are OUT of scope.

**Binding contract(s) with predecessor stories:**

- **Story 2-2 contract** frozen: `GridViewSnapshot` shape UNCHANGED. `Filters` dictionary is the single strings-to-strings map; status filters + search query are packed as reserved-key entries (`__status`, `__search`) per D3. `CaptureGridStateAction` is the ONLY production write path — filter reducers chain to it, they do not bypass it.
- **Story 3-6 contract** consumed: `DataGridNavigationEffects` handles all IO (`HandleCaptureGridState` / `HandleRestoreGridState` / `HandleClearGridState`). Story 4-3 does NOT add new effects — it adds reducers that chain into existing action plumbing. Storage key format `{tenantId}:{userId}:datagrid:{viewKey}` unchanged.
- **Story 3-6 contract** frozen: `GridViewPersistenceBlob` SCHEMA UNCHANGED. `GridViewPersistenceBlobSchemaLockedTests.BlobSerializesToExpectedJsonShape` continues to pass — no field addition, no field rename (D3 L01 discipline).
- **Story 4-1 contract** consumed: `Emit{Strategy}Body` helpers render the Data branch body. 4-3 injects filter UI ABOVE the grid body (as a sibling of the data body, inside the Data branch). Role strategies render unchanged — filters are universal to every strategy. Default / ActionQueue / StatusOverview / DetailRecord / Timeline / Dashboard all emit the same filter surface. DetailRecord and Timeline filter the `state.Items` collection prior to rendering, same as Default/ActionQueue.
- **Story 4-2 contract** consumed: `FcStatusBadge` is NOT reused directly for chips — chips use raw `FluentBadge` with `Appearance=Filled|Tint` toggling. However, `SlotAppearanceTable` IS reused as the color source; importing it into `FcStatusFilterChips` means a future addition of a slot to `SlotAppearanceTable` flows into chip rendering automatically. L01 cross-story contract honored.
- **Story 1-4 / 1-5 contract** consumed: `ColumnModel.BadgeMappings` and `ColumnModel.TypeCategory` are read by the emitter. 4-3 does NOT modify either; it adds a derived `SupportsFilter` property that wraps `TypeCategory` routing (additive on `ColumnModel`; no ctor arg added — derived from existing data).
- **Story 3-4 contract** consumed: `IShortcutService` for the `/` binding. Conflict handling per Story 3-4 D3 last-writer-wins; HFC2108 logs if an adopter re-registers `/`.
- **Story 3-5 contract** unchanged: `IBadgeCountService` is NOT consumed by 4-3 (count badges vs status badges are semantically distinct — Story 4-2 already clarified). Status chips read their data from `state.Items`, not from the count service.

**Scope guardrails (what NOT to implement in 4-3):**

- ❌ **Virtual scrolling** (Story 4.4) — generated view continues to render the full `Items` collection. `<Virtualize>` and `ItemsProvider` land in 4.4. Filter UI must be cheap per-render (debounce anchors are component-scoped, not per-row).
- ❌ **`FcColumnPrioritizer`** (Story 4.4) — column visibility toggling is not in 4-3 scope. Filter UI renders for every visible column regardless of priority.
- ❌ **Expand-in-row detail** (Story 4.5) — `ExpandedRowId` field of `GridViewSnapshot` is unchanged by 4-3; 4.5 is the producer.
- ❌ **`FcEmptyState` CTA** (Story 4.6) — 4-3's `FcFilterEmptyState` shows when FILTERS are active; 4-6 replaces Story 4-1's `FcProjectionEmptyPlaceholder` for the zero-total-items case. The two components are visually distinct and serve different semantics.
- ❌ **Field descriptions as tooltips** (Story 4.6) — column-header tooltips from `[Description]` attributes are Story 4.6.
- ❌ **`FcFieldPlaceholder`** (Story 4.6) — unsupported-type rendering is 4.6.
- ❌ **Bounded-context-scoped global search** — UX-DR40's "global search" is per-projection (adopter-implemented); cross-context search is Story 3-4's command palette (already shipped).
- ❌ **Full-text search reference implementation** — 4-3 ships the interface + wiring only. No Elasticsearch adapter, no Postgres FTS adapter.
- ❌ **SignalR-triggered refresh of filter state** (Epic 5) — filter change → query happens; SignalR reconciliation layers on top in Epic 5.
- ❌ **Filter capability override via Epic 6 customization** — Epic 6 Slot-level replacement is the path for adopters who need non-standard filter UI (e.g., date range picker on a DateTime column); 4-3 does NOT ship the override mechanism.
- ❌ **Query parameter ETag cache invalidation semantics** (Epic 5) — 4-3 emits a query with filter/sort/search; the cache layer treats the full `QueryRequest` as the cache key composition per existing NFR19. 4-3 does NOT modify the cache.
- ❌ **Boolean column filters (3-state: Yes/No/All)** — the value space is tiny enough that a dedicated chip would make sense, but 4-3 defers to keep the Boolean column lane untouched. Adopters who need Boolean filtering implement it via Epic 6 (documented in Dev Notes).
- ❌ **Nested-property column filters (`Order.Customer.Name`)** — 4-3 filters on declared projection properties only; Story 1-5 already restricts columns to direct properties. Not in scope.
- ❌ **Localized filter operators (`contains`, `starts with`, `=`, `≠`)** — 4-3 uses a single "contains, case-insensitive" semantic for Text and Enum; `≥` for Numeric and DateTime (lower-bound threshold); exact-match for Enum values selected from a dropdown is Story 4.4 territory. For v1 the column filter is a substring match routed to the server — the server decides exact operator semantics.

## Story

As a business user,
I want to filter, sort, and search within DataGrid views — with column filters, status toggle chips, and an optional global search — and have my preferences remembered across sessions,
so that I can quickly find the items I need without re-applying my filters every time.

**Business value:** Until 4-3 lands, every generated DataGrid requires the business user to eyeball-scan 47 rows to find the three `Pending` orders they care about. After 4-3 ships, the user clicks a Pending status chip once and the grid collapses to 3 rows; next session the chip is already active. For a credit analyst triaging a queue or a fulfillment ops lead chasing exceptions, this is the difference between "I can use this" and "I need a custom view". NFR87 (first-task completion < 30s with zero training) is UNREACHABLE without 4-3. It is also the story that finally operationalizes Story 2-2's reducer-only state and Story 3-6's effect-only persistence — 4-3 is the producer that turns the half-a-state-machine into a closed loop. Every generated view, across every role strategy from 4-1, gains filter/sort/search for free.
