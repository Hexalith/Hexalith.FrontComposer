# Story 9.1: Build-Time Drift Detection

Status: ready-for-dev

> **Epic 9** - Developer Tooling & Documentation. Covers **FR7**, **NFR8**, **NFR97**, and the first build-time handoff toward Stories **9-2**, **9-4**, and **9-5**. Builds on the existing SourceTools incremental generator, HFC diagnostic catalog discipline, Story 1-8 hot-reload/rebuild findings, Story 6-2/6-3/6-4 customization contract-version patterns, and Story 8-6 schema fingerprinting boundaries. Applies lessons **L01**, **L06**, **L07**, **L08**, **L10**, and **L14**.

---

## Executive Summary

Story 9-1 adds a build-time drift detector for FrontComposer generated UI contracts:

- Compare the current parsed domain declaration shape against a trusted generated-UI baseline snapshot.
- Emit actionable HFC diagnostics when projection properties, command form fields, bounded-context names, renderer-impacting metadata, or known generated contract expectations drift.
- Keep the detector inside the SourceTools incremental pipeline and under the NFR8 < 500 ms rebuild budget.
- Avoid runtime reflection and runtime environment data; drift evidence is compile-time structural metadata only.
- Leave CLI inspection/migration UX to Story 9-2, full diagnostic documentation governance to Story 9-4, and public docs-site publication to Story 9-5.

---

## Story

As a developer,
I want the framework to detect mismatches between my backend domain declarations and the generated UI at build time,
so that I catch breaking changes as compile-time diagnostics instead of discovering them as silent runtime bugs.

### Adopter Job To Preserve

An adopter should be able to rename, remove, add, or retag a `[Projection]`, `[Command]`, or `[BoundedContext]` declaration and receive a precise build diagnostic explaining what changed, what generated UI behavior is affected, and how to reconcile the baseline. The build should teach the fix without requiring the adopter to run the app, click through generated pages, or inspect `obj/` output by hand.

## Acceptance Criteria

| AC | Given | When | Then |
| --- | --- | --- | --- |
| AC1 | A `[Projection]`-annotated type has a trusted prior generated-UI baseline | The source generator parses the current type shape | It compares current structural metadata with the baseline and emits compile-time drift diagnostics for mismatches. |
| AC2 | A projection property is removed | The build runs | A diagnostic identifies the projection type, removed property name, affected UI surface, and docs link; the default severity is Warning unless configured stricter. |
| AC3 | A projection property is added | The build runs | A diagnostic identifies the added property and states whether it creates a new DataGrid column, detail field, MCP projection field metadata, or unsupported placeholder. |
| AC4 | A likely rename occurs, with one removed property and one added property of compatible category in the same declaration | The build runs | Rename diagnostics are emitted only when the comparison can deterministically match exactly one removed member and one added member within the same stable declaration identity. Ambiguous matches are reported as separate add/remove drift. The message follows the Epic 9 wording shape: `Property '{OldName}' was expected on {TypeName} but not found. '{NewName}' was added. If this is a rename, update the generated output. See HFC{id}.` |
| AC5 | A projection or command property type/category changes, such as `string` to `int` or nullable to non-nullable | The build runs | A Warning diagnostic states that generated form input, DataGrid column, filter behavior, badge/format behavior, or MCP descriptor metadata may change rendering behavior. |
| AC6 | A `[BoundedContext]` name changes | The build runs | A diagnostic states that navigation grouping, generated registration, persisted session state, MCP resource grouping, and badge/action-queue grouping may be affected. |
| AC7 | Display metadata changes, such as `[Display(Name)]`, `[Display(GroupName)]`, `[Description]`, `[ProjectionRole]`, `[ProjectionFieldGroup]`, `[ColumnPriority]`, `[RelativeTime]`, `[Currency]`, `[ProjectionEmptyStateCta]`, `[ProjectionBadge]`, `[Icon]`, `[Destructive]`, or `[RequiresPolicy]` | The build runs | Drift is classified by impact category and emits at most one bounded diagnostic per affected declaration and category. |
| AC8 | A trusted baseline is missing for a project that opted into drift detection | The build runs | The detector emits a single actionable baseline-missing diagnostic that distinguishes first-run/no-baseline state from invalid configured baseline paths and explains the 9-1 manual reconciliation path plus the future Story 9-2 CLI owner. |
| AC9 | A baseline exists but is empty, malformed, too large, duplicate-keyed, invariant-violating, generated by an unsupported schema version, or produced by an unsupported algorithm | The build runs | The detector fails closed with deterministic Error diagnostics, suppresses unsafe partial drift comparison, and never silently passes unverifiable content. |
| AC10 | Drift detection runs in an unchanged compilation | Incremental generator tracking is enabled | Existing `Parse`, `ParseCommand`, and template parse outputs remain Cached/Unchanged; unchanged `AdditionalText` baseline content and unrelated file edits do not re-run domain parsing or unrelated generated outputs. |
| AC11 | Drift detection runs after adding or modifying one domain property | Performance is measured | Median incremental rebuild added overhead remains below the NFR8 500 ms budget on a representative bounded fixture with warmup excluded, repeated measured iterations, and recorded median/p95 for cache-hit and cache-miss paths. |
| AC12 | The detector emits diagnostics | A developer views the compiler output | Each message includes stable What, Expected, Got, Fix, and DocsLink fields or structured equivalent, uses HFC SourceTools IDs, carries `DiagnosticDescriptor.HelpLinkUri`, and exposes bounded properties such as `BaselinePath`, `DeclarationPath`, `DeclarationName`, `MemberName`, `DriftKind`, `ExpectedShapeHash`, `ActualShapeHash`, `SchemaVersion`, and `AlgorithmVersion` when available. |
| AC13 | Drift affects hidden, tenant-scoped, policy-gated, MCP-only, or user-authored baseline metadata | The build runs | Diagnostics are structural and sanitized; they never include tenant IDs, user IDs, claims, tokens, command payload values, query rows, cache keys, ETags, local absolute paths, raw JSON fragments, generated source snippets, oversized arbitrary text, or raw exception text. |
| AC14 | A trim-enabled or native-AOT host relies on the default reflection `IActionQueueProjectionCatalog` | The analyzer can see the project setting and missing adopter override evidence | A build-time diagnostic warns that the reflection catalog is trim-incompatible and points to the source-generated catalog/override path. |
| AC15 | A host policy catalog check cannot be proven at build time | The build runs | The story records the limitation explicitly; runtime validators remain authoritative and the build-time diagnostic is emitted only where evidence is reliable. |
| AC16 | The detector emits many changes in a large projection or command | Output is generated | Diagnostics are sorted by bounded context, declaration kind, declaration name, member name, then drift kind using ordinal comparison; the first 50 drift diagnostics are emitted after sorting, followed by one deterministic truncation summary that reports the omitted count. |
| AC17 | A baseline update is intentional | The developer follows the documented workflow | The story provides an internal deterministic comparison service/result model that Story 9-2 can wrap with CLI inspect/migration commands, but Story 9-1 does not ship public CLI verbs, command UX, code fixes, source rewrites, or a public API shape. |
| AC18 | Existing generated code is compiled after drift detection changes | The regression suite runs | Normal builds are diagnostics-only and do not rewrite trusted baselines or alter generated UI output. Existing projection/command generation remains byte-stable across repeated runs and equivalent input ordering, unless an explicitly invoked future baseline-generation path changes metadata. |
| AC19 | Drift detection runs under non-default cultures such as `fr-FR` or `tr-TR` | Diagnostics and generated output are compared | Classification, sorting, hashes, numeric formatting, diagnostics, and generated bytes remain culture-invariant unless localization is explicitly introduced by a later story. |

## Dev Notes

### Existing SourceTools State

- `FrontComposerGenerator` is already an `IIncrementalGenerator` using `SyntaxProvider.ForAttributeWithMetadataName` for `[Projection]`, `[Command]`, and `[ProjectionTemplate]`. Preserve this shape; drift detection should add comparison inputs, not replace the parse pipeline.
- Current generator fan-out:
  - Projection: Razor view, Fluxor feature, actions, reducers, registration.
  - Command: form, actions, lifecycle feature, registration, renderer, last-used subscriber, lifecycle bridge, optional full-page route.
  - Shared: projection template manifest and MCP manifest.
- Existing tracked step names include `Parse`, `ParseCommand`, and `ParseProjectionTemplate`; benchmark and cache tests assert these names.
- `DomainModel`, `CommandModel`, and `PropertyModel` are pure data and already include most fields needed for drift comparison. Some fields such as `SourceFilePath`, `SourceLine`, and `SourceColumn` are useful for locations but must not become baseline identity.

### Diagnostic ID Discipline

- SourceTools diagnostics live in HFC1000-HFC1999.
- Current `AnalyzerReleases.Unshipped.md` allocates through HFC1057.
- `DiagnosticCatalogTests` reflect over `DiagnosticDescriptors` and `FcDiagnosticIds`; update both when adding new IDs.
- Several older generator descriptors do not yet carry `HelpLinkUri`; Story 9-1 should set `HelpLinkUri` on new descriptors and avoid broad churn to older IDs unless needed.
- HFC1010 already exists as the hot-reload/full-rebuild guidance ID; do not reuse it for general drift. Either route only the documented full-rebuild categories through HFC1010 or allocate new drift-specific IDs.

### Baseline Contract

Use one structural baseline source of truth. Recommended shape:

```json
{
  "schemaVersion": "frontcomposer.generated-ui-baseline.v1",
  "algorithm": "frontcomposer-structural-v1",
  "contracts": [
    {
      "family": "projection",
      "type": "Acme.Shipping.ShipmentProjection",
      "boundedContext": "Shipping",
      "displayName": "Shipments",
      "role": "ActionQueue",
      "properties": [
        {
          "name": "Priority",
          "category": "Enum",
          "nullable": false,
          "displayName": "Priority",
          "description": null,
          "columnPriority": 1,
          "fieldGroup": null,
          "displayFormat": "Default"
        }
      ]
    }
  ]
}
```

The final JSON schema can differ, but it must remain deterministic, reviewable, and free of runtime values.

Baseline ownership and trust boundary:

- Story 9-1 owns the SourceTools baseline schema models, parser, validator, comparison result model, and diagnostics.
- The trusted baseline is a checked-in or analyzer-configured `AdditionalText` artifact. Story 9-1 must not read `obj/`, generated source text, application runtime services, environment values, live tenant/user state, or reflection output as trust evidence.
- Baseline identity is structural: schema version, algorithm version, contract family, bounded context, namespace/type name, and member identity. Source file paths and line/column values may locate diagnostics but must not define trust identity.
- Unknown schema, unknown algorithm, malformed JSON, empty/whitespace content, duplicate identities, invariant violations, and oversized content are trust failures. They emit deterministic Error diagnostics and suppress downstream comparison for that baseline.
- Missing baseline is not the same failure as invalid baseline. A missing baseline gets one actionable first-run/reconciliation diagnostic; an existing invalid baseline gets the highest-precedence trust-failure diagnostic.
- Story 9-1 may generate deterministic metadata needed by SourceTools, but it must not automatically rewrite trusted checked-in baselines during normal builds. Story 9-2 owns the public baseline inspect/update workflow.

### Developer Workflow

During build, SourceTools compares current declarations against the checked-in structural baseline. If drift is found, diagnostics identify the changed declaration, drift kind, affected generated UI behavior category, and the reconciliation action available in Story 9-1. In Story 9-1 the developer can inspect or manually update the checked-in baseline artifact; Story 9-2 owns the public CLI workflow for inspecting, updating, dry-running, or migrating baselines.

Allowed 9-1 affected-behavior labels are bounded to structural generated UI contract categories:

- Projection surface: generated route/view, DataGrid column, detail field, filter/sort/search behavior, role/badge/format/priority/group/empty-state metadata, unsupported placeholder, and MCP projection field metadata.
- Command surface: generated form field, input category, validation/rejection metadata, destructive/icon/density metadata, lifecycle bridge metadata, and command registration.
- Bounded-context surface: navigation grouping, generated registration, persisted session grouping, MCP resource grouping, and badge/action-queue grouping.
- Template/customization surface: template, slot, view descriptor, and generated contract expectation metadata already present in SourceTools IR.

### Diagnostic Contract

Story 9-1 should reserve a contiguous HFC SourceTools range after HFC1057 unless Story 9-4 has already assigned the final range. Suggested names are placeholders; implementation should follow the current repository naming convention.

| Precedence | Category | Default severity | Behavior |
| --- | --- | --- | --- |
| 1 | Missing baseline | Warning | Emit one actionable first-run/reconcile diagnostic; no drift comparison for the missing baseline. |
| 2 | Empty or malformed baseline | Error | Fail closed and suppress drift comparison. |
| 3 | Unsupported schema version | Error | Fail closed and suppress drift comparison. |
| 4 | Unsupported algorithm version | Error | Fail closed and suppress drift comparison. |
| 5 | Oversized baseline or exceeded shape bounds | Error | Fail closed and suppress drift comparison. |
| 6 | Duplicate identity or invariant violation | Error | Fail closed and suppress drift comparison. |
| 7 | Structural drift | Warning unless configured stricter | Emit deterministic add/remove/rename/type/nullability diagnostics. |
| 8 | Metadata or renderer-impacting drift | Warning unless configured stricter | Emit one bounded diagnostic per declaration/category. |
| 9 | Trim/AOT reflection-catalog evidence | Warning/Info depending evidence strength | Warn only on statically reliable evidence; runtime validators remain authoritative. |
| 10 | Truncation summary | Warning | Emit after the first 50 sorted drift diagnostics and include omitted count. |

Diagnostic properties should be stable, bounded, and sanitized. Use these keys where applicable: `BaselinePath`, `DeclarationPath`, `DeclarationName`, `MemberName`, `DriftKind`, `ExpectedShapeHash`, `ActualShapeHash`, `SchemaVersion`, and `AlgorithmVersion`. Omit absent values or use a fixed sentinel; never emit raw JSON, generated source text, absolute local paths, secrets, payload values, or exception text.

### Comparison Seam

The Story 9-2 handoff is an internal deterministic SourceTools seam, not a CLI design. Keep comparison logic available as a pure internal service/function that accepts parsed declaration models, validated baseline models, and bounded options, then returns comparison results/diagnostic facts. Do not introduce public command verbs, console formatting, code fixes, source rewrites, or public package API shape in Story 9-1.

### Performance and Incrementality

Measure the NFR8 target as added incremental-generator overhead for drift detection on a representative bounded fixture, not as total clean build wall-clock time. Exclude warmup runs, record median and p95 where practical, and cover cache-hit and cache-miss paths. Keep one focused benchmark/perf guard plus lower-level incremental invalidation tests; do not create a broad machine-fragile performance suite.

### Architecture Constraints

- SourceTools targets `netstandard2.0` and references `Microsoft.CodeAnalysis.CSharp` with `PrivateAssets="all"`.
- The current package pin is `Microsoft.CodeAnalysis.CSharp` 4.12.0; do not move it inside this story unless a build break forces it.
- Implement Story 9-1 under `src/Hexalith.FrontComposer.SourceTools` generator/analyzer internals with tests under `tests/Hexalith.FrontComposer.SourceTools.Tests`. Do not add public CLI, runtime rendering package, Shell runtime, MCP runtime, or docs-site dependencies in this story.
- Prefer existing serializer/package patterns. Do not add runtime package dependencies or reflection-based baseline parsing for the detector.
- Microsoft Learn confirms `IIncrementalGenerator` instances should not store state directly; keep comparison state in incremental values, not generator instance fields.
- Microsoft Learn confirms `AdditionalTextsProvider` supplies non-code files to an incremental generator; use that for baselines.
- Microsoft Learn confirms `ForAttributeWithMetadataName` only transforms nodes carrying a matching attribute metadata name and notes partial-type behavior; partial declarations matter for drift and hot-reload classification.
- `DiagnosticDescriptor.HelpLinkUri` is the Roslyn API for diagnostic documentation links; use it on new descriptors.

### Scope Guardrails

Do not implement these in Story 9-1:

- Public CLI commands for inspecting or updating baselines. Owner: Story 9-2.
- Automated source rewrites, rename refactoring, code fixes, or migration tools. Owner: Story 9-2 / Story 9-4.
- Public comparison APIs, command-line output formatting, or baseline update UX. Owner: Story 9-2.
- Public diagnostics documentation site pages. Owner: Story 9-4 / Story 9-5.
- MCP runtime schema negotiation, client fingerprint comparison, or agent-visible compatibility responses. Owner: Story 8-6.
- Broad DI lifetime analysis, service-locator audits, policy-catalog whole-host proof, assembly scanning, runtime type inspection, or runtime authorization redesign. Owners: deferred follow-ups / Story 9-4.
- Visual specimen drift, accessibility screenshot gates, or generated UI screenshot baselines. Owner: Story 10-2.
- Mutation testing rollout. Owner: Story 10-4.
- A new baseline file watcher outside Roslyn incremental inputs.
