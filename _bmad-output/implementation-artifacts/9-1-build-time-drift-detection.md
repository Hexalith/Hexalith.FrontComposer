# Story 9.1: Build-Time Drift Detection

Status: ready-for-dev

> **Epic 9** - Developer Tooling & Documentation. Covers **FR7**, **NFR8**, **NFR97**, and the first build-time handoff toward Stories **9-2**, **9-4**, and **9-5**. Builds on the existing SourceTools incremental generator, HFC diagnostic catalog discipline, Story 1-8 hot-reload/rebuild findings, Story 6-2/6-3/6-4 customization contract-version patterns, and Story 8-6 schema fingerprinting boundaries. Applies lessons **L01**, **L06**, **L07**, **L08**, **L10**, and **L14**.

---

## Executive Summary

Story 9-1 adds a build-time drift detector for FrontComposer generated UI contracts:

- Compare the current parsed domain declaration shape against a trusted generated-UI baseline snapshot.
- Emit actionable HFC diagnostics when projection properties, command form fields, bounded-context names, renderer-impacting metadata, or known generated contract expectations drift.
- Keep the detector inside the SourceTools incremental pipeline and under the NFR8 < 500 ms rebuild budget.
- Avoid runtime reflection and runtime environment data; drift evidence is compile-time structural metadata only.
- Leave CLI inspection/migration UX to Story 9-2, full diagnostic documentation governance to Story 9-4, and public docs-site publication to Story 9-5.

---

## Story

As a developer,
I want the framework to detect mismatches between my backend domain declarations and the generated UI at build time,
so that I catch breaking changes as compile-time diagnostics instead of discovering them as silent runtime bugs.

### Adopter Job To Preserve

An adopter should be able to rename, remove, add, or retag a `[Projection]`, `[Command]`, or `[BoundedContext]` declaration and receive a precise build diagnostic explaining what changed, what generated UI behavior is affected, and how to reconcile the baseline. The build should teach the fix without requiring the adopter to run the app, click through generated pages, or inspect `obj/` output by hand.

## Acceptance Criteria

| AC | Given | When | Then |
| --- | --- | --- | --- |
| AC1 | A `[Projection]`-annotated type has a trusted prior generated-UI baseline | The source generator parses the current type shape | It compares current structural metadata with the baseline and emits compile-time drift diagnostics for mismatches. |
| AC2 | A projection property is removed | The build runs | A diagnostic identifies the projection type, removed property name, affected UI surface, and docs link; the default severity is Warning unless configured stricter. |
| AC3 | A projection property is added | The build runs | A diagnostic identifies the added property and states whether it creates a new DataGrid column, detail field, MCP projection field metadata, or unsupported placeholder. |
| AC4 | A likely rename occurs, with one removed property and one added property of compatible category in the same type | The build runs | The diagnostic message follows the Epic 9 wording shape: `Property '{OldName}' was expected on {TypeName} but not found. '{NewName}' was added. If this is a rename, update the generated output. See HFC{id}.` |
| AC5 | A projection or command property type/category changes, such as `string` to `int` or nullable to non-nullable | The build runs | A Warning diagnostic states that generated form input, DataGrid column, filter behavior, badge/format behavior, or MCP descriptor metadata may change rendering behavior. |
| AC6 | A `[BoundedContext]` name changes | The build runs | A diagnostic states that navigation grouping, generated registration, persisted session state, MCP resource grouping, and badge/action-queue grouping may be affected. |
| AC7 | Display metadata changes, such as `[Display(Name)]`, `[Display(GroupName)]`, `[Description]`, `[ProjectionRole]`, `[ProjectionFieldGroup]`, `[ColumnPriority]`, `[RelativeTime]`, `[Currency]`, `[ProjectionEmptyStateCta]`, `[ProjectionBadge]`, `[Icon]`, `[Destructive]`, or `[RequiresPolicy]` | The build runs | Drift is classified by impact category and emits at most one bounded diagnostic per affected declaration and category. |
| AC8 | A trusted baseline is missing for a project that opted into drift detection | The build runs | The detector emits a single actionable baseline-missing diagnostic that explains how to generate or accept the baseline without failing open silently. |
| AC9 | A baseline exists but is malformed, too large, generated by an unsupported schema version, or produced by an unsupported algorithm | The build runs | The detector fails closed with a bounded diagnostic and does not compare unverifiable content. |
| AC10 | Drift detection runs in an unchanged compilation | Incremental generator tracking is enabled | Existing `Parse`, `ParseCommand`, and template parse outputs remain Cached/Unchanged; drift-specific inputs do not force full regeneration. |
| AC11 | Drift detection runs after adding or modifying one domain property | Performance is measured | Median incremental rebuild remains below the NFR8 500 ms budget on the existing benchmark harness. |
| AC12 | The detector emits diagnostics | A developer views the compiler output | Each message includes What, Expected, Got, Fix, and DocsLink fields or structured equivalent, using HFC SourceTools IDs and `DiagnosticDescriptor.HelpLinkUri` where possible. |
| AC13 | Drift affects hidden, tenant-scoped, policy-gated, or MCP-only generated metadata | The build runs | Diagnostics are structural and sanitized; they never include tenant IDs, user IDs, claims, tokens, command payload values, query rows, cache keys, ETags, local absolute paths, or raw exception text. |
| AC14 | A trim-enabled or native-AOT host relies on the default reflection `IActionQueueProjectionCatalog` | The analyzer can see the project setting and missing adopter override evidence | A build-time diagnostic warns that the reflection catalog is trim-incompatible and points to the source-generated catalog/override path. |
| AC15 | A host policy catalog check cannot be proven at build time | The build runs | The story records the limitation explicitly; runtime validators remain authoritative and the build-time diagnostic is emitted only where evidence is reliable. |
| AC16 | The detector emits many changes in a large projection or command | Output is generated | Diagnostics are bounded, deterministic, sorted, and capped with an explicit truncation category; no unbounded diff output is produced. |
| AC17 | A baseline update is intentional | The developer follows the documented workflow | The story provides the library/test seam that Story 9-2 can wrap with CLI inspect/migration commands, but Story 9-1 does not ship a public CLI. |
| AC18 | Existing generated code is compiled after drift detection changes | The regression suite runs | Existing projection/command generation remains byte-stable unless the story intentionally changes emitted baseline metadata; all generated code still compiles. |

## Tasks / Subtasks

- [ ] T1. Define the drift baseline contract (AC1-AC9, AC13, AC16)
  - [ ] Add SourceTools-owned baseline models for projection, command, bounded-context, renderer-impacting metadata, and generated artifact expectations.
  - [ ] Keep models SDK-neutral and runtime-data-free. Do not reference Shell components, Fluent UI types, EventStore clients, MCP SDK DTOs, tenant/user services, or runtime reflection outputs.
  - [ ] Include only structural fields already present in `DomainModel`, `CommandModel`, `PropertyModel`, `RazorModel`, `CommandFormModel`, `RegistrationModel`, and MCP descriptor transforms.
  - [ ] Include a baseline schema version and algorithm identifier so unsupported baselines fail closed instead of comparing incompatible content.
  - [ ] Cap maximum baseline file size, declaration count, property count, nesting depth, and diagnostic count through explicit options.

- [ ] T2. Load baselines through the incremental pipeline (AC8-AC11, AC16)
  - [ ] Use Roslyn `AdditionalTextsProvider` / analyzer config inputs for checked-in baseline files rather than ad hoc file IO in emitters.
  - [ ] Keep `ForAttributeWithMetadataName` discovery for `[Projection]`, `[Command]`, and `[ProjectionTemplate]`; drift inputs are an additional provider, not a replacement parse path.
  - [ ] Ensure baseline changes invalidate only drift comparison and shared generated baseline metadata, not unrelated projection/command outputs.
  - [ ] Add tests proving unrelated file edits keep existing tracked steps Cached/Unchanged.
  - [ ] Add malformed, unsupported-version, oversized, and missing-baseline fixtures.

- [ ] T3. Implement structural comparison and classification (AC1-AC7, AC13, AC16)
  - [ ] Compare by stable declaration identity: namespace + type name + contract family + bounded context, with explicit behavior for global namespace.
  - [ ] Classify property added, removed, likely rename, type/category change, nullability change, display/description change, enum member/badge change, role/group/priority change, command density change, derivable/non-derivable split change, authorization policy change, destructive/icon metadata change, and bounded-context name change.
  - [ ] Reuse existing type category and transform logic instead of duplicating field inference rules.
  - [ ] Treat likely rename as a heuristic only; never rewrite source or update baselines automatically in this story.
  - [ ] Sort diagnostics by type identity, category, then property name using ordinal comparison.

- [ ] T4. Add HFC diagnostics and catalog discipline (AC2-AC9, AC12, AC14-AC16)
  - [ ] Reserve a contiguous SourceTools ID block after the current HFC1057 allocation, unless Story 9-4 has already assigned a final range.
  - [ ] Add descriptors to `DiagnosticDescriptors`, constants to `FcDiagnosticIds`, and rows to `AnalyzerReleases.Unshipped.md`.
  - [ ] Use `DiagnosticDescriptor.HelpLinkUri` for each new diagnostic where supported by the current Microsoft.CodeAnalysis package.
  - [ ] Ensure every diagnostic message carries What, Expected, Got, Fix, and DocsLink fields or equivalent structured parameters.
  - [ ] Add catalog tests proving IDs are unique, shaped as HFCxxxx, documented in release notes, and linked to canonical diagnostics docs.

- [ ] T5. Emit or refresh baseline metadata for generated UI contracts (AC1, AC3, AC5, AC7, AC17, AC18)
  - [ ] Define the checked-in baseline file naming and folder contract. Prefer a deterministic SourceTools-owned location under the adopter project, not `obj/`.
  - [ ] Generate baseline material from the same parse/transform data that produces Razor, Fluxor, registration, MCP manifests, and template/slot/view descriptor metadata.
  - [ ] Do not hash or diff generated C# source text; compare structural metadata so formatting or banner churn does not create false positives.
  - [ ] Preserve existing generated output hint names: namespace-qualified projection hints and `.Command` command hint prefixes.
  - [ ] Document that Story 9-2 owns the user-facing CLI command to inspect or update these baselines.

- [ ] T6. Add trim/AOT reflection-catalog diagnostics where evidence is reliable (AC14, AC15)
  - [ ] Detect `PublishTrimmed=true` / native AOT project settings only through analyzer config/build metadata available to SourceTools.
  - [ ] Warn when the default `ReflectionActionQueueProjectionCatalog` path appears to be used without a source-generated or adopter-supplied `IActionQueueProjectionCatalog`.
  - [ ] If adopter override evidence is not statically knowable, emit guidance as Warning/Info only and leave runtime validation as authoritative.
  - [ ] Keep this diagnostic narrow to the documented Story 3-5 G22 gap; do not start a broad DI-lifetime analyzer in this story.

- [ ] T7. Tests and verification (AC1-AC18)
  - [ ] Unit tests for baseline parsing, schema-version rejection, unsupported algorithm, oversized/truncated output, missing baseline, and malformed baseline.
  - [ ] Classification tests for property add/remove/rename/type/nullability, bounded-context change, display metadata, role/group/priority, enum/badge, command density, derivable split, destructive/icon, and policy metadata changes.
  - [ ] Integration tests using `CSharpGeneratorDriver` and `AdditionalText` baselines proving diagnostics appear at the correct source location and have `HelpLinkUri`.
  - [ ] Incremental caching tests proving unrelated edits do not invalidate existing generator outputs.
  - [ ] Performance tests extending `IncrementalRebuildBenchmarkTests` or a sibling benchmark to keep median delta below 500 ms.
  - [ ] Regression tests proving generated Razor/Fluxor/registration/MCP output still compiles after adding drift detection.
  - [ ] Redaction tests with tenant/user/token/path/ETag/payload sentinels in source and baseline fixtures.
  - [ ] Targeted suite: `tests/Hexalith.FrontComposer.SourceTools.Tests/Diagnostics`, `.../Incremental`, `.../Integration`, and existing emitter snapshot tests only when intentionally touched.
  - [ ] Full regression: `dotnet build Hexalith.FrontComposer.sln -p:TreatWarningsAsErrors=true -p:UseSharedCompilation=false`.

## Dev Notes

### Existing SourceTools State

- `FrontComposerGenerator` is already an `IIncrementalGenerator` using `SyntaxProvider.ForAttributeWithMetadataName` for `[Projection]`, `[Command]`, and `[ProjectionTemplate]`. Preserve this shape; drift detection should add comparison inputs, not replace the parse pipeline.
- Current generator fan-out:
  - Projection: Razor view, Fluxor feature, actions, reducers, registration.
  - Command: form, actions, lifecycle feature, registration, renderer, last-used subscriber, lifecycle bridge, optional full-page route.
  - Shared: projection template manifest and MCP manifest.
- Existing tracked step names include `Parse`, `ParseCommand`, and `ParseProjectionTemplate`; benchmark and cache tests assert these names.
- `DomainModel`, `CommandModel`, and `PropertyModel` are pure data and already include most fields needed for drift comparison. Some fields such as `SourceFilePath`, `SourceLine`, and `SourceColumn` are useful for locations but must not become baseline identity.

### Diagnostic ID Discipline

- SourceTools diagnostics live in HFC1000-HFC1999.
- Current `AnalyzerReleases.Unshipped.md` allocates through HFC1057.
- `DiagnosticCatalogTests` reflect over `DiagnosticDescriptors` and `FcDiagnosticIds`; update both when adding new IDs.
- Several older generator descriptors do not yet carry `HelpLinkUri`; Story 9-1 should set `HelpLinkUri` on new descriptors and avoid broad churn to older IDs unless needed.
- HFC1010 already exists as the hot-reload/full-rebuild guidance ID; do not reuse it for general drift. Either route only the documented full-rebuild categories through HFC1010 or allocate new drift-specific IDs.

### Baseline Contract

Use one structural baseline source of truth. Recommended shape:

```json
{
  "schemaVersion": "frontcomposer.generated-ui-baseline.v1",
  "algorithm": "frontcomposer-structural-v1",
  "contracts": [
    {
      "family": "projection",
      "type": "Acme.Shipping.ShipmentProjection",
      "boundedContext": "Shipping",
      "displayName": "Shipments",
      "role": "ActionQueue",
      "properties": [
        {
          "name": "Priority",
          "category": "Enum",
          "nullable": false,
          "displayName": "Priority",
          "description": null,
          "columnPriority": 1,
          "fieldGroup": null,
          "displayFormat": "Default"
        }
      ]
    }
  ]
}
```

The final JSON schema can differ, but it must remain deterministic, reviewable, and free of runtime values.

### Architecture Constraints

- SourceTools targets `netstandard2.0` and references `Microsoft.CodeAnalysis.CSharp` with `PrivateAssets="all"`.
- The current package pin is `Microsoft.CodeAnalysis.CSharp` 4.12.0; do not move it inside this story unless a build break forces it.
- Microsoft Learn confirms `IIncrementalGenerator` instances should not store state directly; keep comparison state in incremental values, not generator instance fields.
- Microsoft Learn confirms `AdditionalTextsProvider` supplies non-code files to an incremental generator; use that for baselines.
- Microsoft Learn confirms `ForAttributeWithMetadataName` only transforms nodes carrying a matching attribute metadata name and notes partial-type behavior; partial declarations matter for drift and hot-reload classification.
- `DiagnosticDescriptor.HelpLinkUri` is the Roslyn API for diagnostic documentation links; use it on new descriptors.

### Scope Guardrails

Do not implement these in Story 9-1:

- Public CLI commands for inspecting or updating baselines. Owner: Story 9-2.
- Automated source rewrites, rename refactoring, code fixes, or migration tools. Owner: Story 9-2 / Story 9-4.
- Public diagnostics documentation site pages. Owner: Story 9-4 / Story 9-5.
- MCP runtime schema negotiation, client fingerprint comparison, or agent-visible compatibility responses. Owner: Story 8-6.
- Broad DI lifetime analysis, service-locator audits, policy-catalog whole-host proof, or runtime authorization redesign. Owners: deferred follow-ups / Story 9-4.
- Visual specimen drift, accessibility screenshot gates, or generated UI screenshot baselines. Owner: Story 10-2.
- Mutation testing rollout. Owner: Story 10-4.
- A new baseline file watcher outside Roslyn incremental inputs.
