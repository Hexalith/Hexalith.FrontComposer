### What Went Well

1. **FC-CUST became explicit.** The epic produced separate contracts for Level 2 templates, Level 3
   field slots, Level 4 full-view replacements, and override accessibility diagnostics.

2. **Precedence is now a stable architecture decision.** Story 6.1 recorded the L4 -> L2 -> default

   active body.

3. **Confirm-and-pin avoided unnecessary churn.** Most of the customization surface already existed.
   The team added focused pins and contract artifacts instead of rewriting registries or generated
   output.

4. **Real defects were found because tests targeted invariants.** Story 6.1 exposed a cache-key
   `GetHashCode()` gap. Story 6.4 exposed a registration-discovery race in the accessibility analyzer.

5. **Diagnostic phase honesty improved.** Story 6.2 corrected HFC1038-HFC1041 wording for Level 3,
   Story 6.3 documented HFC1042/HFC1046 as reserved and HFC1043-HFC1045 as registry/startup, and

### Challenges And Growth Areas

1. **File List drift still recurred.** Stories 6.1, 6.2, and 6.4 all needed review-side corrections
   for omitted E2E/test-summary/sample files. Story 6.3 passed the gate cleanly, proving the process
   can work when changed files are reconciled against git reality before review promotion.

2. **Historical story labels remain noisy.** Source comments still refer to older labels such as
   Story 6-2, 6-3, 6-4, 6-6, and 9-4. The right policy remains edit-only-when-nearby, but story
   records must keep mapping current plan numbers to historical labels.

3. **Diagnostic catalog metadata is ahead of runtime reality.** Some HFC1042-HFC1046 descriptors and
   registry metadata exist in SourceTools, but the live Level 4 path does not emit all of them at
   build time. Epic 7 must account for phase/disposition, not just ID/severity.

4. **Local full-lane verification remains environment-sensitive.** Release builds were clean and
   focused fallback lanes were green, but exact local solution VSTest and browser runs were repeatedly
   blocked by socket permissions.

## 3. Key Learnings

- **A cataloged diagnostic ID is not the same as a proven emission path.** Epic 7 tooling must model
  descriptor metadata, build analyzer output, startup registry logs, runtime render diagnostics, and
  reserved IDs separately.
- **Override power needs framework-owned boundaries.** Level 4 can replace a body, but shell,
  lifecycle, loading/empty policy, authorization, telemetry, and dev diagnostics remain owned by
  FrontComposer.
- **Analyzer concurrency needs completion barriers.** Registration-discovered Level 3/4 accessibility
  analysis must read collected component references only after collection is complete.
- **Evidence hygiene has to become mechanical.** Manual File List reconciliation is still failing
  often enough that Epic 7 should treat changed-file inventory as a tooling target.

## 6. Significant Discovery Check

Epic 6 surfaced one planning-relevant discovery for Epic 7: the diagnostic catalog cannot be treated
as a flat list of build diagnostics. Some IDs are true SourceTools analyzer/generator diagnostics,
some are startup/runtime registry diagnostics, and some are reserved catalog entries. Epic 7 Story 7.3
and `frontcomposer inspect` should model this explicitly.

Epic 7 does not need a full replan, but its diagnostic stories need an input constraint: "cataloged"
does not mean "build-emitted." Any tooling text or JSON schema that exposes diagnostics should be able
to preserve phase and disposition.
