## What Went Well

- **Canonical seam discipline held.** MCP commands and resources consumed `ICommandService`, `IQueryService`, Story 7 tenant context, Story 7 authorization metadata, Story 8 descriptor registries, and Story 8 lifecycle services instead of building a second domain API.
- **Fail-closed thinking became the default.** Missing tenant gates, missing policy gates, malformed schema hints, stale descriptors, hidden tools, cross-tenant resources, unsupported algorithms, and projection failures were repeatedly pushed toward safe categories before side effects.
- **Deferred hardening was traceable and then closed.** Story 8-4 deferred taxonomy/snapshot gaps into 8-4a. Story 8-6 deferred runtime-gate wiring into 8-6a. The follow-up stories were not vague backlog notes; they had named files, acceptance criteria, owners, and verification.
- **Agent usefulness and security were balanced.** Unknown tool responses include visible suggestions and tool lists, but only after tenant/policy filtering. Projection output is readable Markdown, but row values and descriptor text are escaped, bounded, and treated as inert data.
- **Schema drift became an explicit product surface.** Fingerprints, negotiation categories, baselines, migration deltas, and diagnostics give agents and developers a stable way to detect mismatch instead of failing mysteriously.
- **The skill corpus turned framework rules into package-owned knowledge.** Agent guidance is versioned, embedded, validated, exposed as resources, and separated from team-specific overlays.
- **Review layers found real issues.** The repeated Blind Hunter, Edge Case Hunter, and Acceptance Auditor passes caught fail-open gates, claimed-vs-implemented gaps, raw JSON duplicate limitations, lifecycle ordering hazards, schema-gate non-wiring, and public API break decisions.

## Challenges

- **Epic 8 grew follow-up stories because the first pass exposed deeper contracts.** Projection rendering needed 8-4a for taxonomy and immutable snapshots. Schema versioning needed 8-6a for runtime negotiation wiring. That is good discipline, but it shows the original story cuts underestimated cross-story hardness.
- **Security-sensitive optional or default behavior kept reappearing.** The same pattern surfaced in several forms: default allow-all gates, caller-supplied additive-drift flags, optional gate absence, and hidden-unavailable responses with overly specific categories. Each case had to be converted into required seams, explicit registration, or fail-closed categories.
- **Runtime behavior sometimes lagged behind contract design.** Story 8-6 initially produced fingerprint and negotiation infrastructure without production wiring. The retro lesson is blunt: a helper that is not called by admission paths is not a shipped safety feature.
- **Test claims sometimes outran test reality.** Story 8-3 and 8-6a both recorded cases where ledgers or acceptance checkboxes were marked complete before the exact concurrency, replay, schema, or revalidation test was fully pinned.
- **Cross-package boundaries became more complex.** Introducing `Hexalith.FrontComposer.Mcp`, then schema fingerprints, then the runtime-only Schema project added real architectural value, but it also increased dependency-boundary review cost.
- **Some readiness facts remain absent from artifacts.** Deployment status, stakeholder acceptance, and release-readiness evidence for Epic 8 were not recorded separately from engineering completion.

## Key Lessons

1. **Generated descriptor is now a security boundary.** Tool lists, schema validation, projection resources, skill resources, and schema negotiation all depend on descriptor correctness. Descriptor registry integrity needs the same seriousness as auth code.
2. **Runtime wiring must be part of acceptance for safety infrastructure.** Negotiators, validators, gates, and diagnostic mappers are incomplete until production admission paths call them before side effects.
3. **Agent self-correction is only safe after visibility filtering.** Suggestions, visible lists, empty-state commands, and schema messages must never reveal hidden tools, tenants, policy names, or resource existence.
4. **Artifact-led deferral can work when it is precise.** 8-4a and 8-6a are positive examples: deferred work had story numbers, acceptance criteria, files, tests, and completion evidence.
5. **Untrusted text remains untrusted even in Markdown.** Labels, descriptions, row values, skill content, diagnostics, and generated examples all need bounds, escaping, redaction, and inert rendering.
6. **A compatible schema decision cannot weaken current validation.** Additive compatibility may allow an older client to keep working, but current server validation, defaulting, bounds, tenant checks, and sanitization still win.
7. **The review log is part of the system memory.** Many of the best lessons came from review deltas, not the original story text. Preserving those ledgers is valuable.

## Significant Discoveries

Epic 8 does not require Epic 9 to be replanned, but it adds three durable constraints:

1. **Schema negotiation is an admission concern.** Any CLI, docs, or diagnostics story that presents schema status must preserve the same precedence ordering: hidden/unknown first, stale/integrity/algorithm/baseline next, mismatch last.
2. **Skill corpus and documentation are now linked release surfaces.** Breaking a shipped corpus example should trigger migration-guide work, not just a docs edit.
3. **MCP descriptor and renderer contracts are public enough to deserve diagnostics.** They may be SDK-neutral, but they are adopter-visible through tools, resources, fingerprints, and generated manifests.

Epic update required: No Epic 9 rewrite. Add explicit Epic 9 documentation and diagnostic coverage for Epic 8 MCP/schema/skill categories, and keep the AC5 revalidation follow-up visible.

## Commitments

- Keep MCP as a generated, typed, tenant-aware surface over existing FrontComposer seams.
- Treat every agent-visible response as a potential security boundary.
- Preserve hidden/unknown equivalence across tools, resources, lifecycle, and schema negotiation.
- Keep schema compatibility helpful, but never let it bypass current validation or authorization.
- Record readiness separately from engineering completion.
