### What Went Well

1. **Epic 6 diagnostic-phase lessons were applied.** Stories 7.1 and 7.3 preserved the distinction
   between build analyzer diagnostics, startup/runtime registry diagnostics, reserved catalog rows,
   and synthetic migration sidecar evidence.

2. **CLI contracts became precise.** `inspect` now documents generated manifest-file count semantics,
   post-filter fail flags, sidecar sanitization, and severity threshold filtering. `migrate` now
   documents dry-run/apply, safe write paths, diff budgets, and the synthetic-only HFCM9002 boundary.

3. **Review found real product-quality issues.** The epic closed low-to-high findings: text/JSON
   summary parity, `--severity hidden` include-all semantics, text diff aggregate budgeting, stale
   evidence counts, and Testing redaction for secret values containing commas.

4. **Drift detection stayed architecture-safe.** Story 7.4 proved the existing opt-in pipeline,
   preserved `LoadDriftBaselines` incremental isolation, and kept HFC1070 as the only legitimate
   `CompilationProvider`-based adjacent diagnostic.

5. **The Testing package became adopter-ready.** Story 7.5 pinned host wiring, direct composition,
   deterministic fakes, not-modified query/page paths, redaction, public API baseline policy, package
   contents, and clean-consumer behavior.

### Challenges And Growth Areas

1. **File List drift improved but did not disappear.** Stories 7.3 and 7.5 still needed review-side
   File List corrections for E2E, package, or modified source files. Epic 6 action E6-AI-2 remains
   incomplete.

2. **Historical Story 9 labels are still confusing.** Drift and migration sidecar code/docs still
   contain older Story 9 references. The team correctly recorded these as provenance or label debt,
   but user-facing docs must avoid sending adopters to obsolete story ownership.

3. **Evidence records are still manually curated.** Story 7.4 review corrected stale pass counts in
   contract and test-summary artifacts. This is a process issue, not a code issue.

4. **Local default test execution remains environment-sensitive.** The team has a workable fallback
   through direct xUnit v3 in-process execution, but every story must continue naming exact blocked
   lanes rather than treating a socket failure as generic success.

## 3. Key Learnings

- **Confirm-and-pin works when the contract is explicit.** The successful stories first named the
  v1 contract, then pinned only the missing evidence or narrow bug.
- **Text output must be treated as an API.** JSON paths were usually better specified; review found
  parity gaps in text summaries and migration diff budgets.
- **"Hidden" means include-all, not just the lowest canonical severity.** Sidecar input can contain
  non-canonical values, so filters need a total-input model.
- **Synthetic evidence must stay labeled synthetic.** HFCM9002 sidecar handling is useful today, but
  no production SourceTools emitter exists yet.
- **Testing utilities need privacy review.** Assertion helpers can leak sensitive data just as easily
  as runtime logs if redaction is too naive.
