## Executive summary (Feynman-level, ~45 sec)

Today three "UX completeness" surfaces ship incomplete or silently-broken: **(a)** projection empty states render a minimal `"No {entities} yet."` placeholder with no CTA (Story 4-1 ADR-053 parked the full flow here); **(b)** field descriptions authored via `[Display(Description="...")]` or `[System.ComponentModel.Description("...")]` are INVISIBLE in every generated view тАФ the Parse stage ignores them, the emitter never renders them; **(c)** unsupported field types (`Dictionary<string,List<T>>`, custom value objects, `ReadOnlyMemory<T>`, etc.) are SILENTLY DROPPED at Transform тАФ `RazorModelTransform.cs:29` contains `if (property.IsUnsupported) { continue; }`, directly violating **FR9** ("unsupported types never render silently") + **UX-DR55** ("never silently omit"). Story 4-6 closes all three in one scoped story:

1. **Empty-state completion.** `FcProjectionEmptyPlaceholder` keeps its frozen 3-parameter surface (ADR-053) but gains 2 APPEND-ONLY parameters (`CtaCommandName` + `SecondaryText`) and a new body (anatomy inlined directly per Challenge C3 тАФ no separate helper component) that renders: a large muted `FluentIcon`, a `Body1` primary message "No {entities} yet.", an OPTIONAL `FluentAnchor Appearance="Accent"` CTA (semantically a navigation link, not an action тАФ per Challenge C6) wrapped in `<AuthorizeView>` so unauthorized users see the empty-state without the CTA (per AC2.5 тАФ pre-mortem P2 prevention), resolved through a CTA discovery chain (per-projection `[ProjectionEmptyStateCta(nameof(...))]` attribute тЖТ bounded-context creation-verb-prefix fallback ordered alphabetically among matches тЖТ null when read-only тАФ per revised D4), an OPTIONAL `FluentLabel Typography.Body2` secondary text sourced via a hardcoded resource-key naming convention (`{ProjectionTypeFqn}_EmptyStateSecondaryText` тАФ no `FcShellOptions` override surface in v1 per Challenge C4). Read-only projections (no available commands, no CTA attribute) render without the anchor тАФ AC specifies this explicitly. Role-differentiated copy is a single resource-template lookup keyed by `ProjectionRole`; ActionQueue reads `EmptyStateActionQueueMessageTemplate` ("No {0} awaiting action"), all other roles fall through to `EmptyStateMessageTemplate` ("No {0} yet").

2. **Field description plumbing.** The Parse stage (`AttributeParser`) extracts description strings from BOTH `System.ComponentModel.DescriptionAttribute` (primary тАФ the C# idiomatic location) AND `System.ComponentModel.DataAnnotations.DisplayAttribute.Description` (secondary тАФ aligns with `[Display(Name=)]` precedent already parsed). Both paths surface on a new `PropertyModel.Description: string?` field (append-only; `Equals`/`GetHashCode` updated тЖТ IR cacheability per Story 1-4 preserved). Precedence: `[Description]` wins over `[Display(Description=)]` when both are present (matches the .NET Framework convention тАФ `[Description]` is the older, more specific annotation). `ColumnModel.Description` flows through verbatim. The emit stage consumes it in TWO surfaces: (a) DataGrid column headers emit a `HeaderCellItemTemplate` that wraps the header text in a `FluentTooltip` when description is non-null (tooltip visible on hover + focus, `aria-describedby` wires to screen readers); (b) detail/form views (Story 4-1 DetailRecord body + Story 4-5 expand-in-row detail body via the factored `EmitDetailRecordBody`) emit a secondary `FluentLabel Typography.Caption` line BELOW the field value. The column-sort + filter affordances (Story 4-3) stay on the header surface тАФ tooltip composes as a sibling, not a replacement.

3. **Unsupported-type honesty (the critical fix).** `RazorModelTransform.Transform` STOPS dropping Unsupported columns. Every `PropertyModel` (supported OR unsupported) becomes a `ColumnModel` тАФ the `TypeCategory.Unsupported` branch signals emit-stage dispatch to `FcFieldPlaceholder` INSTEAD of `PropertyColumn<T,TProp>`. The emitter emits a `TemplateColumn` whose cell content is `<FcFieldPlaceholder FieldName="@nameof({Property})" TypeName="{UnsupportedTypeFqn}" IsDevMode="@(RenderContext.Mode == RenderMode.DevMode)" />`. Adopters see a dashed-border card in the column cell + a build-time HFC1002 warning (unchanged тАФ the warning was already firing, just the runtime drop was contradicting it). `FcFieldPlaceholder`'s hardcoded English body is replaced with an `IStringLocalizer` lookup (5 new EN+FR resource keys). Dev-mode highlight (red-dashed border per UX-DR3) reads `RenderContext.Mode` тАФ the FULL DevModeOverlay (Ctrl+Shift+D) is Epic 6 scope; 4-6 ships ONLY the parameter + CSS class wiring so Epic 6 overlay activation flips the visual without re-opening 4-6.

The design honors the "capability contract" principle: `ColumnModel` schema gains ONE append-only field (`Description`), `PropertyModel` gains ONE (`Description`), **no breaking changes to `GridViewSnapshot` / persistence blob** (none of this state persists тАФ descriptions are build-time metadata, empty-state is a pure render decision), decision budget lands inside L06 feature-story cap (тЙд25; this story lands at 19/25 with a 6-decision buffer), every cross-story seam explicit per L01. Scope is deliberately narrow: **empty-state body completion, description plumbing (Parse тЖТ IR тЖТ Emit), unsupported-type honesty (stop silent drop), resource localization for `FcFieldPlaceholder`**. Dev-mode overlay activation (Ctrl+Shift+D + annotation panel per component-strategy ┬зDevModeOverlay), `[ProjectionEmptyStateCta]` CUSTOM icon overrides (icon is Epic 6-1 annotation territory), custom per-projection empty-state Razor templates (Epic 6-2), ICU-pluralized empty-state copy ("1 order" vs "2 orders"), and dev-mode-highlight visual specimen verification are OUT of scope.

## Story

As a business user,
I want helpful empty states that guide me toward action, field descriptions that explain what columns mean, and clear indicators for any fields the framework can't auto-render,
so that I'm never confused by blank screens, cryptic column names, or missing data.

**Business value:** Until 4-6 lands, three papercut UX failures degrade every auto-generated projection view:

1. **Silent omission.** A developer who models an order with `public Dictionary<string, LineItem> Items { get; init; }` sees the column DISAPPEAR from the generated DataGrid тАФ no warning in the browser, no runtime hint, just missing data. The build emits HFC1002 but the busy developer doesn't read generator warnings. The business user sees an order with zero line items. The operator dismisses the data as "probably fine." Silent data loss is the cardinal sin of auto-generation (UX-DR55). 4-6 makes omission IMPOSSIBLE: every property appears either as a data column or as a visually-loud dashed-border card.

2. **Cryptic column names.** A developer names a property `Sla` (for "Service Level Agreement"); the business user sees a column labeled "Sla" with no context. A `[Description("Service level commitment in business days, contractually defined per customer tier")]` annotation would solve it тАФ but 4-6 is the first story to actually surface the description. Today the attribute is ignored by Parse. After 4-6, hovering the column header reveals the description as a `FluentTooltip`; screen readers announce it via `aria-describedby`. Zero-cost documentation.

3. **Blank screens.** A first-time user opens the Orders page. Zero orders exist yet. The current 4-1 placeholder renders a single grey icon and `"No orders yet."` тАФ no guidance, no CTA. The user closes the tab. After 4-6, the same user sees the full anatomy: icon + message + `"Send your first Create Order"` button that navigates directly to the command form. First-visit friction тЖТ first-visit action.

This story is the "capstone of Epic 4": Stories 4-1 through 4-5 shipped DataGrid rendering, badges, filter/sort, virtualization, and expand-in-row; 4-6 polishes the three quality-of-life surfaces that would otherwise haunt v1 adopters for months.

## Migration note for 4-6 adopters

**This story changes user-visible behavior on first rebuild after merge.** Three breaking-visual surfaces appear automatically; adopters MUST audit projections before merging 4-6.

### Surface 1 тАФ Unsupported columns become visible

**What changes:** Pre-4-6, projection properties whose types fall outside the `MapTypeCategory` supported set (`Dictionary<K,V>`, `ReadOnlyMemory<T>`, custom value objects, ref structs, etc.) were SILENTLY DROPPED from the generated DataGrid (`RazorModelTransform.cs:29` had `if (property.IsUnsupported) { continue; }`). After 4-6, every unsupported property emits a dashed-border `FcFieldPlaceholder` card in its own grid column with the warning icon, type annotation, and docs link. **Live demos with N silently-dropped columns will visually grow N dashed-border cards on first rebuild.**

**Audit before merge:**
1. `grep -rE "\[Projection\]" src/` тАФ list all projection types.
2. For each projection, inspect its property types against `RazorModelTransform.MapTypeCategory` (lines тЙИ286-294 of pre-4-6 source).
3. For each unsupported property, decide:
   - **Keep the placeholder** (recommended for new adopters тАФ placeholder loudly signals modeling gap; build warning HFC1002 still fires for visibility).
   - **Add a custom renderer via Epic 6-3 slot-level field replacement** (when shipped; until then, placeholder is the only path).
   - **Remove the unsupported property from the projection** (the property may belong on the aggregate root but not on the read-side projection).
4. Coordinate with stakeholders: any LIVE DEMO scheduled for the first sprint after 4-6 ships must be rehearsed with the new visual shape.

### Surface 2 тАФ Description tooltips appear on annotated columns

**What changes:** Properties annotated with `[Description("...")]` or `[Display(Description="...")]` (both attribute paths exist in many real codebases via inherited `System.ComponentModel` usage) тАФ the descriptions were INVISIBLE pre-4-6 (Parse stage ignored them). After 4-6, every annotated column header gains a small inline `(i)` info icon adjacent to the header text, with the description as a hover/focus tooltip. **Adopters with pre-existing `[Description]` annotations on projection properties will see new icons appear next to many column headers without any code change on their side.**

**Audit before merge:**
1. `grep -rE "\[Description\(|\[Display\(.*Description=" src/` тАФ find pre-existing annotations on projection properties.
2. Review the description text тАФ was it written for developer audiences (acceptable as tooltip), or did it contain placeholder/TODO text that shouldn't be user-facing?
3. Update or remove inappropriate descriptions BEFORE merge.

### Surface 3 тАФ Empty-state CTAs may suggest unexpected commands

**What changes:** Pre-4-6, the empty-state placeholder rendered `"No {entities} yet."` with no CTA button. After 4-6, an `FluentAnchor Appearance="Accent"` CTA button appears, suggesting the first writable command in the projection's bounded context (per revised D4: creation-verb-prefix-matching commands ranked first; alphabetical fallback otherwise). **For projections in bounded contexts with "creative" command vocabulary (`MintBadge`, `ForgeContract`, `CraftPotion`), the heuristic may pick a non-creation command тАФ the adopter must explicitly annotate `[ProjectionEmptyStateCta(nameof(MintBadge))]` to override.**

**Audit before merge:**
1. For each `[Projection]`, identify its bounded context (`[BoundedContext]`).
2. List all writable commands in that BC.
3. Mentally apply revised D4 тАФ does the verb-prefix match yield the desired CTA? If not, add `[ProjectionEmptyStateCta(nameof(YourCreationCommand))]` to the projection type.
4. For read-only projections (no writable commands), no action needed тАФ empty state renders without CTA per AC2.

### Process-notes ledger entry

This section's existence is itself a process pattern worth capturing: **L12 candidate** for `_bmad-output/process-notes/story-creation-lessons.md` тАФ "UX-change stories that flip default behavior MUST include an explicit Migration note section listing each visible-surface change, audit steps, and decision options. Pure-additive feature stories don't need this; behavior-flip stories like 4-6 (silent-drop тЖТ loud-emit) do." Document after 4-6 merges and the migration playbook is validated by adopter feedback.
