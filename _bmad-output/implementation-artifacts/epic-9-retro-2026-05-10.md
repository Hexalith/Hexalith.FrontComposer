## Successes

- Developer-facing ambiguity was converted into executable governance. Drift detection, CLI inspection, IDE parity, diagnostics, and documentation all gained validation hooks instead of relying on convention.
- Cross-story contracts were explicit. Story 9.1 created drift and generated-output handoffs; Story 9.2 consumed and exposed them; Story 9.3 documented the IDE contract; Story 9.4 governed diagnostic metadata; Story 9.5 published and validated the docs surface.
- The team repeatedly used review findings as design input. Story records show review patches resolving ordering, path safety, redaction, diagnostic registry, docs slug, migration sidecar, and validation gaps.
- Documentation became a product surface, not an afterthought. DocFX, Diataxis placement, marker extraction, producer fingerprints, diagnostic reference pages, CLI docs, IDE parity references, and migration guidance now have one validation entrypoint.
- Safety boundaries stayed visible. Story records repeatedly defend against local path leakage, tenant/user/token exposure, generated-output mutation, submodule recursion, path traversal, symlink/junction issues, and unsafe CLI writes.

## Challenges And Friction

- Epic 9 created a large governance surface. Diagnostic registry, docs validation, IDE evidence, CLI migration safety, generated-output contracts, and API compatibility checks now need active care.
- Several stories required multiple review passes because initial acceptance looked complete while edge cases were still open. That happened around drift diagnostics, CLI path safety, IDE evidence freshness, diagnostic stubs, and docs validation.
- Some implementation records carried temporary or synthetic seams that needed explicit ownership. The HFCM migration diagnostic namespace and some sidecar behavior remain governed by Story 9.2 follow-up work.
- Documentation completeness is partly structural and partly editorial. Story 9.5 now validates shape and publication, but per-diagnostic prose quality and richer relationships still need authoring work.
- Sprint status still lists `epic-9` as `in-progress` even though all Epic 9 stories are done. This retro updates only the retrospective key, preserving the current tracking convention.

Murat (Test Architect): "The recurring pattern is useful: the first version often proved the happy path, then review forced the hostile-path contract into the design."

## Key Lessons

1. "Inspectable generated output" is not one feature. It requires stable paths, compiler diagnostics, CLI reports, IDE evidence, documentation links, and redacted machine-readable output to agree.
2. Governance needs a single source of truth. The diagnostic registry and docs validation command reduced drift by giving metadata one canonical place to live.
3. Evidence manifests are a strong pattern for manual or vendor-dependent behavior. IDE parity cannot be fully automated without overreaching, but it can still be governed.
4. Review should target artifacts as much as code. Several important fixes lived in docs stubs, story records, registry JSON, validation scripts, and CI wiring.
5. Deferred work must name an owner and a surface. Epic 9 did this better over time, especially in `deferred-work.md`, but some follow-ups are still broad enough to deserve tightening before Epic 10 quality gates depend on them.

## Risks For Epic 10

- Quality gates may amplify deferred governance gaps. For example, diagnostic registry inconsistencies, docs placeholder policy, or CLI migration edge cases can become blocking once Epic 10 formalizes adopter confidence.
- Test and docs surfaces are now broad. Epic 10 should avoid adding parallel validation entrypoints that drift from the new docs and diagnostic governance.
- Several Epic 10 stories depend on stable evidence artifacts. Evidence manifests, generated-output paths, diagnostic docs links, and package validation baselines should be treated as contracts.
- Accessibility and visual specimen work will likely expose places where docs examples, generated UI, and test fixtures need more concrete product data.
