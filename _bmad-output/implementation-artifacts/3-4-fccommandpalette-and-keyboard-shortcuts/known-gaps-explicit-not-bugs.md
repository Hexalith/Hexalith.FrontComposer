| G17 | Adopter-side `IPaletteAliasProvider` extensibility point for disabling or re-mapping the D23 shortcut aliases (`?`, `help`, `keys`, `kb`, `shortcut`). | G18 | Modifier-bearing shortcuts (`Ctrl+K`, `Ctrl+,`) fire even when keyboard focus is inside an adopter-embedded code editor that defines its own conflicting semantic (Monaco's `Ctrl+K Ctrl+F` "format", CodeMirror's `Ctrl+K` "delete line"). The D5 text-input guard exempts modifier combos deliberately because `Ctrl+K` must work globally. Mitigation: adopters who embed such editors register their own handler via `IShortcutService.Register("ctrl+k", ...)` AFTER the shell (D3 last-writer-wins) to override the palette binding within their component scope. | **v1.x** (per-component shortcut-scoping is an Epic 6 extensibility concern) | A per-component "suppress-in-contenteditable" flag on `IShortcutService.Register` would solve this more cleanly than the last-writer-wins workaround, but adds API surface for a niche case. Documented so adopter embedders don't hit a silent regression. |
| G19 | Shadow-DOM targets may evade the D5 text-input guard. If `e.target` is a shadow-host (e.g., a third-party web-component editor wraps its inputs in shadow DOM), `tagName` checks miss the inner `<input>` / `<textarea>`. 3-4's guard uses a simple target-tagname check; a `composedPath()`-based check would walk into the shadow tree. | **v1.x** (requires a Task 0 follow-up spike + JS-interop helper to access `event.composedPath()` from Blazor) | No 3-4-embedded adopter is known to use shadow-DOM editors; the gap is theoretical in v1. If a pilot adopter reports text-input keystrokes being swallowed by `g h` chord, v1.x adds the composedPath walk. |
| G20 | **Trusted-adopter security model for `IShortcutService` registration.** v1 assumes all code running inside `AddHexalithFrontComposer`'s host is trusted ŌĆö adopter NuGet packages that ship `services.AddScoped<IShortcutService, EvilService>()` AFTER `AddHexalithFrontComposer` will REPLACE the framework service via DI last-registration-wins, and any malicious implementation (keystroke exfiltration, cross-site forwarding) is not prevented at the framework level. The `IShortcutService` interface lives in `Contracts/` ŌĆö any package referencing Contracts can implement it. | **v1.x** (supply-chain hardening) / **Story 9-4** (Roslyn analyzer complements this at build time) | Framework uses `TryAddScoped<IShortcutService, ShortcutService>()` (clarified 2026-04-21) ŌĆö this is the idiomatic .NET DI pattern for framework defaults. Adopter override remains fully supported via `services.AddScoped<IShortcutService, MyImpl>()` AFTER `AddHexalithFrontComposer`: `AddScoped` unconditionally replaces prior registrations, so D3's last-writer-wins design intent is preserved. Only a second `TryAddScoped<IShortcutService, ŌĆ”>()` call from an adopter would silently no-op, and that pattern is not idiomatic for override. The security model is: adopters, NuGet packages, and the hosting infrastructure are all in the same trust boundary as the framework. Hostile-adopter scenarios are out of scope for v1; deployers should audit their dependency tree. Documented explicitly post-elicitation Red Team analysis so deployers know the assumption rather than inferring it. |
| G23 | **`PaletteScorerBench` NFR5 micro-benchmark (`< 100 ╬╝s per candidate`, `< 100 ms for 1000 candidates`) not shipped in 3-4.** DAR defers to "inspection-only rationale": the algorithm's allocation profile is bounded (top-50 cap, no heap churn in hot path); worst-case empirical scoring over realistic adopter manifests (~20-50 candidates) is sub-millisecond in dev-env measurements; the 150 ms debounce absorbs the total roundtrip envelope. No telemetry of production scorer latency exists yet. | **Story 10-4** (mutation + property-based testing + performance guardrails) / **Story 10-2** (accessibility + performance CI gates) | Benchmark infrastructure is a CI-gate investment; adopting BenchmarkDotNet for a single bench without a broader perf-regression pipeline is premature. Revisit when (a) an adopter reports palette typing lag; (b) an adopter ships > 200 candidates into `IFrontComposerRegistry.GetManifests()`; (c) Story 10-4 lands the perf-gate pipeline. Until then, `inspection-only` is the documented rationale with explicit revisit triggers. |
| G21 | **Palette activation silently discards adopter-form unsaved work.** When a user presses Ctrl+K while editing an adopter-owned form, types a query, and activates a navigation result, `NavigationManager.NavigateTo(...)` fires and the form's unsaved state is lost with no confirmation dialog. Blazor's default `<EditForm>` has no dirty-state tracking that the palette could observe, so 3-4 has no framework-level hook to intercept. | **v1.x** (requires a `IPaletteNavigationGuard` extensibility point that adopter forms can register to veto or prompt) | A confirmation prompt pattern (like `beforeunload`) would require per-form dirty-state wiring that adopters must author; adding it to 3-4 couples the palette to an un-built form-state abstraction. v1 ships the pragmatic behaviour (navigation wins); v1.x adds the guard extensibility point once Epic 6 template overrides establish a form-state convention. Document so adopters are aware before shipping palette to users editing long forms. |
| G22 | **Shortcut-registration discoverability is log-based only.** When multiple adopters register shortcuts with possible collisions (D3 / HFC2108), the only runtime diagnosis path is scraping `HFC2108_ShortcutConflict` Information logs for `{CallSiteFile, CallSiteLine}` payloads. There is no in-app UI that visually lists every registered shortcut with its source. The `"shortcuts"` query (AC6) shows shortcuts but only the user-facing label + description ŌĆö not the registration call-site. | **Story 6-5** (dev overlay ŌĆö adds a developer-mode UI that renders `IShortcutService.GetRegistrations()` with call-site info as a diagnostic surface) | Log-scraping is fragile (logs may not be captured in all envs; noisy in dev). A dev-mode overlay would surface "who registered what" visually and reduce time-to-diagnose for adopter-package conflicts. Story 6-5's scope (`Ctrl+Shift+D` dev overlay) is the natural home; 3-4 ships the `GetRegistrations()` enumeration API which 6-5 consumes. |
