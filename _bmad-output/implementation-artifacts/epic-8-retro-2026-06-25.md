### What Went Well

1. **The visual direction stayed inside architecture constraints.** Story 8.1 removed the teal
   header fill, Story 8.2 locked the accent-as-thread rule, and later stories preserved the guard.

2. **The epic converted design intent into reusable framework assets.** The new rail and
   `FcPageToolbar` are framework-owned patterns rather than host-specific page tweaks.

3. **Source generator changes were scoped and pinned.** Story 8.4 added sticky generated
   `FluentDataGrid` headers, and Story 8.7 moved `[ProjectionBadge]` status members to `FcStatusIcon`
   through the emitter instead of editing generated output.

4. **Review found real defects before closure.** The reviews caught dead `::part()` selectors,
   dead scoped CSS on Fluent components, a no-op filled icon variant, stale UI-coupled nav handlers,
   an invalid Playwright touch assumption, and one missing custom-logo accessibility assertion.

5. **Count/status separation became clearer.** Generated status enum cells now use icons with
   accessible labels; numeric count, navigation, filter-chip, and optimistic summary surfaces remain
   `FluentBadge` pills.

### Challenges And Growth Areas

1. **Fluent component DOM assumptions repeated.** Stories 8.4 and 8.6 both shipped initial CSS that
   did not attach to rendered DOM. Future visual stories need rendered-DOM or computed-style proof
   when styling Fluent web components.

2. **Some tests initially proved source intent rather than user-visible behavior.** Review had to
   harden tests so they detected no-op selectors, no-op icon variants, and inaccessible logo markup.

3. **Story task wording drifted from implementation reality.** Story 8.5 had stale task text about
   `FrontComposerNavigationCapabilityBadgeTests`; the actual coverage lived elsewhere.

4. **Reference docs lagged behind the implementation.** After Story 8.3 and Story 8.5, public docs
   still described a locked 7-parameter shell surface and the deleted `FcCollapsedNavRail`.

5. **Browser evidence remains partially deferred to CI.** Story records consistently named socket
   blockers, but visual baselines and Playwright browser lanes still need environment-backed closure.

## 3. Key Learnings

- **For Fluent components, attach proof matters.** CSS isolation and `::part()` assumptions are not
  enough when the rendered DOM is owned by Fluent components.
- **Icon variants must be visually distinct, not just semantically named.** Story 8.5 proved that
  a filled variant sharing the regular path is a product defect even when state and aria attributes
  are correct.
- **Accessibility branches need first-class bUnit pins.** The custom-logo `aria-hidden` branch and
  status icon accessible names should not rely only on browser lanes.
- **Reference docs need a post-epic sweep whenever public component surfaces change.** The appended
  shell parameters and deleted rail component were easy to miss because older contract language was
  still plausible.
- **Preserved state schema can outlive a UI producer.** `SidebarExpandedAction` and collapsed-group
  state remain intentionally retained even after deleting `FcCollapsedNavRail`; docs must describe
  current UI behavior without erasing compatibility context.
