## 2. What Went Well

1. **The EventStore boundary was corrected early.** Story 5-1 anchored FrontComposer to EventStore-owned REST endpoints and `/hubs/projection-changes`, while keeping `/projections-hub` as superseded planning text rather than a second default.
2. **SignalR remained nudge-only.** Stories 5-1 through 5-5 consistently re-query through REST with ETag validation after projection changes. That avoided a parallel projection-data path.
3. **Degraded UX stayed non-blocking.** Story 5-3 added inline connection status, immediate `Syncing` escalation, and form-state preservation without modal interruption, route changes, or raw draft persistence.
4. **Reconnection became deterministic enough to test.** Story 5-4 added reconnect epochs, visible-lane reconciliation, sweep markers, schema-mismatch fallback, and stale-epoch protection.
5. **Pending-command handling improved trust.** Story 5-5 added bounded pending command state, exactly-once terminal resolution rules, optimistic status badges, and reconnect outcome summaries without client-side command replay.
6. **Governance caught real risk.** Story 5-6 turned infrastructure-coupling and telemetry redaction into blocking checks, and its review found real false-negative problems before closure.
7. **Testing became a product asset.** Story 5-7 turned scattered realtime fakes into a deterministic fault-injection harness with checkpoint controls and FR24-FR29 traceability.

Dana (QA Engineer): "The test story is the clearest win. We stopped treating disconnect behavior as something only a live environment can prove."

Charlie (Senior Dev): "And we kept deleting ambiguity: no payload-over-SignalR, no silent cache truth, no unbounded runtime state as a casual default."

## 4. Key Lessons

1. **Nudge-only SignalR was the right invariant.** It gave every story the same correctness rule: projections come from REST queries, cache validation, and reducer-visible state changes.
2. **Response classification belongs below UI code.** Story 5-2 paid off repeatedly because DataGrid, badges, forms, reconnect, polling, and pending commands consumed typed outcomes instead of raw HTTP status switches.
3. **Reliable realtime needs bounded state everywhere.** ETag cache entries, visible lanes, pending commands, log suppression buckets, and harness queues all needed explicit caps or disposal assertions.
4. **Exactly-once user outcomes need a first-terminal-wins model.** Story 5-5 showed that duplicate, stale, and reordered outcomes are normal in degraded-network flows, not edge trivia.
5. **Governance tests need adversarial test fixtures, not only happy-path scanners.** Story 5-6's blockers showed that a deny-list is only as strong as its false-negative fixtures.
6. **Deterministic fault injection is the foundation for future confidence.** Story 5-7's checkpoint harness is the reusable path for reconnect, rejoin, pending-command, and telemetry failure tests.
7. **Deferrals must be reconciled after follow-up patches.** The deferred-work ledger is useful, but superseded entries should be marked explicitly so future stories do not chase closed gaps.

## 7. Significant Discoveries

1. **The architecture document is stale on the SignalR hub path.**
   - Impact: New contributors may follow `/projections-hub` from planning text instead of the implemented `/hubs/projection-changes` EventStore contract.
   - Recommended action: Update architecture and docs to mark `/projections-hub` as superseded or configured-only.

2. **Pending command status is structurally ready but lacks a real provider.**
   - Impact: Story 5-5's terminal observation flow is functional once `IPendingCommandStatusQuery` is backed by a real EventStore status endpoint; until then, default behavior short-circuits through the null provider.
   - Recommended action: Create an explicit follow-up for EventStore command status query/provider integration and ETag/429/503 parity.

3. **`FcNewItemIndicator` is implemented but not fully produced by DataGrid flow.**
   - Impact: The component, state service, resources, and tests exist, but producer and host-mount wiring remain follow-up work.
   - Recommended action: Track the producer and DataGrid host wiring with the DataGrid customization or source-generator story that owns row insertion behavior.

4. **Governance checks are now blocking, but their own scanner quality needs maintenance.**
   - Impact: Story 5-6 closed blockers, but deferred items remain around parser robustness, generated baseline coverage, redaction matrix completeness, and docs/exporter guidance.
   - Recommended action: Fold scanner and telemetry polish into Story 9-4 and Story 9-5 rather than reopening Epic 5.

5. **The fault-injection harness is valuable but not yet a productized testing package.**
   - Impact: Current harness APIs are test-support assets, not adopter-facing public contracts.
   - Recommended action: Move formal `Hexalith.FrontComposer.Testing` extraction and public API compatibility to Epic 10 or the packaging story.

**Epic update required:** No rewrite of Epic 5. Yes to targeted follow-up tracking in Epic 9 and Epic 10, plus documentation correction for EventStore hub path and realtime customization boundaries.
