## 2. What Went Well

1. **The command pipeline became end-to-end, not just parser output.** Story 2-1 added command IR, parser, form transform, form emitter, lifecycle actions, registration output, stub command service, and Counter sample wiring. That created a complete vertical slice.
2. **Binding decisions were explicit and useful.** Each story carried decision tables that resolved real implementation ambiguity: `ICommandServiceWithLifecycle`, density thresholds, `Subscribe()` instead of `IObservable<T>`, wrapper threshold behavior, destructive classification, and form-guard scope.
3. **Review discipline caught material issues before they became framework behavior.** Examples include label-resolution precedence in 2-1, density diagnostic renumbering in 2-2, generic command rejection in 2-3, pulse-band exclusivity in 2-4, and AC-vs-decision drift in 2-5.
4. **Testing scaled with risk.** The suite grew from source-generator parse/emit tests into Shell behavior tests, bUnit component tests, lifecycle state-machine properties, snapshot baselines, and options-validation tests.
5. **Known gaps were usually named with owning stories.** Story 2-4 and 2-5 did a good job assigning reconnect, visual-regression, destructive-agent, and shell-focus gaps to future story owners instead of leaving vague "future work".
6. **UX safety improved materially.** Rejections preserve form input, destructive commands require confirmation, abandonment protection covers full-page forms, and idempotent outcomes avoid presenting safe duplicate outcomes as failures.

Dana (QA Engineer): "The best part is that the tests moved closer to user-visible behavior. We were not just checking that code was generated; we were checking lifecycle, accessibility, and safety behavior."

Charlie (Senior Dev): "The generated-code seams held up, but they got more complex. The review passes paid for themselves."

## 4. Key Lessons

1. **Generated UI stories need a contract table for every emitted cross-package reference.** Story 2-4's Shell project reference was accepted deliberately, but it should not become invisible debt.
2. **Lifecycle semantics need one owner.** Stable submission, rejection, idempotency, reconnect replay, and agent execution all reuse the same vocabulary. Without a canonical semantic table, future stories can render the same transition differently.
3. **Spec authority must be reconciled after review decisions.** Story 2-5 showed that AC prose can drift behind binding decisions. Reviews should close both code and story text.
4. **Browser validation needs a single reusable harness path.** Deferring browser checks story-by-story creates repeated negotiation. Story 5-7 and Story 10-2 should absorb that pattern into a standard lane.
5. **Known-gap quality improved, but central documentation did not keep pace.** The local story files are rich. The project-level architecture and constraint guides still need cleanup.

## 8. Significant Discoveries

1. **Consumer-domain Shell dependency is now real.**
   - Impact: Future generator stories must account for emitted components that live outside Contracts/SourceTools. A `Components` package split may be needed later to avoid making Shell the default dependency for all generated UI.

2. **Lifecycle transition semantics need a canonical table before reconnect and agent surfaces deepen.**
   - Impact: Epic 5 and Epic 8 can otherwise interpret `IdempotencyResolved`, self-reconnect replay, cross-user already-confirmed outcomes, and stale transition timing inconsistently.

3. **Browser/E2E validation ownership is still fragmented.**
   - Impact: Story 5-7 and Story 10-2 should make the browser harness reusable so future stories do not repeatedly defer or re-negotiate the same validation class.

4. **Architecture and constraint documentation is no longer just stale; it is actively downstream of shipped implementation.**
   - Impact: Epic 6 customization stories and Epic 9 tooling stories can make wrong assumptions unless the architecture catches up.

**Epic update required:** No immediate rewrite to Epic 3 is useful because Epic 3 is already completed. Future story updates are required for Epic 5, Epic 6, Epic 9, and Epic 10 planning.
