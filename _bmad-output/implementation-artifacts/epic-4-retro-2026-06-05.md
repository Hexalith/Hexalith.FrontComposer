### What Went Well

1. **Safety ordering stayed explicit.** Story 4.5 re-ran the Story 4.1 destructive, Story 4.2
   abandonment, Story 4.3 FC-CNC, and Story 4.4 authorization guardrails after retry changes, which
   prevented retry from moving ahead of authorization or pending-state boundaries.

2. **FC-CNC made the v1 fallback concrete.** Story 4.3 recorded block-not-queue as the one-at-a-time
   policy, added a circuit-local `CommandExecutionAdmissionGate`, and proved denied local submits do
   not dispatch lifecycle, pending-state, or EventStore side effects.

3. **Authorization became both presentational and enforced.** Story 4.4 pinned
   `FcAuthorizedCommandRegion`, generated protected forms, command palette filtering, direct
   dispatch decoration, and policy catalog validation without changing the broader ASP.NET Core
   authorization model.

4. **Retry was intentionally narrow.** Story 4.5 limited retry to EventStore pre-accept transient
   failures, reused the same `MessageId` across attempts, and kept slow accepted commands on the
   existing pending polling and `NeedsReview` path.

5. **Accessibility and localization got real review pressure.** The final review caught a French
   fallback gap for retry-exhaustion copy, and `FcPendingCommandSummary` now announces active pending
   entries plus rejected and needs-review summaries.

### Challenges And Growth Areas

1. **File List drift still recurred.** Story 4.3, Story 4.4, and Story 4.5 each needed review-side
   File List corrections. This continues the Epic 2 and Epic 3 process weakness.

2. **Initial tests sometimes proved too little.** Story 4.5's first same-`MessageId` retry test used
   a constant ULID factory and could not catch per-attempt ID regeneration. Review strengthened it
   with a counting factory.

3. **Generated-code behavior now carries more cross-cutting ordering.** The command form emitter is
   doing more: auth, `BeforeSubmit`, re-auth, FC-CNC, lifecycle, dispatch, pending registration, and
   warning reset behavior. The behavior is pinned, but future changes need explicit ordering tests.

4. **Story-number drift remains in code comments.** Some implementation comments still reference
   historical story labels. The behavior is correct, but the comments make future investigation
   noisier.

## 3. Key Learnings

- **Confirm-and-pin remains effective for brownfield surfaces.** Stories 4.1 and 4.2 added high-value
  guardrails around existing destructive and abandonment behavior without production churn.
- **Policy decisions need two gates.** Presentation-only authorization is not enough; Story 4.4
  correctly kept `AuthorizingCommandServiceDecorator` as the dispatch backstop.
- **Concurrency policy is a side-effect boundary, not just a disabled button.** FC-CNC matters because
  denied submits return before lifecycle mutation, pending registration, and HTTP dispatch.
- **Retry must not blur "not accepted" and "accepted but slow."** Pre-accept retry exhaustion resets
  the generated form to `Idle`; accepted commands remain pending and may later become `NeedsReview`.
- **Review tests should fail for the exact invariant.** The same-`MessageId` lesson from Story 4.5
  should be reused for MCP command identity tests in Epic 5.

## 4. Next Epic Preparation - Epic 5

Epic 5 covers the AI-agent MCP surface: dynamic command tools, lifecycle subscription, projection and
skill-corpus resources, fail-closed security gates, and schema fingerprint negotiation.

### Dependencies From Epic 4

| Dependency | State | Preparation decision |
|---|---|---|
| Generated command safety ordering | Implemented and pinned | MCP command invocation must not bypass authorization, validation, server-controlled identity injection, or schema negotiation |
| FC-AUTH policy metadata | Implemented and pinned | Treat `[RequiresPolicy]` as a generated descriptor input for MCP admission, but keep host policy evaluation authoritative |
| FC-RETRY scope | Implemented for Shell EventStore dispatch only | Do not introduce MCP tool retry policy in Epic 5 unless a separate MCP contract explicitly owns it |
| Pending command identity | Stable from Epic 3 and preserved in Epic 4 | Lifecycle subscription should accept `messageId` or `correlationId`, but side-effecting calls must still inject framework identity server-side |
| Opaque failure discipline | Reinforced by FC-AUTH and FC-RETRY | MCP auth/tenant/schema failures should remain indistinguishable where the plan requires hidden-equivalent responses |

### Critical Path

1. **Define the MCP command invocation ordering before implementing Story 5.1.**
   Owner: Developer + Architect.
   Success criteria: admission, schema negotiation, arg validation, derivable injection, DataAnnotations,
   dispatch, and lifecycle tracking order is explicit and pinned.

2. **Keep Shell retry policy out of MCP v1 unless intentionally contracted.**
   Owner: Product + Developer.
   Success criteria: MCP docs and tests state whether tools rely on the underlying command service
   behavior or add no extra retry loop.

3. **Use strong identity tests for MCP tools.**
   Owner: Test Architect.
   Success criteria: tests prove `TenantId`, `UserId`, `MessageId`, and `CorrelationId` are not
   accepted from agent input and are injected server-side.

4. **Close or automate File List reconciliation.**
   Owner: QA automation maintainer.
   Success criteria: the next completed story has no review-side File List correction.

## 6. Significant Discovery Check

No discovery invalidates Epic 5, but one constraint is important: Epic 4's retry policy is a Shell
EventStore adapter behavior, not an MCP tool policy. Epic 5 should build agent-facing command
admission, lifecycle subscription, fail-closed gates, and schema negotiation without assuming a new
agent retry loop.
