### What Went Well

1. **Confirm-and-pin worked as an Epic 2 delivery mode.** Stories 2.1, 2.2, 2.4, 2.5, 2.6, 2.7, and
   2.8 either made no production code changes or constrained the production surface tightly. The work
   strengthened existing behavior with named, durable pins instead of reworking stable components.

2. **The review loop caught false confidence.** Story 2.3 proved the strongest example: generated
   `[ProjectionBadge]` columns looked correct in source snapshots but failed at runtime because
   `PropertyColumn` could not accept child content. The fix moved badge columns to the renderable
   `TemplateColumn<T>` path and added a rendered-grid pin.

3. **Acceptance criteria were handled honestly.** Story 2.6 did not silently pass the "new item"
   indicator clause. The team verified the live nudge seam carries only projection type and tenant id,
   confirmed no row identity producer exists, and recorded a PO-accepted deferral to command lifecycle.

4. **FC-TBL became an owned public surface.** Story 2.8 added
   `src/Hexalith.FrontComposer.Shell/PublicAPI.FcTbl.Shipped.txt` and
   `FcTblPackageBoundaryTests`, closing the Story 1.0 spike's public-surface gap.

5. **Accessibility coverage moved from component-only to generated-grid paths.** Expand-in-row,
   filter-hidden announcements, palette combobox behavior, badge aria labels, and row-detail regions
   gained rendered or integration pins.

### Challenges And Growth Areas

1. **Record-keeping drift persisted.** Epic 1 AI-1 was not fully solved. Several Epic 2 reviews still
   had to reconcile File Lists and counts after QA automation added files. This improved, but it is
   still a workflow defect rather than a one-off story issue.

2. **Local solution-level VSTest remained blocked.** Story files repeatedly recorded
   `SocketException (13): Permission denied` before VSTest execution. The in-process fallback gave useful
   evidence, but it is not the same command as the project-context local rule.

3. **Legacy story-number references now conflict with current planning.** Several source/story notes
   refer to pending-command/new-item work as Story 5-5, while current sprint status defines Story 5-5
   as MCP schema-fingerprint negotiation and places command lifecycle in Epic 3. This is now documented
   in the FC-TBL contract follow-up language.

4. **Some Epic 1 escalations remain open.** FC-LYT, FC-A11Y, FC-L10N, and FC-DOC still carry
   escalated-owner status. Epic 2 could proceed because defaults are encoded and tested, but the
   owner decisions remain important.

## 3. Key Learnings

- **Rendered pins are necessary when generator output composes third-party components.** Source-string
  assertions alone missed the Story 2.3 `PropertyColumn`/`TemplateColumn` issue.
- **Deferral is acceptable only when it names the missing seam and owner.** Story 2.6 is a good
  pattern: the row-identity producer gap is explicit, PO-accepted, and tied to command lifecycle.
- **Focused package-boundary baselines are useful for narrow public surfaces.** Story 2.8 avoided a
  broad Shell freeze while still making FC-TBL intentional and test-enforced.
- **Current planning keys must override stale source comments.** Future story creation should use the
  current sprint-status keys for Epic 3 command lifecycle, even where older comments say Story 5-5.
