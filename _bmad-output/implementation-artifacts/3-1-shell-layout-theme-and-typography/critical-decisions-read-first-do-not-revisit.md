| D1 | `FcShellOptions.AccentColor` defaults to `"#0097A7"` (exact hex, case-insensitive regex `^#[0-9A-Fa-f]{6}$`). Named colors (`"teal"`), rgb()/hsl() syntax, 8-hex alpha forms (`#RRGGBBAA`) are REJECTED at startup. | Zero-override invariant + Fluent UI `ThemeSettings.Color` expects hex. Restricting format at the boundary eliminates an entire class of runtime interpolation bugs + CSS injection vectors. | Task 1.1, 10.8; AC2 |
| D2 | `Hexalith.FrontComposer.Typography` is a `public static class` in `Hexalith.FrontComposer.Contracts/Rendering/` with 9 `public static readonly Typography` fields mapped exactly per the UX spec ramp table (AppTitle→Title1, BoundedContextHeading→Subtitle1, ViewTitle→Title3, SectionHeading→Subtitle2, FieldLabel→Body1Strong, Body→Body1, Secondary→Body2, Caption→Caption1, Code→Body1). "Code" additionally sets a monospace family via a companion `TypographyStyle.Code` string constant because Fluent UI v5's `Typography` enum does not express family switching. | Stable public API surface that survives Fluent UI mapping changes (patch/minor/major cadence documented in XML doc). Constants, not enum — Fluent UI's `Typography` enum is the implementation. | Task 1.2, 10.10; AC5 |
| D3 | `FrontComposerShell` owns `FluentLayout` + `FluentProviders` + `Fluxor.Blazor.Web.StoreInitializer` placement. Adopters call `<FrontComposerShell>@Body</FrontComposerShell>` from their `MainLayout.razor` — they do NOT import `FluentLayout`, `FluentProviders`, or `StoreInitializer` themselves. | Zero-override scale: one framework-owned composition point; adopters don't re-derive the shell layout and can't accidentally omit `FluentProviders` (a common cause of silent Fluent UI failures). Consolidates the 3 integrations that Story 1-3 + 1-8 + 2-1 dev-notes each called out separately. | Task 5.1, 9.1; AC1; ADR-027 |
| D4 | `FrontComposerShell` exposes 5 public `RenderFragment` parameters + 1 string + 1 typed: `HeaderStart`, `HeaderEnd`, `Navigation`, `Footer`, `ChildContent`, `AppTitle` (string?), `HideHeaderPaletteTrigger` (bool, default `false`, set to `true` in 3-4 once the real trigger ships). `HeaderCenter` (breadcrumbs area) is NOT a parameter in 3-1 — Story 3-2 adds it when the sidebar ships. | Parameter set is append-only through v1 (same discipline as `FcLifecycleWrapper` per Story 2-4 D1). Consumers gain slots for customization without reaching into shell internals. | Task 5.1, 10.1; AC1 |
| D5 | `FcThemeToggle` is a `<FluentMenuButton>` rendering three child items (Light, Dark, System). Icons: Fluent `Icons.Regular.Size20.WeatherSunny` (Light), `WeatherMoon` (Dark), `DesktopMac` (System — the UX spec says "System" not "Auto", copy must match). Selected value shows the corresponding icon; icons are decorative (`aria-hidden="true"`) and the `aria-label` comes from `FcShellResources.ThemeToggleAriaLabel`. | `FluentMenuButton` is the v5-native control; using `<select>` or a custom popover re-invents the wheel and fails the zero-override strategy. The label "System" matches UX-DR25 verbatim. | Task 6.1, 10.2; AC3 |
| D6 | `IThemeService.SetThemeAsync(ThemeSettings)` is called exactly twice per circuit lifecycle: (a) once in `FrontComposerShell.OnAfterRenderAsync(firstRender: true)` using the accent color + the hydrated `ThemeState` (System if not stored), (b) every time the user picks a new theme value via `FcThemeToggle` — dispatched `ThemeChangedAction` is the trigger; the effect awaits the service call before persisting. We do NOT call `SwitchThemeAsync` (would double-toggle), `ClearStoredThemeSettingsAsync` (would drop OUR Fluxor state), or the `SetThemeAsync(string color, bool isExact)` overload (stores Fluent's own localStorage key — ambiguity with our tenant-scoped storage). | Fluent UI's `IThemeService` has both auto-persisting and non-auto-persisting paths; we pick the single explicit-apply path so Fluxor + `IStorageService` remain the single persistence writer. Rejects "let Fluent UI persist too" — double-persistence causes restore drift. | Task 5.1, 6.1, 10.3, 10.9; AC3; ADR-028 |
| D7 | Single write path into `FrontComposerThemeState` is the reducer for `ThemeChangedAction(CorrelationId, NewTheme)` (unchanged from Story 1-3). 3-1 does NOT introduce new actions. `FcThemeToggle` + `FcSystemThemeWatcher` + `ThemeEffects.HandleAppInitialized` all dispatch the existing action. | Preserves Story 2-3 D19 single-writer invariant precedent (applied to lifecycle state; here applied to theme state). Any new writer must be a deliberate future-story decision. | Task 6.1, 6.2; AC3 |
| D8 | `StorageKeys.DefaultTenantId = "default"` + `DefaultUserId = "anonymous"` constants are DELETED. `StorageKeys.BuildKey(tenantId, userId, feature[, discriminator])` keeps its signature but callers MUST resolve values from the actual `IUserContextAccessor` interface — `string? TenantId { get; }` + `string? UserId { get; }` (flat properties, no `.Current`, no nested `User`). When either is null, empty, or whitespace per the interface's XML contract (matched via `string.IsNullOrWhiteSpace`), the CALLING effect SHORT-CIRCUITS (logs `HFC2105` Information + returns). `LocalStorageService.GetAsync`/`SetAsync`/`RemoveAsync` never receive a key with empty segments — the effect layer is the boundary. | Memory feedback `feedback_tenant_isolation_fail_closed.md` + L03. Protects against cross-tenant bleed when real auth lands in Epic 7 — no "silent fallback" legacy to migrate. Matches the `LastUsedValueProvider` D31 precedent. The accessor's XML contract at `src/Hexalith.FrontComposer.Contracts/Rendering/IUserContextAccessor.cs:17-31` treats null / empty / whitespace as semantically equivalent, so `IsNullOrWhiteSpace` is the only correct guard. | Task 4.1, 4.2, 4.3, 10.5, 10.6; ADR-029 |
| D9 | `LocalStorageService.SetAsync` enqueues into a `Channel<PendingWrite>` (unbounded-single-reader) and returns a completed task; the drain worker awaits one `InvokeVoidAsync("localStorage.setItem", key, value)` at a time. Back-pressure is bounded by `LocalStorageMaxEntries` (LRU eviction keeps the channel coalesced). `FlushAsync` awaits `Channel.Reader.Completion` OR a tracked `pending.Count == 0` signal. | Architecture §522 ("`SetAsync` is fire-and-forget internally (async write, doesn't block render path)"). Channels + single-reader matches Fluxor effect concurrency patterns; no `SemaphoreSlim` needed. Unbounded is safe because LRU caps entries upstream. | Task 2.1, 2.2, 10.4; AC6 |
| D10 | `FcSystemThemeWatcher` is a **separate component**, not a feature of `FcThemeToggle`. It renders no DOM, subscribes on `OnAfterRenderAsync(firstRender)`, unsubscribes on `Dispose`, and dispatches ONLY when `ThemeState.CurrentTheme == ThemeValue.System`. Disposed on circuit teardown via `IAsyncDisposable`. | Separation of concerns: toggle = user intent writer, watcher = OS preference observer. Dispatch guard avoids fighting the user who explicitly picked Light while OS is Dark. Mirrors the Story 2-2 D37 "scoped component, lazy JS module" pattern. | Task 6.2, 10.3; AC3 |
| D11 | Typography mapping is a "living table" with the version-pinning rule: patch = no changes, minor = documented changelog + before/after screenshot, major = restructurable. `Typography.cs` XML comment codifies this. The version constant `Hexalith.FrontComposer.Contracts.ContractsMetadata.TypographyMappingVersion = "3.1.0"` is asserted in `TypographyConstantsTests.cs` so a drift is caught in CI. | UX spec §127-133 "living specification" rule applied to code. Without the version lock, a dependency bump could silently remap `ViewTitle` → `Subtitle1`, drifting every generated view with no audit trail. | Task 1.2, 10.10; AC5 |
| D12 | `FcShellResources` (EN + FR pair) lives in `Hexalith.FrontComposer.Shell/Resources/`. Keys present in EN resx must be present in FR resx (parity enforced by `FcShellResourcesTests.CanonicalKeysHaveFrenchCounterparts`). No machine translations — FR copy is hand-authored in this story. Lifecycle state labels that Story 2-4 referenced but never materialised ("Submitting…", "Syncing…", "Confirmed", etc.) are filled in here (lifts G2 from Story 2-4 Known Gaps forward). | Story 2-4 Known Gap G2 cited "localisation gap — FR lifecycle copy deferred". 3-1 closes the gap because the resource file infrastructure is new and the cost of adding 7 extra strings is 5 minutes. UX-DR60 requires EN + FR at v0.1. | Task 8.1, 10.7; AC7 |
| D13 | The six semantic color slots (`--fc-color-accent/neutral/success/warning/danger/info`) are declared in ONE place: `FrontComposerShell.razor.css`. Generated renderers + `FcLifecycleWrapper` consume them via CSS variable reference. The command-lifecycle → slot binding table (Idle=Neutral / Submitting=Accent / Acknowledged=Neutral / Syncing=Accent / Confirmed=Success / Rejected=Danger) is snapshot-tested via `SlotMappingRegressionTests.cs`. | Single source of truth + CI regression lock. Story 2-4 lifecycle colors implicitly matched these slots; 3-1 names them explicitly and tests the mapping. Any renaming surfaces in CI before it ships. | Task 7.1, 7.2; AC4; ADR-031 |
| D14 | `FcShellOptions` grows to 14 properties (10 pre-3-1 + 4 new). Split into separate option classes (`FcLifecycleOptions`, `FcFormOptions`, `FcThemingOptions`) is explicitly **deferred to Story 9-2** per L10 discipline. 3-1 documents the cross-concern count trigger (theming is the 3rd concern beyond lifecycle + forms) but does NOT split because mid-Epic refactors cost more than they save (per Story 2-5 dev-notes precedent). | Story 2-4 + 2-5 both crossed soft thresholds and deferred; 3-1 crossing the hard threshold flips the priority but does NOT change the release sequencing — Epic 3 needs to ship, 9-2 is already budgeted. | Task 1.1 (note-only); Known Gap G1 |
| D15 | LRU eviction key = `long` timestamp (`TimeProvider.GetUtcNow().UtcTicks`) NOT a `DateTimeOffset` or sequence counter. `Dictionary<string, long>` tracks last-access. On `GetAsync`, update the timestamp (read-touched); on `SetAsync`, write + update; on eviction, scan `OrderBy(kvp => kvp.Value).First().Key`. `O(n)` scan is acceptable because `n ≤ LocalStorageMaxEntries` which is capped at 10_000. | `long` is the most compact, cross-target serialisation; `TimeProvider` is already registered (Story 2-4). `OrderBy` is `O(n)` but `n` is small and LRU correctness beats `O(log n)` sorted structure complexity. | Task 2.1, 10.4 |
| D16 | `LocalStorageService.GetKeysAsync(prefix)` filters client-side after fetching `localStorage.length` + `localStorage.key(i)` in a single JS-interop round-trip (one `InvokeAsync<string[]>` call that returns all keys). Does NOT iterate per-index. | Minimises JS-interop chatter — critical in Blazor Server where each interop is a network round-trip. One-shot fetch + LINQ-side filtering is faster than N round-trips. | Task 2.1, 10.4 |
| D17 | `beforeunload.js` registers via a `BlazorShell` JS module loader that imports on demand — NOT via a `<script>` tag in `App.razor`. Module is imported once per circuit in `FrontComposerShell.OnAfterRenderAsync(firstRender)`. Unload handler calls `DotNetObjectReference.invokeMethodAsync("FlushAsync")` and awaits up to 200 ms before returning (browser will forcibly terminate after ~2 s anyway). | Architecture §523 ("FlushAsync called via beforeunload JS interop hook in App.razor") — we move the hook into `FrontComposerShell` so adopters don't manually edit `App.razor` (a common source of 1-line-forgot bugs). 200 ms is the budget that Chrome/Edge honor before background-promise termination. | Task 2.3, 3.1, 10.4 |
| D18 | `LocalStorageService` is registered as `Scoped` (per-circuit in Blazor Server, per-app in WASM) via `services.Replace` in `AddHexalithFrontComposer`. `InMemoryStorageService` remains the `Scoped` default for Server test hosts; the Counter.Web sample gets `LocalStorageService` wired because Blazor Server + browser localStorage works today (we call `InvokeAsync` with `JSRuntime` available by the time `FrontComposerShell.OnAfterRenderAsync` fires). | Architecture §521 lists both impls at v0.1; picking `LocalStorageService` for Server-or-WASM alike is simpler than auto-detecting render mode. Tests use `InMemoryStorageService` as before (overridden via bUnit `ComponentFactory`). | Task 2.3, 10.4; ADR-030 |
| D19 | `FcShellResources.AppTitle` defaults to `"Hexalith FrontComposer"` in EN and `"Hexalith FrontComposer"` (unchanged — product name stays Latin) in FR. | D20 | `--layout-header-height` is overridden to `48px` via a `:host { --layout-header-height: 48px; }` rule in `FrontComposerShell.razor.css` (scoped to the component). | D21 | `IStringLocalizer<FcShellResources>` is the ONLY resource-resolution path. `FcThemeToggle`, shell aria-labels, and the placeholder button labels inject it via `[Inject]`. Fluent UI's own localisable strings stay on Fluent UI's `LocalizationResource.BaseAddress` — we do NOT shadow them. | Keeps FrontComposer localisation independent of Fluent UI localisation — two different audiences + version cadences. | D22 | `FcThemeToggle` binds to Fluxor state via `IStateSelection<FrontComposerThemeState, ThemeValue>` (or `IState<FrontComposerThemeState>` then projects) — NOT `[Inject] IState<>` in the code-behind. Dispatches via `[Inject] IDispatcher`. | Standard Fluxor.Blazor.Web pattern; matches Story 1-3 conventions. `IStateSelection<T, U>` lets us rerender only on `CurrentTheme` changes (ignores other theme-state fields we may add later). | Task 6.1, 10.2 |
| D23 | `FcSystemThemeWatcher`'s JS module exports `subscribe(dotnetRef, initialValue)` returning a subscription handle. Subsequent `change` events fire the same callback. | Covers the "user loads page with OS in Dark mode + our stored theme is System" bootstrap case without requiring the watcher to poll. One-shot emission on subscribe is cheap (~1 interop call). | Task 3.2, 6.2, 10.3 |
