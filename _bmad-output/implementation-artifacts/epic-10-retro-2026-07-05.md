### What Went Well

1. **Retrospective action items became executable stories.** Epic 10 mapped each Epic 7 carry-forward
   action to a concrete story and acceptance criteria instead of leaving the work as process memory.

2. **The mechanical evidence gate immediately changed behavior.** Stories 10.2 through 10.5 used
   `eng/validate-story-artifacts.py` to separate story-owned changes from unrelated dirty workspace
   drift, and review completion now fails closed with `artifact_validation_failed`.

3. **Audit-first documentation avoided blind rewrites.** Story 10.2 changed adopter-facing stale
   labels while retaining legitimate provenance in diagnostic metadata, source comments, and
   internal references.

4. **CLI hardening stayed scoped to contract risk.** Story 10.3 added text-output parity coverage
   for summaries, filtering, fail flags, and diff budgets without treating terminal text as a full
   schema or changing CLI runtime behavior unnecessarily.

5. **HFCM9002 moved from ambiguity to a recorded decision.** Story 10.4 selected the not-approved
   production-emission path because no explicit Product + Architecture approval artifact existed,
   then guarded the synthetic/manual-only boundary.

6. **Testing redaction review found a real defect.** Story 10.5's review caught tenant/user leaks
   through JSON property names after the structural redaction rewrite; the fix restored whole-payload
   tenant/user replacement and raised the default Testing lane to 30/30.

### Challenges And Growth Areas

1. **Broad validation remains environment-sensitive.** Multiple stories could not run the standard
   filtered solution lane locally because of MSBuild/VSTest socket restrictions or restricted NuGet
   audit access. The story records handled this honestly, but the pattern remains operational noise.

2. **Concurrent dirty workspace state complicated evidence gates.** Stories 10.3 through 10.5 had
   to document unrelated Shell, CI, pact, sample, and Testing package-boundary drift so the new gate
   would not force fake story ownership.

3. **Contract docs can drift after implementation review fixes.** The Testing host contract still
   described the pre-Story-10.5 delimiter-aware redaction algorithm until this retrospective's doc
   verification pass corrected it.

4. **Story 10.1 exposed tooling ownership friction.** The first implementation pass recorded a
   blocked `.agents` edit path, and the review later completed the gate integration when those files
   were writable. Future process-tooling stories should verify writable roots before claiming a full
   automation loop.

## 3. Key Learnings

- **A review gate is stronger when it is executable.** Story status, File List, dirty-worktree
  classification, and checked-task evidence are now validated mechanically instead of relying on
  reviewer memory.
- **Documentation cleanup needs classification, not global replacement.** Epic 10 preserved valid
  historical provenance while removing current-contract wording that would confuse adopters.
- **Human-readable CLI output is a contract at selected boundaries.** Summary counts, fail flags,
  filters, and budget markers deserve parity tests; JSON-only structural fields do not need to be
  duplicated in text.
- **A negative decision is still architecture work.** Recording HFCM9002 production emission as not
  approved prevented docs from implying a capability that SourceTools does not implement.
- **Privacy fixes need adversarial shapes.** Tenant/user values can leak through dictionary keys and
  property names, not only through scalar JSON values.

## 6. Significant Discovery Check

Epic 10 does not invalidate Epic 11's plan. It does add three planning constraints:

1. **Mechanical gates are now part of the process baseline.** Epic 11 story records should expect the
   Story 10.1 validator and review-completion gate.
2. **Contract docs must be verified after review fixes.** The Testing host contract drift found in
   this retro shows that implementation review can change behavior after the initial doc decision.
3. **Testing privacy evidence should include identifier-bearing keys, not only values.** Story 11.6
   should preserve that model when it expands harness failure modes.

No Epic 11 story text needs to be rewritten during this retrospective. The current plan already
contains the two main decision gates and the Testing harness story that can absorb the privacy lesson.
