## Successes

1. Epic 12 preserved release truth over optimistic closure. The team explicitly separated story completion from release readiness.
2. Ledger ownership improved materially. Story 12.1 and Story 12.2 removed stale Story 11.x ownership markers and replaced them with row-scoped final states, accepted constraints, or named follow-ups.
3. Accepted constraints became auditable. The pending-command null-provider decision includes release-note wording, owner, watcher, trigger, and a governance test that will fail if the constraint is silently invalidated.
4. Release workflow evidence got much harder to misuse. Story 12.4 added typed classification, dry-run carve-outs with `publish_authorized=false`, manifest drift checks, bounded redaction, fallback invalidation, and many negative fixtures.
5. Manual accessibility and stakeholder evidence stopped being implied. Story 12.5 records missing manual AT, device, and stakeholder approvals as blockers instead of inferring pass from automated or representative evidence.

## Challenges

1. Review loops on Story 12.4 were heavy and generated many rounds of patches. That was useful, but it also shows the release workflow has dense shell/helper behavior that is hard to audit in one surface.
2. Submodule pointer refreshes appeared inside governance and evidence work. They were logged, but this pattern still creates avoidable review noise and scope ambiguity.
3. Several artifacts still have story-status versus release-readiness tension. Stories can be `done` while release gates remain blocked; the sprint record needs to communicate that distinction clearly.
4. Evidence schemas remain partly prose-driven. Story 12.5 added machine-readable classification, but deferred items still call for stronger schema validation and cross-pack precedence when more packs exist.
5. The trusted release path is still not fully production-ready because `CR-12-4-Def14` keeps attestation workflow support on an approved fallback path rather than true attestation readiness.

## Key Lessons

1. Release certification needs two statuses: work completion and release readiness. Collapsing them produces false confidence.
2. Accepted constraints need executable guards when possible. Story 12.3.1 made the pending-status constraint harder to forget than a release-note paragraph alone.
3. Inline workflow logic is expensive to review repeatedly. Story 12.4's patch volume supports extracting release prepare/publish logic into dedicated scripts in a follow-up.
4. Manual evidence must remain manual. Automation can support confidence, but it cannot stand in for NVDA, JAWS, VoiceOver, real-device, or stakeholder sign-off evidence.
5. Submodule updates should stay visibly separate from story-scoped release governance work unless the story explicitly owns them.
