D6, D9, D15 amended 2026-04-19 during advanced elicitation review; D15 received a second amendment pass 2026-04-19 for `OperationCanceledException` differentiation + HFC2106 `Reason` structured-field semantic (wording-only clarifications + error-path handling — no surface additions; decision count unchanged at 23). | D1 | `FrontComposerNavigation.razor` consumes `IFrontComposerRegistry.GetManifests()` via `[Inject]` and renders one `FluentNavCategory` per `DomainManifest` (group Id = `BoundedContext`, Title = `Name`) with one `FluentNavItem` per FQN string in `DomainManifest.Projections`. **Commands are NOT rendered as nav items** — the UX spec §141 rule ("Sidebar nav items are projection views only, not commands") is enforced by only iterating `manifest.Projections`. **When a manifest's `Projections` collection is empty (commands-only manifest), the `FluentNavCategory` is NOT rendered (no empty-category shell)** — the `@foreach` block over `Registry.GetManifests().Where(m => m.Projections.Count > 0)` skips such manifests entirely. | Zero-override + framework-owned navigation. The registry already carries the BoundedContext + Projections pair; no new contract needed. Commands appear inside projection views per Story 2-x action-density rules — not in navigation. Empty-category shells would be noise-without-signal; hiding the category is the honest signal ("this BC has no projection views yet"). Clarified 2026-04-18 during party-mode review (Murat + John). | Task 6.1, 10.1; AC1 |
| D2 | Nav item `Href` is **convention-based**: `/{boundedContext-lowercase}/{projectionTypeName-kebab-case}` where `projectionTypeName` is the segment after the last `.` in the projection FQN. Default viewport `Desktop` matches the UX spec §170 rule ("starts expanded for all new users"). | Task 1.2, 10.5; AC3 |

Module shape matches the existing prefers-color-scheme watcher so bUnit interop mocking patterns carry over. | D14 | Persistence triggers: `SidebarToggledAction` / `NavGroupToggledAction` / `SidebarExpandedAction` fire `NavigationEffects.HandlePersistNavigation`. **`NavigationHydratedAction` does NOT trigger persistence** (hydrate is read-only — re-persisting the same blob is wasted I/O and eliminating the hydrate→persist round-trip closes the SSR-to-interactive ordering surface; see ADR-038 + D23). **`ViewportTierChangedAction` does NOT trigger persistence** because `CurrentViewport` is NOT in the serialised blob (ADR-037). Effect reads the current state AFTER the reducer has run (Fluxor ordering: reducer → effect) so the blob reflects the new sidebar state. | Minimal writes: viewport changes ~ hundreds of times per resize drag on a real browser; persisting every change would thrash LocalStorage and consume the LRU budget. Viewport is observation, preferences are the knobs. Hydrate is a read — the blob is already on disk in exactly the shape we'd re-write, so the round-trip buys nothing. Amended 2026-04-18 during party-mode review (Dr. Quinn + Winston). | Task 3.1, 10.6; AC2; ADR-037, ADR-038 |
| D15 | `NavigationEffects.HandleAppInitialized(AppInitializedAction, IDispatcher)` loads the blob from `LocalStorageService`, deserialises `NavigationPersistenceBlob`, and dispatches `NavigationHydratedAction(sidebarCollapsed, collapsedGroups.ToImmutableDictionary())`. Reducer for `NavigationHydratedAction` REPLACES `SidebarCollapsed` + `CollapsedGroups` wholesale. Null / missing blob → no action dispatched; feature defaults apply. No `HFC2106`-equivalent is reserved (symmetric to Story 3-1 D6 `HFC2106_ThemeHydrationEmpty` is reused; `NavigationEffects` logs the same code on empty-blob to keep the diagnostic surface small). **Amended 2026-04-19 (× 2):** deserialization failure / storage I/O exception (corrupt JSON, schema drift, manual tampering) → `HandleAppInitialized` catches at the effect boundary, logs `HFC2106_ThemeHydrationEmpty` with exception payload AND structured field `Reason=Corrupt`, and treats as missing blob (feature defaults apply — hydrate never dispatched). Null / missing blob remains `Reason=Empty` on the same diagnostic ID — remediation for `Empty` is no-op (feature defaults are the intended outcome); remediation for `Corrupt` is manual LocalStorage clear or admin tooling (Story 9-2 scope). `HandlePersistNavigation` wraps `storage.SetAsync` in try/catch via a private `PersistAsync` helper (Task 3.1 prescribes single-site try/catch, not per-method duplication). Failures: `OperationCanceledException` (circuit disposal) → `Logger.LogDebug` silent cancellation path; any other `Exception` → `HFC2105_StoragePersistenceSkipped` at Information with exception payload. Neither propagates to the dispatcher (next successful toggle retries). **Second 2026-04-19 amendment pass** (via advanced-elicitation RD-4 + RT-4 findings) split `OperationCanceledException` from general failure handling to silence expected tab-close noise, and clarified the HFC2106 `Reason=Empty|Corrupt` structured-field semantic to resolve the diagnostic drift between "hydration empty" and "hydration errored". | Hydrate-once pattern inherited from `ThemeEffects`. `NavigationHydratedAction` is a replace-semantic action so reducer merge logic stays single-purpose. Reusing HFC2106 signals "hydration found nothing" across theme, density, and nav. | Task 3.1, 3.2, 10.6; AC2 |
