# Known Gaps (Explicit, Not Bugs)

Each row cites a specific owning story per L10 discipline.

| ID | Gap | Owning story | Rationale for deferral |
|---|---|---|---|
| G1 | Badge counts on `FluentNavItem` (ActionQueue-hinted projections show a count suffix like "Orders (3)"). `FrontComposerNavigation` renders plain labels; `IBadgeCountService` is the missing seam. | **Story 3-5** (home directory, badge counts & new capability discovery) | `IBadgeCountService` requires SignalR hub subscription + per-circuit scope choreography that Story 3-5 owns. 3-2 keeps the FluentNavItem child content area free so 3-5 can wrap `<span>@Label @Badge</span>` without parameter-surface change. |
| G2 | Forced-comfortable density at `ViewportTier <= Tablet` (UX spec §26-28). 3-2 ships the `ViewportTier` enum + dispatch infrastructure; the density-forcing effect that consumes it lives in Story 3-3. | **Story 3-3** (display density & user settings) | 3-3 owns `FrontComposerDensityState` and the `--fc-density` body attribute. Mixing viewport-driven density into 3-2 would move 3-3's core logic forward and blur ownership. |
| G3 | Projection role-hint icons in `FcCollapsedNavRail` (rail icons currently default to `Icons.Regular.Size20.Apps` for every bounded context). Role-hints (ActionQueue / StatusOverview / Timeline) live on the projection type, not the bounded context, so the rail has no authoritative icon mapping today. | **Story 4-1** (projection role hints and view rendering) | `DomainManifest` needs a new `ProjectionRoleHints` map before the rail can resolve an icon per bounded context. Epic 4 extends `DomainManifest`; until then the generic `Apps` icon is honest. |
| G4 | Projection friendly display name (currently the sidebar renders the type name humanized via `ProjectionLabel`). Adopters wanting a different label (e.g., `[DisplayName("Open Orders")]` on the projection) have no contract today. | **Epic 4** (projection rendering — ships `[DisplayName]` + `[ProjectionRoleHint]` attributes per existing architecture.md §394) | `DomainManifest.Projections` is a `IReadOnlyList<string>` — the source generator would need to produce a richer record before 3-2's renderer could read display names. Epic 4 is the natural home. |
| G5 | Breadcrumb content rendered in `HeaderCenter` slot. 3-2 ships the slot (D10) but no default content. | **Story 3-5** (home directory, badge counts & new capability discovery) — home routing + breadcrumb synthesis land together | `HeaderCenter` is append-only on `FrontComposerShell`'s parameter surface (D10), so 3-5 fills in without a parameter-surface bump. |
| G6 | DataGrid filter / sort / scroll position restoration on navigating back to a projection view. `LocalStorageService` is shipped in 3-1 and consumed here for nav state; 3-6 extends it to per-view DataGrid state. | **Story 3-6** (session persistence & context restoration) | 3-6 owns the per-view memory object + cross-session blob pair that DataGrids need. 3-2's `NavigationPersistenceBlob` is one of the several artefacts 3-6 coordinates. |
| G7 | Command palette Ctrl+K / shortcut reference / contextual results at current bounded context. `FrontComposerShell.HeaderStart` currently auto-renders `FcHamburgerToggle`; 3-4 adds a command palette trigger there (or beside it). | **Story 3-4** (FcCommandPalette and keyboard shortcuts) | 3-4 owns `IShortcutService` + `FcCommandPalette`. 3-2 leaves the HeaderStart slot adopter-overridable so 3-4 can add the palette icon without a contract break. |
| G8 | `[SuppressHFC2105]` analyzer escape hatch for intentional no-tenant samples — inherits the G3 gap from Story 3-1 without change. | **Story 9-4** (diagnostic ID system & deprecation policy) | Severity-configurable + suppression attribute require the broader diagnostic governance machinery that 9-4 is scoped to build. Same reasoning as Story 3-1 G3. |
| G9 | `prefers-reduced-motion` CSS rules for sidebar collapse animations. 3-2 does NOT ship custom animations (FluentLayoutHamburger drawer uses Fluent UI's built-in animation which already honors reduced-motion per Fluent UI docs). | **Story 10-2** (accessibility CI gates) | No custom animations to gate in 3-2. Gap retained for cross-story traceability. |
| G10 | Type specimen view including nav sidebar at all densities / viewports / themes. 3-2 ships the primitives; 10-2 assembles them. | **Story 10-2** (accessibility CI gates) | Specimen infrastructure is a 2-week build; 3-2 ships the primitives (ViewportTier, sidebar, rail, hamburger) that 10-2 assembles into the specimen. |
| G11 | RTL (`dir="rtl"`) verification — framework inherits Fluent UI's RTL behaviour but 3-2 does not ship RTL specimen baselines for the sidebar. | **v2** (UX spec §36) | v1 primary audience is EN + FR (LTR); adding RTL CI gates blocks shipping. |
| G12 | Windows High Contrast (`forced-colors`) mode verification for the sidebar rail + drawer. Framework inherits Fluent UI behaviour but does not gate it in CI. | **Story 10-2** (accessibility CI gates) | Requires Playwright `forced-colors` emulation — Story 10-2's scope. |
| G13 | Home directory (`FcHomeDirectory`) as the first-visit destination. 3-2 ships the sidebar; the home page it links to is 3-5. | **Story 3-5** (home directory, badge counts & new capability discovery) | 3-2 is navigation, not landing page. |
| G14 | Projection list pagination / search inside the sidebar — if a single bounded context has dozens of projections, the current flat list inside a category becomes hard to scan. No pagination or filter is shipped. | **v1.x** (UX spec §273 — at >15 projections, command palette becomes primary navigation) | The UX spec explicitly rules out deep sidebar scaling; command palette (Story 3-4) is the scale-aware navigation method. |
| G15 | Drag-to-reorder nav categories or favouriting / pinning individual projections. | **v2** (UX spec §37 mentions drag-to-reorder as inspiration-only, not a v1 commitment) | v1 is read-only sidebar — reorder / pin introduces a new persistence concern (user-ordered list) and conflicts with auto-registration semantics (bounded contexts appear when their domain is registered). |
| G16 | `FluentNavSectionHeader` support (section headers between categories to group related bounded contexts, e.g., "Commerce" header above Orders + Inventory). 3-2 emits a flat list. | **v1.x** (UX spec §12 mentions `FluentNavSectionHeader` as available but no v1 use case) | Requires a new `DomainManifest.Section` or equivalent metadata; not in Epic 4 scope either. Forward-compatible: inserting section headers between the `@foreach` output is mechanical if adopters demand it. |
| G17 | Keyboard focus visibility screenshot diff gate — bUnit asserts tab reachability and CSS custom property inheritance; Playwright screenshot diff against committed baselines is Story 10-2 scope. | **Story 10-2** (accessibility CI gates) | Screenshot-diff infrastructure is a 10-2 deliverable; 3-2 ships the behavioural assertion. |
| G18 | Same-circuit tenant switch (SSO silent re-auth that reuses the Blazor circuit without restart). `FrontComposerNavigationState.CollapsedGroups` retains previous tenant's values until the circuit restarts; the next toggle writes stale groups under the new tenant's storage key. In-memory state leak only — `IUserContextAccessor` scope resolution remains fail-closed per D12. | **Story 3-1** (circuit lifecycle + `IUserContextAccessor` change semantics + `AppInitializedAction` re-fire contract) | Story 3-1 owns the circuit-restart contract and `AppInitializedAction` re-fire semantics on tenant change. Enforcing tenant-scoped in-memory Fluxor state across all features (Theme, Density, Navigation) is a cross-feature concern, not a per-story one. Added 2026-04-19 via advanced elicitation review (PM2 / CH3 / RT1 finding). |
| G19 | `FcCollapsedNavRail` does not scroll or overflow-handle; 48 px × N bounded-context rail buttons stacked vertically overflow the viewport at N > ~12 on the CompactDesktop tier (min 768 px viewport height). Median adopter has 3-8 bounded contexts — acceptable for v1. | **v1.x** (UX spec §273 directs adopters with > ~15 projections/BCs to the command palette — Story 3-4 is the scale-aware navigation surface) | Adding rail scroll / overflow handling before the command palette ships (Story 3-4) inverts the UX spec's scale-aware-nav priority. Command palette is the designated escape for high-BC-count adopters. Added 2026-04-19 via advanced elicitation review (CH6 finding). |
| G20 | Returning-user sticky-collapse confusion. A first-time user collapses the sidebar at Desktop (per D9: sticky-opt-in persisted across tier round-trips and browser sessions). Days later they reopen the laptop and see a 48 px icon rail at Desktop width — may briefly believe the sidebar is broken before noticing the hamburger toggle. | **v2** (user-onboarding gradient — tooltip / state-explanation overlay for "your sidebar preference is saved; click ☰ to expand") | Behavior is intentional per D9 amendment (opt-in stickiness is a feature, not a bug). Hamburger toggle is always visible at any non-full-Desktop state (D8), so the recovery path is one click away. Onboarding affordances (first-time-collapse tooltip, "this is intentional" overlay) are a cross-story UX concern — not 3-2's scope. Added 2026-04-19 via round-table finding RT-7. |
