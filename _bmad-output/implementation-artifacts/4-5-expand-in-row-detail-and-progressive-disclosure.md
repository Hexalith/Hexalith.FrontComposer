## Executive summary (Feynman-level, ~45 sec)

Today every generated projection view renders a flat `FluentDataGrid` — business users who need to inspect an order or verify an invoice's line items must click the row, navigate to a full-page detail, review, then navigate back (losing scroll position and filters along the way). Story 4-1 shipped the `DetailRecord` role body (`FluentCard` + `FluentAccordion` with 6-primary / 7+-secondary split) but left it stranded as a standalone view — nothing composed it into a DataGrid context. Story 4-5 closes the loop: **every generated DataGrid view gains an expand-in-row affordance that, on click, reveals the entity's `DetailRecord`-shaped detail panel INLINE beneath the row, without full-page navigation, WITHOUT losing scroll/filter/virtualize state, and WITH scroll stabilization so serial queue processing (expand row 3 → review → collapse → expand row 4) feels like turning pages, not jumping around.**

1. **Single-expand Fluxor feature.** A new per-concern `ExpandedRowState` Fluxor feature (Shell-scoped) tracks exactly ONE expanded row per view via `(viewKey, itemKey) → ExpandedRowEntry` — the "single expand v1" constraint (UX-DR17, user-journey §208) is enforced at the reducer level: dispatching `ExpandRowAction(viewKey, newKey)` clears any existing entry for that viewKey before writing the new one. `CollapseRowAction(viewKey)` removes the entry. No cross-session persistence — expanded state is ephemeral, within-session only (mirroring 4-4 D5 scroll-position ephemeral policy; restoring a user's "last expanded row" 8 hours later is the same disorienting anti-pattern).

2. **Emitter integration via `EmitStandardDataGrid`.** The existing Story 4-1 `EmitStandardDataGrid` helper in `ProjectionRoleBodyEmitter` grows a new `emitExpandableRows` parameter (default `true` for the four DataGrid strategies — Default / ActionQueue / StatusOverview / Dashboard; Timeline + DetailRecord skip because Timeline uses `FluentStack` and DetailRecord already IS the detail). When true, the emitter adds: (a) `RowClass="@GetRowClass"` + `OnRowClick="HandleRowClickAsync"` on the `FluentDataGrid`, (b) a trailing `TemplateColumn` with `Width="40px"` rendering a `FluentButton Appearance="Stealth" IconStart="Icons.Regular.Size20.ChevronRight"` (rotates 90° to ChevronDown when expanded — CSS class toggle, no icon swap for reduced-motion parity), (c) AFTER the `FluentDataGrid` closing tag, a `FluentCollapsiblePanel`-hosted detail container that reads `IState<ExpandedRowState>.Value.GetEntry(viewKey)` and conditionally renders the `EmitDetailRecordBody` output for the expanded item.

3. **ItemKey-driven expansion identity.** The expanded row is identified by `ItemKey` (the `AggregateId` / `Id` / `Key` / fallback resolved by Story 4-4 D13's `_itemKeyAccessor`). Story 4-4's keyed virtualization means `ItemKey` is already a stable per-row identifier — 4-5 reuses it verbatim. When the user's filter changes and the currently-expanded item scrolls out of the filtered set, the reducer-side invariant `ExpandedRowState.ExpandedItemKey` stays populated but the render-side guard (`state.Items.Any(x => _itemKeyAccessor(x).Equals(state.ExpandedItemKey))`) silently suppresses the detail panel — no exception, no orphan render. On the next explicit filter clear that brings the item back, the panel reappears. On page unload (`view DisposeAsync`), `CollapseRowAction(viewKey)` is dispatched to avoid leaking state when the user navigates away.

4. **Scroll stabilization via existing JS module.** Story 2-2 already shipped `IExpandInRowJSModule` + `fc-expandinrow.js#initializeExpandInRow(elementRef)` — the canonical implementation of UX-DR17 (scrollIntoView with block:'nearest' + prefers-reduced-motion branch + rAF stabilization correction within `SCROLL_CORRECTION_THRESHOLD_PX=4`). 4-5 REUSES this unchanged. The detail container carries `@ref="_detailRef"` and the emitted view's `OnAfterRenderAsync(firstRender)` dispatches `_expandInRowModule.InitializeAsync(_detailRef)` immediately after an `ExpandedRowState` change (tracked via `_lastExpandedItemKey` field). On expand, the JS module runs its scroll-nearest routine against the detail element; on collapse, no JS call is needed (the collapse animation flows from the detail's removal, and the previous-row position is preserved by the DataGrid's virtualization).

5. **DetailRecord body composition + progressive disclosure.** The detail panel's render body is the SAME output as `EmitDetailRecordBody` produces for the `DetailRecord` role — `FluentCard` with first-6 fields + `FluentAccordion` wrapping positions 7+ (D9). This means Story 4-1 DetailRecord role + Story 4-5 expand-in-row share ONE emit path for the inner card/accordion layout — a factored `EmitDetailRecordBody(sb, model, builderName, itemExpression, seqStart)` variant is introduced that 4-1's role body and 4-5's expanded-row host both consume (D10). Change once, both surfaces stay consistent. **Field grouping for secondary accordion** (UX-DR18): 4-5 adds a new OPTIONAL `[ProjectionFieldGroup("GroupName")]` property attribute (Contracts) — properties sharing a `GroupName` render as a single `FluentAccordionItem` labeled with that group; properties with no `[ProjectionFieldGroup]` fall into an "Additional details" catch-all section (alphabetical-chunking fallback per UX-DR18). This is the only new attribute shipped by 4-5; it's append-only and optional (feature-story budget per L06 stays lean).

6. **Prefers-reduced-motion + animation.** The detail container uses Fluent UI's standard expand/collapse transition via `FluentCollapsiblePanel` (the Fluent component that ships with a CSS-driven animation already honoring `prefers-reduced-motion`). No custom animation; no `@keyframes` at story level. When `prefers-reduced-motion: reduce` is active, the JS module's `scrollIntoView` already skips the follow-up rAF correction and uses `behavior: 'auto'` instead of `'smooth'` (Story 2-2 semantic). 4-5 adds NO reduced-motion branching in Razor — inherits from both Fluent and the JS module.

7. **Row actions inside detail (phone + primary surface).** Per UX-DR62, phone-tier viewports (<768px) hide inline action buttons on DataGrid rows via CSS media query — actions appear ONLY inside the expanded detail. For 4-5 this means: (a) the detail panel's trailing `<FluentStack Orientation="Horizontal" HorizontalGap="8">` action zone renders ALL available row commands as `FluentButton Appearance="Secondary"` (consistent with UX-consistency-patterns §table: "Expand-in-row detail | Available commands | Secondary"), (b) on desktop viewports the action zone still renders (actions appear BOTH inline on the row AND inside the detail panel — Notion/Linear precedent, gives power users options and reduces target discovery), (c) phone-viewport inline-row-action hiding lives behind a CSS media query in `fc-projection.scoped.css` (opt-in, matches Story 3-3 responsive density auto-switch precedent). The phone viewport LAYOUT fine-tuning (single-column detail grid, drawer nav) remains Story 10-2 scope; 4-5 ships the emitter-side DOM + CSS hook but does NOT ship the phone-tier visual specimen.

8. **DataGrid state preservation — filter + scroll + virtualize coexistence.** Expanded rows DO NOT break Story 4-4 virtualization because the detail panel renders OUTSIDE the virtualized grid (below it, not inside a row template — no variable-height rows, no `Virtualize` math interference). Filters (4-3), sort (4-3), scroll (4-4 `ScrollTop`), column visibility (4-4 `__hidden`) persist as before — the expanded-row reducer DOES NOT touch `GridViewSnapshot`; it owns its own `ExpandedRowState` feature state and does not participate in the 3-6 persistence pipeline (ephemeral by design per D3). When the user collapses + navigates + returns within-session, filters/sort/scroll restore via 3-6/4-3/4-4 machinery unchanged; the expanded row does NOT restore (fresh state = collapsed) — consistent with UX principle "scroll position is within-session; expanded row is intra-task-ephemeral."

The design honors the "capability contract" principle: `GridViewSnapshot` + persistence blob SCHEMA UNCHANGED (4-5 adds no new persisted field), feature-story decision budget stays inside L06 (≤25 — 4-5 lands at 22/25 after D21 + D22 additions, with a 3-decision buffer), every cross-story seam is explicit (L01), fail-closed tenant/user scoping inherited from Story 3-6 unchanged (L03 — no new scope-check code because nothing new persists). Scope is deliberately narrow: **single-row expand, DetailRecord-shaped detail body, progressive-disclosure via new `[ProjectionFieldGroup]` attribute, scroll stabilization via Story 2-2's existing JS module, phone-viewport DOM hook. Nothing else.** Multi-expand comparison (v2), drag-to-reorder fields inside the detail (Epic 6-3), expanded-row edit mode (Epic 6-2), SignalR live-refresh of an expanded detail panel (Epic 5-4 reconciliation), and phone-tier visual finalization (Story 10-2) are OUT of scope.

## Story

As a business user,
I want to expand an entity row in place to see full details without losing my DataGrid context,
so that I can inspect and act on items without navigating away and losing my scroll position and filters.

**Business value:** Until 4-5 lands, any projection field beyond the ~8 columns that fit in a DataGrid row is invisible unless the developer writes a custom full-page detail view. A fulfillment operator processing 40 orders per morning either (a) dismisses the data they can't see (a silent quality problem), or (b) navigates to a full-page detail and then tediously back to the grid for each order (a 15-second cost per inspection × 40 orders = 10 minutes of friction per morning, per user). After 4-5 ships, the operator clicks the row, the detail opens beneath the row with all fields grouped by shipping / billing / line-items, reads, clicks an inline command button, and the row collapses — zero navigation cost, zero context loss. This is the story that makes FR20 ("expand an entity row in place") real; it is ALSO the story that operationalises Story 2-2's `IExpandInRowJSModule` (shipped dormant since 2-2) in its PRIMARY use case (row-expansion, not compact-inline-form expansion); AND it is the story that connects Story 4-1's `DetailRecord` role body (currently a standalone surface) to the DataGrid ecosystem where it belongs.

### Relevant architecture patterns and constraints

- **Zero-override architecture** (`architecture.md:30`): `FluentDataGrid`, `FluentCollapsiblePanel`, `FluentAccordion`, `FluentAccordionItem`, `FluentCard`, `FluentButton`, `FluentIcon`, `FluentStack` all consumed verbatim. No custom CSS beyond the FOUR classes introduced by 4-5 (`.fc-row-action-column`, `.fc-row-expanded`, `.fc-expand-button[--open]`, `.fc-expand-in-row-detail`). Each serves a layout-structural purpose (phone-hide, highlight, rotation, region-wrap) — no design-token overrides, no Fluent-primitive styling overrides.
- **Convention-based rendering** (`architecture.md:493-497`): "IR metadata (field types, role hints, badge mappings)" — 4-5 adds `FieldGroup` to the IR metadata vocabulary. After 4-5, every generated DataGrid view has expand-in-row baked in + the adopter may group secondary fields without writing custom Razor.
- **Fluxor single-source-of-truth discipline** (Story 1-3 ADR-008): every state change is an immutable `record` action; reducers are PURE `(state, action) => state`; effects are the only dispatch-boundary. 4-5's `ExpandRowAction` + `CollapseRowAction` flow through the standard dispatch pipeline; reducers NEVER chain dispatches (Story 4-4 D4/D7 correction preserved); NO effects (ephemeral state means no persistence effect, no cross-action orchestration).
- **Per-concern Fluxor features** (architecture §527 + L02): `ExpandedRowState` is a new per-concern feature because expanded-row state is semantically orthogonal to `DataGridNavigationState` (persistence-linked) + `LoadedPageState` (pagination-linked). One state, one concern.
- **IR cache stability** (Story 1-4 incremental-build contract): 4-5 adds `PropertyModel.FieldGroup` + `ColumnModel.FieldGroup` as derived fields; `RazorModel` shape unchanged (grouping is a layout concern, not a model-structure concern — the sort order + primary/secondary split stays). IR byte-stability preserved — Story 1-4 cacheability tests stay green.
- **Blazor Server + Blazor Auto compatibility** (UX-DR59): `IJSRuntime` for the reused Story 2-2 `IExpandInRowJSModule` works in both modes; the JS module is circuit-scoped in Server, app-scoped in WASM (standard Blazor scope rules). No new interop surface introduced.
- **Persistence fail-closed** (L03 tenant isolation): Not applicable to 4-5 — nothing new is persisted. State is ephemeral. The inherited Story 3-6 fail-closed discipline remains unchanged because 4-5 does not call 3-6's effects.
- **NFR4 performance budget**: 4-5 does NOT affect the 500-row virtualized render budget. The detail panel is ONE extra component per view (only rendered when expanded), outside the virtualized grid, with no incremental cost to virtualization throughput.

- **Phone-viewport visual verification.** Per D11 + AC7, 4-5 ships the DOM `.fc-row-action-column` class + the `@media (max-width: 767px)` hide rule in `fc-projection.scoped.css`. Actual phone-viewport visual specimen (Playwright at 375px) is **Story 10-2 "accessibility CI gates and visual specimen verification"** scope. 4-5's gate is DOM + CSS presence; Story 10-2's gate is the rendered pixel parity.
- **Detail-panel-inside-virtualized-grid adopter overrides.** Per D6, 4-5 renders the detail panel OUTSIDE the virtualized grid. An adopter Epic 6-3 slot override that tries to place the detail INSIDE a `FluentDataGrid.RowTemplate` (or equivalent v5 API if one materializes) would break `Virtualize`'s constant-`ItemSize` assumption. Documentation guidance + an optional analyzer warning (HFC-TBD) lives in **Epic 6-3 "Level 3 slot-level field replacement"** or **Epic 9-4 "diagnostic ID system and deprecation policy"**.
- **Multi-expand comparison.** Per scope guardrail, single-expand is v1. Multi-expand as a selection-based comparison panel (distinct view mode, NOT stacked expansions) is v2 scope — documented in **design-direction-decision.md** user-journey-flows §208. No owning story yet; waits for post-v1 adopter feedback.
- **Expanded-row edit mode.** Per scope guardrail, the detail is read-only. Inline field editing is **Story 6-2 "Level 2 typed Razor template overrides"** territory. An adopter who wants "click field → edit → save" writes a custom detail template via Epic 6.
- **Drag-to-reorder fields within the detail.** Per scope guardrail, accordion section ordering is first-declared-property precedence (D9). Drag-reorder is **Story 6-3 "Level 3 slot-level field replacement"** customization.
- **SignalR live-refresh of expanded detail.** The detail re-reads `state.Items[itemKey]` on every render, so passive state updates flow through. Optimistic-update visuals for edits to the expanded-item (e.g., "the item I'm inspecting just got approved by another user") are **Epic 5-4 "reconnection reconciliation"** territory. The `FcDesaturatedBadge` from Epic 5 applies uniformly whether a badge is in the grid row or the detail body.
- **Cross-session expanded-row restore.** Per D3, not shipped (UX anti-pattern). If future user research surfaces a legitimate use case, **Epic 9-5** or **post-v1 retrospective** may revisit.
- **AOT/trim analyzer for `object ItemKey` boxing.** The `ExpandedRowEntry.ItemKey` is typed as `object` because Fluxor feature states cannot be generic (Story 4-4 D16 constraint inherited). Value-type keys box on the Dispatcher.Dispatch path. **Epic 9-4 "diagnostic ID system and deprecation policy"** + an analyzer entry (new diagnostic, deferred).
- **CSS `:has()` fallback** (if required per T2.1 spike). `:has()` pseudoclass is cross-browser supported on modern browsers (Chrome 105+, Firefox 121+, Safari 15.4+). IE is dead; Safari <15.4 is < 1% of business-user traffic. If the spike determines `:has()` is the ergonomic choice for styling the parent `<td>` based on `.fc-row-action-column` presence, document as an acknowledged forward-only choice. **Epic 10-2 browser-matrix docs** formalize.
- **Detail panel keyboard shortcuts (`ArrowRight`/`ArrowLeft`).** Per scope guardrail, 4-5 uses native row activation (`Enter` on focused row). Dedicated arrow-key bindings for expand/collapse are **Epic 9-3 "IDE parity and developer experience"** scope (alongside other power-user keyboard shortcuts).
- **Detail-panel density adaptation** (3-col at Compact/Comfortable, 2-col at Roomy, 1-col at phone). 4-5 ships the single 3-col grid (inherited from 4-1 `EmitDetailRecordBody`). Full density-responsive inner grid is **Story 10-2 "accessibility CI gates and visual specimen verification"** territory.
- **`[ProjectionFieldGroup]` name collision across projections** (same name in two different projections). Not a collision at the framework level — each projection's detail surface is independent. No HFC diagnostic needed.
- **`FcExpandInRowDetail` as a reusable Shell API** for adopters who want expand-in-row in custom DataGrids (not generated). The component IS already reusable (`EditorRequired` parameters, scoped behaviour), but documentation lives in **Epic 9-5 "Diataxis documentation site"**.
- **ICU plural handling for `ExpandRowButtonAriaLabelTemplate`.** The template uses `{0}` for the row label (noun), not plural counts — no ICU needed. When (if) the template becomes plural-sensitive ("Expand details for 1 item" vs "for N items"), migrate via **Epic 9-5 ICU migration**.
- **ICU plural handling for `ExpandedRowHiddenByFilterBanner` (v1.1 commitment, NOT v2 deferral; added 2026-04-24).** The banner template `"{0} expanded item hidden by current filter"` is currently grammatical only when `{0} == "1"` (single-expand invariant in v1 means count is always 1). Once multi-expand comparison ships (post-v1 — see scope guardrail), `{0}` ranges across `2..n` and English/French plural rules apply ("1 item hidden" vs "2 items hidden"; "1 élément masqué" vs "2 éléments masqués"). The original spec deferred ICU pluralization to Epic 9-5 (a generic ICU migration umbrella). Pre-mortem analysis (2026-04-24) revised this: because the BANNER is the most-announced suppression path AND screen-reader users are most affected by ungrammatical announcements, the banner's pluralization MUST land alongside multi-expand v2 (NOT after — bundling with the v2 commit avoids two grammar-regression review cycles). Owning surface: **multi-expand v2 story (post-v1, no story-id yet)** — add ICU pluralization to its scope at story-creation time. Until then, the singular-only template ships under the single-expand invariant — adopters using v1 see no grammatical issue.
