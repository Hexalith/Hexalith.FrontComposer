# Critical Decisions (READ FIRST — Do NOT Revisit)

> 20 binding decisions (19 at spec time + D20 added during party-mode review 2026-04-19 — promotes Known Gap G10 aria-live announcement to scope). Feature-story budget is 25 (L06) — 3-3 is comfortably below. Raise a spec-change proposal before revisiting any of them.

| # | Decision | Rationale | Consumed by |
|---|---|---|---|
| D1 | **Four-tier density precedence** is resolved by a single pure function `DensityPrecedence.Resolve(DensityLevel? userPreference, DensityLevel? deploymentDefault, DensitySurface surface, ViewportTier tier) : DensityLevel`. Order (first non-null / applicable wins): **(1)** tier-force override — `tier <= ViewportTier.Tablet` → `DensityLevel.Comfortable` (ADR-040); **(2)** user preference (non-null); **(3)** deployment default (non-null, from `FcShellOptions.DefaultDensity`); **(4)** factory hybrid default for the surface (UX spec §208-214); **(5)** per-component default — not reachable through the central resolver because per-component is a level-4-below-resolver custom-component concern, documented in `DensityPrecedence` XML comments as the "resolver's exit"; a custom component that wants Compact-forever simply does not consult `EffectiveDensity` and hardcodes `data-fc-density="compact"` on its root. | UX spec §197-204 codified as code. Pure function makes the rule reviewable in one file; the exception-free ordering eliminates "which tier wins?" ambiguities. The tier-force override sits at position 1 (not position 4) because it is a hard accessibility floor for 44 px touch targets, not a default — see ADR-040. | Task 1.2, 10.1; AC1, AC4 |
| D2 | `FrontComposerDensityState` is restructured to `public record FrontComposerDensityState(DensityLevel? UserPreference, DensityLevel EffectiveDensity)`. Feature initial state = `new(UserPreference: null, EffectiveDensity: DensityLevel.Comfortable)`. `Comfortable` is the feature-default `EffectiveDensity` because it matches the UX spec §211-213 factory default for 4 of the 5 surfaces (`DetailView`, `CommandForm`, `NavigationSidebar`, and the `Default` fallback). `DataGrid` + `DevModeOverlay` surfaces resolve their own `Compact` default via the surface parameter passed into the precedence resolver — the state's `EffectiveDensity` is the "shell-level" value, not per-surface. **State-rewindow window (Winston review, 2026-04-19):** the rename `CurrentDensity → EffectiveDensity` is safe because no adopter-facing consumer of `CurrentDensity` has shipped. The window during which this claim holds is bounded: Story 3-1 merge commit `bc9afd4` (first appearance of `CurrentDensity` in a `main` tag) through Story 3-3 merge commit `TBD` (lands `EffectiveDensity`). Any adopter pinning between those commits pins the interim `CurrentDensity` shape — that risk is accepted (no v0.x adopters exist; Counter.Web is in-tree). Post-3-3, the pair `UserPreference`/`EffectiveDensity` becomes a pinned public contract; further rewindows require an `[Obsolete]` shim cycle. | Two concerns, two fields. `UserPreference` is the knob; `EffectiveDensity` is the resolver output. Separating them makes the "user set Compact but Tablet forcing applied" scenario representable in state (`UserPreference = Compact, EffectiveDensity = Comfortable`) — which is exactly what the Settings dialog needs to highlight without a per-component re-resolve. Story 3-1 shipped `(DensityLevel CurrentDensity)` with no adopter-facing consumer; renaming to `EffectiveDensity` is a state-shape change, not a breaking public contract. | Task 2.1, 10.4; AC1, AC4 |
| D3 | **`DensityPrecedence.Resolve` is re-run inside the reducer for every triggering action**, NOT computed lazily via a selector. The reducer consumes the current `state.UserPreference`, `IOptions<FcShellOptions>.Value.DefaultDensity`, the action-specific surface context (for `Default` surface in 3-3 — the single surface wired in this story), and the current viewport tier (from `IState<FrontComposerNavigationState>.Value.CurrentViewport`). Reducer method signature uses a static resolver + an injectable dependency closure by construction: the reducer receives the pre-resolved `EffectiveDensity` from the action payload — action producers (effects or components that dispatch) read `FcShellOptions` + `NavigationState.CurrentViewport` and compute the new effective density BEFORE dispatching. **Reducers stay pure** — resolving inside the reducer would require injection, which violates Fluxor's static-reducer discipline. | Fluxor reducers are pure static methods. Resolver injection breaks that. Moving the compute to the action producer (effect / component) preserves purity; reducers just assign `state with { EffectiveDensity = action.NewEffective }`. The alternative — lazy selector — would require every component to re-resolve on every render, multiplying the resolver's cost by component count. | Task 2.2, 3.2, 10.4; AC1 |
| D4 | **`FcShellOptions.DefaultDensity : DensityLevel?` (nullable)** added as the 15th property. Null → "no deployment default; fall through to factory hybrid." Non-null enum values are validated by ASP.NET Core's built-in `EnumDataType` DataAnnotation — no custom regex needed. The Story 3-1 G1 options-class split stays deferred to Story 9-2; 14 → 15 is append-only and does not cross a new threshold. XML comment documents the precedence-tier-2 role. | Explicit nullable enum is cleaner than a sentinel value. Adopters who don't set `DefaultDensity` (the common case) get `null` and the factory hybrid applies. Enterprises setting `"Compact"` in `appsettings.json` get the call-center-style default the UX spec §201-202 cites. | Task 1.1, 10.2; AC1 |
| D5 | `DensitySurface` is a `public enum DensitySurface : byte { Default = 0, DataGrid = 1, DetailView = 2, CommandForm = 3, NavigationSidebar = 4, DevModeOverlay = 5 }` in `Contracts/Rendering/` (alongside `DensityLevel`). All six values ship in 3-3 to lock the enum surface; only `Default` is wired through the resolver in 3-3. Downstream consumers pass their surface into `DensityPrecedence.Resolve(surface: DensitySurface.DataGrid, ...)` when they land (4-1 for DataGrid, 4-5 for DetailView, 6-5 for DevModeOverlay). `NavigationSidebar` + `CommandForm` resolve their `Comfortable` factory default through `Default` today (both map to `Comfortable` at the factory layer — see `DensityPrecedence.GetFactoryDefault` switch). | Stable enum wire format — adding values later is append-only; reordering or deleting is a v2 break. Shipping the full enum in 3-3 avoids a parameter-surface bump in every downstream story that consumes it. Integer values are pinned so any future serialisation (e.g., a `DevModeOverlay` setting) can round-trip. | Task 1.2, 10.1 |
| D6 | **Factory hybrid default mapping (UX spec §208-214):** `Default` → `Comfortable`; `DataGrid` → `Compact`; `DetailView` → `Comfortable`; `CommandForm` → `Comfortable`; `NavigationSidebar` → `Comfortable`; `DevModeOverlay` → `Compact`. Encoded as a private static switch in `DensityPrecedence.GetFactoryDefault(DensitySurface) : DensityLevel`. No public surface — adopters who want a different factory default must either set `FcShellOptions.DefaultDensity` (tier 2) or use a per-component hardcoded density (tier 5 — outside the resolver). | The factory defaults are a UX-spec constant, not an adopter knob. Hiding the switch behind a private static makes adopters follow the hierarchy rather than patch individual surfaces. Mirrors the Story 3-2 D22 "breakpoints are UX-spec constants, not adopter-tunable" principle. | Task 1.2, 10.1 |
| D7 | `DensityEffects` gains a new `[EffectMethod] HandleViewportTierChanged(Navigation.ViewportTierChangedAction, IDispatcher)` cross-feature handler that re-resolves `EffectiveDensity` via `DensityPrecedence.Resolve(...)` and dispatches an intra-feature `EffectiveDensityRecomputedAction(DensityLevel NewEffective)` whose reducer assigns `state with { EffectiveDensity = action.NewEffective }`. The existing scope guard (`TryResolveScope`) is NOT checked on the viewport handler because this is a pure compute path — no storage write happens. | Cross-feature reducers are anti-pattern in Fluxor; cross-feature EFFECTS are standard. Having the density effect listen to the navigation feature's action keeps ownership clear (density feature owns its state; navigation feature owns viewport). The intra-feature recompute action keeps the reducer pure per D3. | Task 3.2, 10.4; AC4 |
| D8 | `DensityEffects.HandlePersistDensity` is retargeted to fire ONLY on `UserPreferenceChangedAction` + `UserPreferenceClearedAction`. It serialises `DensityLevel?` (nullable enum) directly under the existing `{tenantId}:{userId}:density` key — null on cleared, the enum value on changed. Story 3-1's `DensityChangedAction`-triggered persistence path is REMOVED (the existing Story 3-1 reducer-and-persist pipeline is replaced). **`DensityHydratedAction` is INTENTIONALLY EXCLUDED** from the persist set (ADR-038 mirror — hydrate is read-only). Storage failure handling mirrors Story 3-2 D15 verbatim: `OperationCanceledException` → `LogDebug`; other `Exception` → `HFC2105` Information with exception payload; neither propagates. Hydrate errors → `HFC2106` at Information with `Reason={Empty|Corrupt}`. | The user's explicit preference is what persists — the computed `EffectiveDensity` reflects viewport + defaults which are observation/config, not preference. Persisting `EffectiveDensity` would lock the user into whatever tier they happened to have when they last toggled. Nullable enum serialisation via `System.Text.Json` is native (no custom converter). | Task 3.1, 3.2, 10.5; AC3 |
| D9 | **CSS `--fc-density` is applied as a `body[data-fc-density]` attribute**, NOT as an inline `style="--fc-density: …"`. The JS module (`fc-density.js`) sets `document.body.dataset.fcDensity = level.toLowerCase()`. Scoped CSS in `FrontComposerShell.razor.css` declares three selector rules: `body[data-fc-density="compact"] { --fc-spacing-unit: 2px; }`, `body[data-fc-density="comfortable"] { --fc-spacing-unit: 4px; }`, `body[data-fc-density="roomy"] { --fc-spacing-unit: 6px; }`. **All generated Razor scoped CSS consumes `var(--fc-spacing-unit)` via `padding: calc(var(--fc-spacing-unit) * N)` multipliers** — zero per-component density logic lives in the consumer (ADR-041). | Attribute-driven CSS cascades cleanly; inline styles fight with Fluent UI's own CSS-variable inheritance. Using `data-*` attributes is the standard HTML5 idiom. One `<body>` mutation per density change (no DOM traversal, no per-component re-render from CSS). Fluent UI tokens are untouched — we compose OUR unit on TOP of Fluent UI's 4 px grid. | Task 4.1, 4.2, 10.6; AC6 |
| D10 | **`FcDensityApplier.razor`** is a headless Blazor component (renders `@* empty *@`) that mounts inside `FrontComposerShell.Content` alongside the other watchers. Subscribes to `IStateSelection<FrontComposerDensityState, DensityLevel>` (projects `EffectiveDensity`), imports `fc-density.js` lazily on `OnAfterRenderAsync(firstRender)`, and invokes `setDensity(level)` on every `StateSelection.Selected` change (fire-and-forget `Task` inside a `Subscribe` lambda — no `await` on the JS void path). First invocation fires with the hydrated/computed initial value. `IAsyncDisposable` releases the JS module + unsubscribes. | One component = one concern. Story 3-1's Theme flow separately applies via `IThemeService.SetThemeAsync` inside `ThemeEffects`, but Density has no Fluent UI equivalent service — we set the attribute ourselves. Mirrors the `FcSystemThemeWatcher` / `FcLayoutBreakpointWatcher` pattern exactly so testing + disposal contracts are identical. Fire-and-forget is safe because a missed density update is self-healing on next change dispatch. | Task 4.2, 4.3, 10.6; AC6 |
| D11 | `FcSettingsButton.razor` renders `<FluentButton Appearance="ButtonAppearance.Stealth" OnClick="OpenDialogAsync" aria-label="@Localizer[\"SettingsTriggerAriaLabel\"]"><FluentIcon Value="@(new Icons.Regular.Size20.Settings())" /></FluentButton>`. Clicking calls `IDialogService.ShowDialogAsync<FcSettingsDialog>(...)` directly — no intermediate action dispatch. Dispatch-vs-direct-call considered and rejected: the dialog open signal does not need to be observable from any other component (no analytics, no test hook that requires it in the Fluxor action log); adding an action introduces a persistent state change ("is dialog open?") that Fluxor would need to track — but `IDialogService` already owns dialog lifetime. | Zero-override: Fluent UI's `IDialogService` is the dialog lifecycle owner. Wrapping it in a Fluxor action to "track open state" doubles the source of truth for dialog visibility. The button is a thin call-site for `IDialogService`. | Task 5.1, 6.1, 10.8 |
| D12 | **Story 3-1 D26 compile-away guard for the Settings button is RETIRED.** `FrontComposerShell.HideHeaderSettingsTrigger` parameter (if it existed in 3-1 — per Story 3-1 D4 it was sized for but may not have landed as a public parameter) is NOT added. The `@if (false)` placeholder block in Story 3-1's `FrontComposerShell.razor` for the Settings icon (if present) is DELETED. | Symmetry with Story 3-2's `HeaderStart` / `FcHamburgerToggle` auto-populate. Zero-setup settings button for the 99% case; override escape hatch for the 1% case. `@if (false)` compile-away guards only make sense BEFORE the feature exists (Story 3-1's state). Now that Story 3-3 ships the feature, the guard is obsolete. | Task 5.2, 8.2, 10.7; AC7 |

Standalone nav-item representation is acceptable because this is a specimen, not production behaviour. | Task 7.1, 7.2, 10.9; AC2, AC5 |
