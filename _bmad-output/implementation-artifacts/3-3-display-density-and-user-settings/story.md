# Story

## Story Statement

**As a** business user,
**I want** to choose my preferred display density and access settings easily,
**So that** the application matches my work style — compact for scanning, roomy for detailed work.

## Story Goal (one sentence)

Ship a framework-owned settings surface that exposes the **four-tier density precedence** (user preference → deployment default → factory hybrid → per-component default) via a `FcSettingsDialog` (FluentDialog with density radio options, embedded `FcThemeToggle`, and a live preview specimen) accessible through an unhidden `FcSettingsButton` in `FrontComposerShell.HeaderEnd` and the `Ctrl+,` shortcut — while also consuming Story 3-2's `ViewportTier` to force `Comfortable` at `Tablet`/`Phone` viewports (44 px touch targets) — all behind the Story 3-1 zero-override invariant and the ADR-029 fail-closed tenant scope.

## Scope Statement

Story 3-3 is the **Epic 3 density + settings body** that builds on 3-1's density Fluxor skeleton and consumes 3-2's `ViewportTier` observation. It retires Story 3-1 D26's compile-away placeholder for the Settings button, delivers the `FcSettingsDialog` UX shell (density + theme on one surface with a live preview — matches epic AC §126), introduces the `FcShellOptions.DefaultDensity` deployment-default tier, and applies `--fc-density` as the single CSS custom property on `<body>` driving every generated view's padding/sizing.

**In scope:**

- `src/Hexalith.FrontComposer.Shell/State/Density/DensityPrecedence.cs` — a `public static class` with a single pure function `DensityPrecedence.Resolve(DensityLevel? userPreference, DensityLevel? deploymentDefault, DensitySurface surface, ViewportTier tier) : DensityLevel` that implements the UX spec §197-204 four-tier rule inclusive of the tier-forcing override.
- `src/Hexalith.FrontComposer.Contracts/Rendering/DensitySurface.cs` — `public enum DensitySurface : byte { Default = 0, DataGrid = 1, DetailView = 2, CommandForm = 3, NavigationSidebar = 4, DevModeOverlay = 5 }` matching UX spec §208-214 factory hybrid defaults. **Lives in `Contracts/Rendering/` alongside the existing `DensityLevel.cs`** (cross-feature enum surface — consumed by `DensityEffects` + downstream Story 4-1/4-5/6-5 consumers). Story 3-3 ships the enum + `Default` surface only — the other surfaces land as they are consumed (DataGrid = Story 4-1, DetailView = Story 4-5, CommandForm = Story 2-1 adopters, NavigationSidebar = Story 3-2, DevModeOverlay = Story 6-5).
- `FrontComposerDensityState` extended from the current single-field `(DensityLevel CurrentDensity)` to a two-field record `(DensityLevel? UserPreference, DensityLevel EffectiveDensity)`. `CurrentDensity` (the 3-1 shape) is RENAMED to `EffectiveDensity`; `UserPreference` is new and holds `null` when the user has not explicitly chosen. **This is NOT a rename-then-break — Story 3-1 shipped `CurrentDensity` but no adopter-facing consumer exists yet, so the state shape is rewindowed with no breaking public contract.** The single adopter-facing constant is the `DensityLevel` enum itself, which is unchanged.
- New `DensityActions`: `UserPreferenceChangedAction(string CorrelationId, DensityLevel NewPreference, DensityLevel NewEffective)`, `UserPreferenceClearedAction(string CorrelationId, DensityLevel NewEffective)` (resets to "follow defaults"), `DensityHydratedAction(DensityLevel? UserPreference, DensityLevel NewEffective)`, `EffectiveDensityRecomputedAction(DensityLevel NewEffective)` (dispatched by the cross-feature viewport handler). **Every action carries the pre-resolved `NewEffective` payload** — per D3 + ADR-039, action producers call `DensityPrecedence.Resolve(...)` BEFORE dispatching so reducers stay pure (no DI in reducers). The existing `DensityChangedAction` from Story 3-1 is KEPT (backward-compatible entry point for a Story 3-4 "set density" command-palette action); its reducer assigns the incoming level to both `UserPreference` + `EffectiveDensity` until 3-4 migrates callers to `UserPreferenceChangedAction`.
- `DensityReducers` rewritten to recompute `EffectiveDensity` via `DensityPrecedence.Resolve(...)` on every action: `UserPreferenceChangedAction`, `UserPreferenceClearedAction`, `DensityHydratedAction`, and `Navigation.ViewportTierChangedAction` (cross-feature reducer listener — the reducer reads `IOptions<FcShellOptions>.Value.DefaultDensity` and the current viewport tier from the action payload, recomputes effective density, emits `state with { EffectiveDensity = newEffective }`).
- `DensityEffects` extended: on `UserPreferenceChangedAction` / `UserPreferenceClearedAction`, persist ONLY the user preference (`DensityLevel?`) under the existing `{tenantId}:{userId}:density` key. `DensityHydratedAction` is intentionally NOT in the persist-trigger set (ADR-038 mirror — hydrate is read-only). Fail-closed scope guard inherited from Story 3-1 D8 / ADR-029 unchanged.
- `FcShellOptions.DefaultDensity : DensityLevel?` — nullable; null means "no deployment default, fall through to factory hybrid." Parameter surface count 14 → 15 (Story 3-1 G1 options-class split stays deferred to Story 9-2; 14 → 15 is append-only and does not cross a new threshold). Validation: no `[RegularExpression]` needed since the enum is the type; `DataAnnotations` auto-covers enum-in-range.
- `src/Hexalith.FrontComposer.Shell/wwwroot/js/fc-density.js` — tiny ES module exporting `setDensity(level)` which writes `document.body.dataset.fcDensity = level.toLowerCase()` as its SINGLE side effect. **No inline `style.setProperty("--fc-density", ...)` write** — ADR-041 forbids inline style (fights Fluent UI's own CSS-variable inheritance). The `--fc-spacing-unit` CSS custom property is declared in `FrontComposerShell.razor.css` via three `body[data-fc-density="X"] { --fc-spacing-unit: Npx }` rules; the data-attribute IS the single signal. Imports symmetrically with `fc-prefers-color-scheme.js` and `fc-layout-breakpoints.js` (no subscribe/unsubscribe — this module is write-only).
- `FcDensityApplier.razor` + `.razor.cs` — headless component that lives inside `FrontComposerShell.Content` alongside `FcSystemThemeWatcher` and `FcLayoutBreakpointWatcher`. Subscribes to `IState<FrontComposerDensityState>` via `IStateSelection<FrontComposerDensityState, DensityLevel>` (projects `EffectiveDensity`) and invokes `fc-density.js#setDensity(level)` on every change (fire-and-forget). First invocation in `OnAfterRenderAsync(firstRender)` with the initial state value.
- Six semantic density padding rules added to `FrontComposerShell.razor.css` via a `body[data-fc-density="compact|comfortable|roomy"]` selector chain — defines `--fc-spacing-unit` variables that every generated view consumes via `padding: calc(var(--fc-spacing-unit) * N)`. Rules: Compact = `--fc-spacing-unit: 2px`; Comfortable = `--fc-spacing-unit: 4px`; Roomy = `--fc-spacing-unit: 6px`. These unit-level variables map onto Fluent UI's 4 px grid — they DO NOT override Fluent UI tokens; they inform OUR scoped CSS emitted for generated views (Story 2-2 renderers, Story 3-2 sidebar, etc.).
- `FcSettingsButton.razor` + `.razor.cs` — a thin wrapper around `FluentButton Appearance="ButtonAppearance.Stealth"` with the Fluent `Icons.Regular.Size20.Settings` icon, rendered inside `FrontComposerShell.HeaderEnd` via auto-populate (symmetric to Story 3-2 D8 for `FcHamburgerToggle` in HeaderStart). Clicking the button calls `IDialogService.ShowDialogAsync<FcSettingsDialog>(...)` **directly** — no intermediate Fluxor action (per D11 rationale: Fluent UI's `IDialogService` already owns dialog lifetime; wrapping in an action would double the source of truth for dialog visibility). **Story 3-1 D26 / ADR-032 `@if (false)` compile-away guard for this button is retired in Task 5.** `FrontComposerShell.HideHeaderSettingsTrigger` parameter is REMOVED (hide-before-ready is obsolete now that the feature ships).
- `FcSettingsDialog.razor` + `.razor.cs` + `.razor.css` — a `FluentDialog Modal="true" Width="480px"` containing: (a) a `FluentRadioGroup<DensityLevel>` with three radio buttons (Compact / Comfortable / Roomy) bound to `NavState.UserPreference ?? EffectiveDensity`; (b) the existing `FcThemeToggle` embedded verbatim (DO NOT duplicate its logic — the toggle is a self-contained component); (c) a `FcDensityPreviewPanel` showing a 3-element specimen (one DataGrid row, one form field, one nav item) that renders at the currently-selected radio value WITHOUT waiting for the user to press "Apply" — changes take effect live per epic AC §126.
- `FcDensityPreviewPanel.razor` — a pure-presentational component that accepts a `DensityLevel` parameter (decoupled from Fluxor so the preview can show a density DIFFERENT from the currently-applied one). Renders a 3-element specimen inside a `data-fc-density="{level}"`-scoped wrapper so the preview's CSS padding reflects the selected level without mutating global state.
- 13 new resource keys in `FcShellResources.resx` + `.fr.resx`: `SettingsDialogTitle`, `SettingsDialogCloseAriaLabel`, `DensitySectionLabel`, `DensityCompactLabel`, `DensityComfortableLabel`, `DensityRoomyLabel`, `ThemeSectionLabel`, `DensityPreviewHeading`, `CtrlCommaShortcutHint`, `DensityForcedByViewportNote`, `RestoreDefaultsLabel` (renamed from `ResetToDefaultsLabel` per D13 amendment — Freya review), `RestoreDefaultsHelperText` (new — "Clears density preference and sets theme to follow system."), `PreviewOnlyBadgeText` (new — "Preview only — Comfortable is active." for D14 forced-viewport badge), `DensityAnnouncementTemplate` (new — "Density set to {0}." for D20 aria-live announcer). Existing `SettingsTriggerAriaLabel` (seeded in Story 3-1 resx) is NOT modified. Parity test `CanonicalKeysHaveFrenchCounterparts` enforces FR counterparts.
- `FrontComposerShell.razor` modification: insert `<FcDensityApplier />` alongside `<FcSystemThemeWatcher />` and `<FcLayoutBreakpointWatcher />`; auto-populate `HeaderEnd` with `<FcSettingsButton />` when the adopter leaves it null (symmetric to HeaderStart auto-populate in Story 3-2 D8). When the adopter supplies a non-null `HeaderEnd` fragment, the adopter wins — Story 3-1 D26 had this "override escape hatch" already.
- Ctrl+, shortcut — wired via `@onkeydown="HandleGlobalKeyDown"` on the `.fc-shell-root` `<div>` in `FrontComposerShell.razor` (inline single-binding). On `event.Key == ","` AND `event.CtrlKey == true` (and no Shift/Alt/Meta modifiers), calls `IDialogService.ShowDialogAsync<FcSettingsDialog>(...)` directly — same path as the button per D11 (no Fluxor action intermediate). `CtrlCommaSingleBindingTest` (Task 10.10a, Winston review) pins the exactly-one-binding invariant to guard Story 3-4 migration. Full `IShortcutService` infrastructure with registration + conflict detection lands in **Story 3-4**; 3-3 ships the inline binding so the epic AC §124 "Ctrl+, shortcut" clause is satisfied today without blocking on 3-4.
- `IDialogService.ShowDialogAsync` integration — leverages Fluent UI v5's built-in dialog service (no custom overlay); Story 3-3 does NOT introduce a new dialog primitive.

**Out of scope (Known Gaps / downstream stories):**

- `IShortcutService` abstraction with registration + conflict detection at build time → **Story 3-4**. 3-3 ships the inline Ctrl+, binding directly on the shell; 3-4 migrates it to the service-based pattern when the command palette needs the same shortcut infrastructure.
- Command palette access to the settings dialog (Ctrl+K → "Settings" → opens `FcSettingsDialog`) → **Story 3-4**.
- Per-component density overrides (Level-1 annotation gradient allowing a developer to mark a single `[Projection]` as always-Compact) → **Story 6-1** (Level-1 annotation overrides).
- Factory hybrid defaults beyond the `Default` surface (DataGrid-surface compact, CommandForm-surface comfortable, etc.) — the `DensitySurface` enum ships with all 6 values, but only the `Default` surface resolver path is wired in 3-3. Each downstream surface landing story (4-1 for DataGrid, 4-5 for DetailView, 2-x Epic 2 stories for CommandForm, 3-2 already ships NavigationSidebar-comfortable behaviour implicitly) adds its surface-specific branch when it consumes the resolver.
- Multi-tenant deployment-default density (different `DefaultDensity` per tenant via runtime options monitor) → **v1.x**. v1 ships `FcShellOptions.DefaultDensity` as a startup singleton bound from `IConfiguration`.
- Persistence of density preferences to server-side storage for cross-device sync → **v2** (inherits Story 3-1 UX spec §106 rationale verbatim).
- Roomy-specific extended-spacing verification at Playwright screenshot diff → **Story 10-2** (density parity specimen). 3-3 ships Roomy in bUnit + manual verification; 10-2 adds the CI screenshot gate.
- `forced-colors` mode contrast verification of the settings dialog controls → **Story 10-2**.
- RTL verification of the settings dialog → **v2** (UX spec §36).
- Factory hybrid defaults validation against the UX spec's Factory Default table at startup (surfacing an `HFC2xxx` diagnostic if an adopter's `DefaultDensity` is "below" the UX recommended floor for their primary surface type) → **v1.x**. 3-3 trusts the adopter's `DefaultDensity` choice without a contract-checking diagnostic.
