## 2. What Went Well

1. **Zero regressions, clean build, every story.** Release build was `0 warnings / 0 errors`
   under `TreatWarningsAsErrors=true` on all seven stories, and the pre-existing 13-failure
   test baseline was reproduced *identically* against the prior commit in every story
   (stash → rebuild → re-run). The team never once shipped a new failure and never once
   misattributed a pre-existing one.

2. **Disciplined scope guarding.** The locked **7-parameter `FrontComposerShell` surface**
   (`FrontComposerShellParameterSurfaceTests`) stayed untouched and green across all stories.
   The **NFR17 `SetAsync` tripwire** held at count **7** with the whitelist untouched. Gated
   baselines (`producer-fingerprints.json`, `api-summary-baseline.txt`,
   `PublicAPI.*`) were byte-identical. Confirm-and-pin stories made **zero `src/` changes** —
   exactly as designed.

3. **The Story 1.0 spike paid off.** It empirically answered all four 🔴 AR5 bootstrap/table
   API questions against a throwaway host, then left behind a *permanent* 8-test regression
   suite pinning those answers against live `src/` so they can't silently regress before
   Story 1.1/Epic 2 consume them. The reviewer's judgment call — keep the valuable suite,
   make the documentation honest, rather than delete passing work — was the right one.

4. **Adversarial review caught real defects before "done".** Two genuine correctness bugs
   were found and fixed in review: (1.1) a false bootstrap diagnostic that claimed
   `AddHexalithDomain<TMarker>()` "was called" on the empty-marker path; (1.2) **M1** — a
   page that rebinds `FcPageLayout.Mode` after first render was silently ignored because the
   value was only pushed inside the `firstRender` gate. Neither would have been caught by a
   rubber-stamp review.

5. **Pattern reuse, done well.** Story 1.2's `FcPageLayoutCoordinator` deliberately mirrored
   the established `LayoutHamburgerCoordinator` child→shell cascade — consistent with house
   style and easy to review.

6. **Honest verification under environmental constraints.** Where a ready-gate lane could not
   run locally (1.3 e2e a11y: Playwright unsupported on the OS + Node v22 < v24; 1.5 full
   docfx: Razor-compiler/docfx-metadata analyzer mismatch), the team said so explicitly,
   ran the runnable structural subset, and cited the standing green CI job — **no false
   "passed locally" claim was ever made.**

## 3. What Was Challenging / Growth Areas

> Systems and processes, not people. No blame.

1. **🔁 Recurring #1 — File List / test-count drift after the QA-automation pass.** In
   **every single story** the adversarial review found the story's **File List omitted
   net-new artifacts** and the test counts were stale. The cause is structural: the
   `bmad-qa-generate-e2e-tests` pass runs *after* `dev-story`, adds test files (and
   `_bmad-output/.../tests/*-summary.md`), but **does not write back** to the story's File
   List or counts — so the review has to reconcile it each time.
   - 1.0 R1/R2/R3 (omitted regression suite, "only spike note" false in 3 places)
   - 1.2 L2 (omitted `1-2-test-summary.md`)
   - 1.3 (added `1-3-test-summary.md`; counts 4→6 / 1675→1677)
   - 1.4 MED×2 (omitted `1-4-test-summary.md`; "2 tests/1679" → "4 tests/1681")
   - 1.5 MED-1 (omitted governance test + summary)
   - 1.6 MED (omitted `settings.page.ts`, `settings-persistence.spec.ts`, `test-summary-1-6.md`)

2. **🔁 Recurring #2 — authoring sentinels leaking into deliverables.** Stray
   `</content>` / `</invoke>` tool-call tags contaminated authored files: **1.3 HIGH**
   (leaked tags at the tail of the FC-A11Y contract) and **1.4** (`</content>` at
   `1-4-test-summary.md:73`, plus a transient parse error during authoring). An
   LLM-authoring artifact reaching a committed file is the single highest-severity recurring
   class in the epic.

3. **All five contracts are `escalated`, not `confirmed`.** A direct consequence of YOLO:
   no live Product/UX or Tenants-author sign-off was available, so FC-LYT, FC-A11Y, FC-L10N,
   FC-DOC (and the FC-SETTINGS note's AC3 wording) all carry `status: escalated`,
   `owner: … (pending)`. Open inputs remain unresolved: FC-LYT default-mode choice + exact
   max-measure value; FC-L10N density-preview-string scope + host-owned domain-label
   fallback; FC-DOC cross-link convention; FC-SETTINGS "one reducer / one effect" literal
   wording.

4. **Two ready-gate enforcement lanes cannot be verified locally.** The e2e a11y lane
   (FC-A11Y AC3) and the full `docfx` gate (FC-DOC Gate-2d) both depend on environment that
   this host lacks. Today they rely entirely on CI. That is acceptable but fragile — a
   local contributor cannot self-check either gate before pushing.

5. **Copy-a-pattern-without-the-difference risk.** The 1.2 **M1** bug came from mirroring
   `FcHamburgerToggle` (which registers a *stable `@ref`*) for `FcPageLayout` (which carries
   a *changeable value* `Parameter`). The pattern was right; the difference (ref vs. value)
   was missed. Worth remembering when Epic 2 reuses coordinator/cascade patterns for grids.

## 4. Key Insights

- **Confirm-and-pin is a legitimate, high-leverage story shape** for a brownfield codify
  epic — it produced contracts + a durable regression net with near-zero `src/` risk. Epic 2
  shifts to genuinely-new generation/rendering, so expect the risk profile (and the volume of
  review findings on `src/`) to rise.
- **The adversarial review is the quality backbone.** Every story reached `done` only after
  it caught either a real bug (1.1, 1.2) or a documentation-integrity defect (all of them).
  Keep it; do not let YOLO erode it.
- **The 13-failure baseline is well-managed but real.** It is consistently traced to other
  owners (Epic-9 artifacts, governance, Verify snapshots, Linux case-sensitivity, CLI
  path-parsing). It is *not* Epic 1 debt, but it is a standing fog that every story has to
  re-prove around. Burning it down would remove that recurring tax.
