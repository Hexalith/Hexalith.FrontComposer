topic: Common UX for Hexalith applications using Hexalith modules
updated: 2026-07-05T09:18
---

- (decision) The UX spine covers the common UX shared by all Hexalith applications that use Hexalith modules.
- (decision) Each Hexalith module contributes exactly one primary application menu entry that routes to the module main page.
- (decision) A module main page may be described as that module workspace or dashboard and acts as the parent surface for the module's UX pages.
- (decision) The module main page contains tabs for the module UX pages rather than placing each module UX page directly in the application-level menu.
- (decision) Stakes are mixed consumer-facing and administrator usage. The common UX must support both professional administration workflows and consumer-facing module surfaces.
- (decision) Primary form factor is web desktop.
- (decision) Visual direction must remain professional and use Fluent UI Blazor components. Do not create custom CSS unless needed; always use Fluent UI theme colors and avoid custom colors unless required.
- (decision) Authoritative brownfield sources include all scanned planning artifacts under _bmad-output/planning-artifacts that are relevant to FrontComposer common UX.
- (decision) Core user journey: a user searches for a domain aggregate item such as a party, then adds a new item or modifies an existing item.
- (override) The new common-UX rule supersedes any brownfield pattern that exposes each module UX page or projection as a primary application menu entry. Application navigation has one entry per module; module workspaces own the tabbed page navigation. Epic 8 projection-flyout behavior requires reconciliation with this rule.
- (assumption) Use fast-path UX drafting for the first spine pass: no creative-tool mockups yet; draft decisions from user input and confirmed brownfield sources with open items explicitly marked.
- (event) Draft DESIGN.md and EXPERIENCE.md created for the common Hexalith application UX spine.
- (decision) Module tabs are route-backed or otherwise deep-linkable. Command palette entries, browser refresh, support links, CTAs, tests, and command-result links may land on a specific module UX page without adding additional application-level menu entries.
