frontcomposer - UX Design Validation Report

_bmad-output/planning-artifacts/ux-designs/ux-frontcomposer-2026-07-05

The spine pair is a useful draft for the common Hexalith module UI: it captures FrontComposer/Fluent governance, one-entry-per-module IA, support-safe microcopy, and projection-confirmed command truth. It is not yet a clean downstream contract because the sources include the full FrontComposer PRD while the flows cover only a narrow Parties operator scenario, several component names/specs drift across the two spines, and load-bearing route/flyout decisions remain open.

No critical defects were found. The high-severity issues should be resolved, or the declared source scope narrowed, before the pair is promoted beyond draft.

Flow coverage

thin

Checked PRD UJ-1 through UJ-6 against EXPERIENCE Key Flows. Coverage is narrow and not source-name preserving.

high

Source user journeys are not covered verbatim

prd.md:50; EXPERIENCE.md:132
The PRD names six journeys, but EXPERIENCE has only two Parties flows for Alex and Maya. Bootstrap, MCP, compatibility, and testing journeys are absent or not named from source.
Fix: Narrow sources to the common application UX sources this spine actually covers, or add/rename key flows that trace UJ-1 through UJ-6.

Token completeness

adequate

Token references resolve internally, and Fluent UI inheritance makes semantic token references acceptable for this project. Status coverage still needs a small explicitness pass.

medium

Status and contrast tokens are not explicit enough

DESIGN.md:28; visual-refresh.md:211
The design declares success, warning, and error tokens, but the visual-refresh source requires success, error, unknown/neutral, warning, and info mappings plus contrast verification.
Fix: Add missing neutral/info status token semantics or explicitly state the Fluent semantic color source and contrast verification rule.

Component coverage

thin

Visual and behavioral component inventories exist, but their names and rows do not line up reliably.

high

Component names and rows are not aligned across spines

DESIGN.md:51; EXPERIENCE.md:62
DESIGN defines kebab-case component tokens while EXPERIENCE uses different labels. Several components have only visual or only behavioral treatment.
Fix: Create one canonical component table or glossary shared by both files, with one visual row and one behavioral row per load-bearing component.

State coverage

adequate

Core shell, grid, stale, reconnecting, rejected, and destructive states are covered. Some command lifecycle states need explicit treatment.

medium

Command lifecycle states are not fully covered in State Patterns

EXPERIENCE.md:69; EXPERIENCE.md:73
The component pattern names `IdempotentConfirmed`, `NeedsReview`, and `Degraded`, but the state table omits explicit rows for several lifecycle states.
Fix: Add rows for Submitting, Syncing, IdempotentConfirmed, NeedsReview, and Degraded, including recovery and live-region rules.

Visual reference coverage

thin

No mockups, wireframes, or imports exist yet. That is acceptable for a fast-path draft, but it should remain draft until spine-only surfaces are intentional.

low

No visual artifacts are linked or present

EXPERIENCE.md:170
The first visual reference implementation remains open, so downstream consumers must infer key surfaces from prose.
Fix: Link the first reference implementation or key-screen mock, or mark these surfaces as spine-only with rationale.

Bloat & overspecification

strong

The spines are compact and mostly decision-oriented. They avoid restating the full PRD and keep brand/visual decisions in DESIGN while behavior remains in EXPERIENCE.

Inheritance discipline

thin

Source paths resolve, but duplicate PRD sources and divergent source lists make source extraction ambiguous.

medium

Source inheritance is ambiguous

DESIGN.md:7; prd.md:12
Both spines point to both the canonical PRD and the BMad run copy, while the PRD says those are copies of the same requirements. DESIGN and EXPERIENCE source lists also differ.
Fix: Use one PRD source of record, align shared source lists, and add scope notes for visual-only or behavior-only sources.

Shape fit

thin

Required sections are present and DESIGN section order is canonical. The remaining open questions keep the contract from being closed.

high

Load-bearing decisions remain open inside the spine

EXPERIENCE.md:168
Default tab naming, tab route encoding, projection flyouts, and the first visual reference implementation affect routing, IA, tests, and story handoff.
Fix: Resolve the route/tab/flyout decisions or move them into a non-blocking backlog with the current default decision in the spine.
